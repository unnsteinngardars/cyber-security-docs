📐Why acquisition matters

Acquisition is the act of producing a forensically sound copy of digital media so the original can be set aside untouched. Everything that follows in the case is built on this copy; if it is contaminated, every later finding becomes challengeable in court.

  • Bit-for-bit — every sector, including unallocated space and slack, not just live files. Carving (Section 05) depends on this.
  • Hash before & after — MD5 + SHA-256 of source and image. Identical hashes prove the image was not altered.
  • Work on copies — analyse a working copy; keep the master image read-only.
  • Chain of custody — log every transfer, action, examiner, and tool version against the evidence ID.
💾E01 vs raw (dd) — the artefact itself

raw / dd is a flat sector-by-sector copy. No compression, no metadata — biggest, simplest, universally readable. Mount it with anything (AIM, losetup, xmount).

E01 (EnCase Evidence File / Expert Witness Format) is the de-facto forensic container. It stores the bitstream plus embedded metadata, splits into segments (.E01, .E02, …), compresses, and writes per-chunk CRCs.

  • Header carries case number, examiner, evidence ID, acquisition notes.
  • Embedded MD5 / SHA-1 let the format self-verify on every read.
  • CRC per 32 KB chunk localises corruption instead of trashing the whole image.
  • Compression typical 1.3–3× depending on data entropy.
  • Read by every forensic suite (FTK, EnCase, Autopsy, X-Ways, libewf).

AFF4 (modern, used by WinPmem) can bundle memory, pagefile and disk into one container — common in IR work.

🛠FTK Imager — finding & producing the artefact

FTK Imager (Exterro / AccessData) is free, runs on Windows, runs portable from a USB stick. The standard taught-course tool. It can:

  • Image a physical drive, logical volume, single folder, or memory.
  • Write output as raw dd, E01, SMART, or AFF.
  • Preview a mounted image — file tree, hex view, registry hives — before doing a full acquisition.
  • Mount an image read-only as a Windows volume (helpful for quickly browsing without AIM).
  • Capture memory and protected system files (SAM, SYSTEM, SOFTWARE, $MFT) from a live host where Windows file locks would normally block you.

Capture flow → File → Create Disk Image → Physical/Logical → choose format → fill case metadata → set destination on a different drive → verify hashes after.

FTK Imager CLI — E01 of a physical drive
ftkimager \\.\PHYSICALDRIVE1 case01 --e01 --compress 6 --case-number 001 --examiner "U.G."

🚧Tableau & write blockers — preventing artefact contamination

A write blocker sits between your workstation and the suspect drive, passing reads through but silently dropping every write at the protocol layer. Without one, just attaching the disk lets Windows update last-access times, mount the journal, and break the source hash before you even open FTK Imager.

  • Hardware (Tableau, WiebeTech, CRU) — physical bridges for SATA / SAS / USB / NVMe. Forensically validated, court-accepted. Tableau is the orange brick you see in every classroom.
  • Software — Linux blockdev --setro, Windows USB write-protect registry flag, or boot a forensic distro (CAINE, SIFT, Tsurugi) that mounts every disk read-only.
  • Always test — hash the source, mount, attempt a write, hash again. Record the model and firmware on the chain-of-custody form.

Pair Tableau on the wire with FTK Imager on the workstation side for the textbook evidence-sound workflow.

Cross-references for corroboration

The E01 itself isn't an evidence finding — it is the substrate that every other artefact in this cheat sheet is parsed from. So "corroboration" here means making the acquisition itself defensible:

  • Hash chain — source-hash on the chain-of-custody log must match the hash FTK Imager embedded in the E01, must match a fresh hash of the working copy. Three independent values, all equal.
  • E01 metadata vs. CoC log — examiner name, case number and timestamps embedded in the E01 header must match the paper log. Mismatch is a serious red flag in court.
  • Tableau LEDs / logs — write-blocker status (green = read-only) photographed during acquisition.
  • Memory dump (Section 02) — if RAM was captured first, its acquisition timestamp should precede the disk image's.
  • Windows shutdown timestamp (Section 08, SYSTEM hive) — once the image is parsed, the last shutdown time in SYSTEM should align with when you pulled the plug.
