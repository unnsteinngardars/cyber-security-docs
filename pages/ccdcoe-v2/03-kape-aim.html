🗜Arsenal Image Mounter — the volume that isn't there

AIM (Arsenal Recon) is driver-backed: it presents an E01 / VHD / raw / VMDK to Windows as if it were a real SCSI disk. Unlike a software-only mount, the kernel sees an actual volume — so NTFS, BitLocker, VSS and every Windows-native tool just works on it.

  • Read Only — safe default. The image is exposed as a read-only volume. Hash stays intact. Use when the NTFS journal is clean.
  • Write Temporarily — Windows still thinks it can write; AIM intercepts every write into a sparse overlay file on your analysis disk. Dismount discards the overlay. The source E01 is never touched.
  • Why Write Temporarily matters: a dirty NTFS volume will refuse to mount until the journal replays. Read Only blocks that replay and Windows sulks. The overlay lets the replay happen — into the overlay — so the volume mounts cleanly.
  • Same applies to chkdsk, to VSS revert, and to anything that wants to touch the disk before showing you files.
ℹ️Always confirm the mode on the dialog. The overlay file is on your drive — make sure you have headroom for it (a busy NTFS can write hundreds of MB during journal replay).
⚙️KAPE — Targets & Modules, in that order

KAPE (Kroll Artifact Parser/Extractor, Eric Zimmerman) is a two-phase engine. Phase one collects, phase two parses. The two are deliberately decoupled so you can collect on the suspect host and parse on the lab box.

  • Target = a YAML recipe pointing at filesystem paths/files. e.g. KapeTriage, RegistryHives, EventLogs, WebBrowsers, WindowsExecutables. Targets ride RawCopy / RawAccessRead under the hood so locked system files (SAM, $MFT, EVTX in use) come out anyway.
  • Module = a YAML recipe wrapping a parser CLI — MFTECmd, EvtxECmd, RECmd, PECmd, AmcacheParser, JLECmd, SBECmd, LECmd. Modules consume what Targets produced.
  • Triage set = the KapeTriage compound target. It's a curated bundle pulling ~90% of what an IR examiner actually opens. One flag, one folder, full sweep.
  • gKape = the GUI front-end. Same YAML logic, point-and-click. The CLI is what shows up in the report.
🧭Workflow — E01 → AIM → KAPE → parsed output

The taught flow, end to end:

  • Mount the E01 with AIM. Pick Read Only first; if the volume refuses to come online, dismount and retry as Write Temporarily.
  • Note the drive letter Windows assigned — say F:. That becomes KAPE's --tsource.
  • Run KAPE with a Target (collection only) to produce a triage tarball.
  • Run KAPE with a Module (parsing only) against the collected tarball — or chain both in one invocation.
  • Output goes to a working folder on your analysis disk, never back onto the mounted image.
Collect the full triage set from a mounted E01
kape.exe --tsource F: --target KapeTriage --tdest C:\Triage\out --tflush
Parse a previously-collected tarball with the EZ-tools module bundle
kape.exe --msource C:\Triage\out --module !EZParser --mdest C:\Triage\parsed --mflush
Live host, walk Volume Shadow Copies too
kape.exe --tsource C: --target KapeTriage --tdest E:\out --vss
Chained — collect & parse in one run
kape.exe --tsource F: --target KapeTriage --tdest C:\T\out --module !EZParser --mdest C:\T\parsed --tflush --mflush
⚠️--vss multiplies runtime & output. Every shadow copy gets walked as if it were its own volume. Worth it for recovering deleted artefacts, but plan disk space (×N copies) and clock-time accordingly.
🛑On a live host, never --tdest back onto the system drive. Aim the output at an external/network volume — otherwise KAPE's own writes pollute the very artefacts it's collecting.

📦Why it matters — the triage tarball

Instead of carving one artefact at a time across an hour-long session, KAPE produces a single, reproducible triage tarball — a folder structure mirroring the source paths, every interesting file in its place.

  • Reproducible — the YAML targets are the recipe. Anyone with the same KAPE version gets the same output.
  • Audit-trailed — KAPE writes a _kape.cli + log per run, listing every file it touched and how.
  • Idempotent — re-running on the same source produces the same tarball (timestamps aside).
  • Hand-offable — zip the tarball, hand it to the next examiner. Their first task isn't "collect"; it's "analyse".
🧨Common traps in the classroom
  • Mounting a dirty volume as Read Only → Windows refuses, you think the image is corrupt. Re-mount as Write Temporarily.
  • Running KAPE from inside the mounted volume — don't. KAPE binaries on the analysis disk, output on the analysis disk, source = the mount.
  • Forgetting --tflush / --mflush — leaves stale output from previous runs. Use them when starting clean.
  • Spaces in paths — quote them. --tdest "C:\My Cases\out".
  • Antivirus on the analysis host will quarantine EZ-tools (Eric Zimmerman's parsers). Exclude the KAPE folder before the run.
  • Mismatched casing between Target and Module names — KAPE is forgiving on Windows, but log it correctly in the report.
Cross-references for corroboration

KAPE is upstream of nearly every later section — its tarball is the substrate the rest of the course parses. Pin each artefact back to where it gets analysed in depth:

  • $MFT / $LogFile / $UsnJrnl → MFTECmd & the NTFS internals deep-dive in §04. The same $MFT KAPE pulls is what timeline reconstruction is built on.
  • Registry hives (SAM / SYSTEM / SOFTWARE / SECURITY / NTUSER / UsrClass) → §06–§09. SAM logon counts cross-check against Security.evtx 4624/4625.
  • Prefetch & AmCache → execution evidence in §10. Prefetch run-times corroborate AmCache install dates and Security.evtx process-creation events.
  • LNK / JumpLists / ShellBags → user-activity reconstruction in §12–§13. LNK target IDs reference ShellBag GUIDs — two independent witnesses to the same folder visit.
  • EventLogs vs SAM — Security.evtx 4624 events (logons) corroborate against SAM's per-account last-logon timestamp (§07). Disagreement = log tampering signal.
  • Browser history vs ShellBags — a downloaded file shows up in Chrome/Firefox history and in the Downloads ShellBag and in a LNK in Recent. Three witnesses to one event.
  • --vss output ↔ file carving (§05) — shadow copies recover older versions of artefacts; carving recovers unallocated remnants. Together they reach files that no single source still has.
  • Hiberfil / Pagefile → fold straight into memory forensics (§02) — Volatility can ingest both.
