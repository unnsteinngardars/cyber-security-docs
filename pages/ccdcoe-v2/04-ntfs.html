📐NTFS fundamentals — sectors, clusters, volumes, files

NTFS is Windows' native filesystem. The taught hierarchy, smallest to largest:

  • Sector — the smallest addressable unit on disk. Classically 512 B; modern Advanced Format disks present 4 KB.
  • Cluster — NTFS's smallest allocation unit, sized at format. Default 4 KB = 8 × 512 B sectors. Every file consumes a whole-cluster multiple — the leftover is slack (§05).
  • Volume — one NTFS instance, addressed by its $Boot sector and Volume Serial Number (recorded in $Volume).
  • File — every object: regular files, directories (just files whose payload is $INDEX_ROOT), and even the filesystem metadata itself.

The defining NTFS truth: everything is a file. The MFT, the bitmap, the journal — each is an entry in the MFT, indexable like any other file. That self-referential layout is what makes NTFS so forensically rich.

🗃The $MFT — the spine of the filesystem

The Master File Table is an array of fixed-size 1024 B records. One record per file (including itself — record 0). Each record is a header followed by a sequence of typed attributes.

  • Records carry a sequence number incremented on each reuse — the pair record-seq is how a file reference (FRN) survives deletion.
  • If a file's attributes won't fit in 1024 B, an $ATTRIBUTE_LIST in the base record points at extension records.
  • Records can be marked unallocated; the contents persist until reused, so MFTECmd will surface them as deleted entries.
  • The MFT itself is non-resident — its own $DATA runs map to clusters across the volume.

The $MFT is the index of everything that exists, ever existed, or briefly existed on the volume.

🧩MFT record attributes — what lives inside a 1024-byte record

Every MFT record is a header plus a sequence of attributes, each with its own type ID, length, and resident/non-resident flag. The set you'll see on virtually every record:

📦Resident vs non-resident $DATA

NTFS stores file contents in the $DATA attribute. Where it lives depends on size:

  • Resident — payload fits inside the MFT record itself (practically ≤ ~700 B). No clusters allocated. The file is the MFT entry.
  • Non-resident — header stays in the record; payload lives in clusters described by compact data runs (LCN, length pairs). The MFT entry is a map, not the data.

Why this matters: when a tiny file is deleted, its record is flagged unused but the bytes persist verbatim inside $MFT until that slot is recycled. Run MFTECmd on a fresh $MFT dump and you will routinely recover small ransom notes, scripts, config snippets and credential blobs in full — no carving required.

Conversely, a non-resident file's clusters can be overwritten while the MFT record still describes them — cross-check $Bitmap before trusting recovered data runs.

🕒MACB & timestomping detection

NTFS carries two sets of four timestamps per file:

  • Modified — last content change
  • Accessed — last open / read (often disabled on modern Windows)
  • Changed — last MFT-record metadata change
  • Born — file creation time

The visible set lives in $STANDARD_INFORMATION ($SI). The hidden set lives in $FILE_NAME ($FN) and only updates on file create / rename / move.

Timestomping tools call SetFileTime to rewrite $SI — making malware look weeks old. They almost never touch $FN. The asymmetry is the detection.

🎯Timestomping rule of thumb: flag any file where $SI.Born > $FN.Born, or where $SI sub-second component is .0000000 while $FN shows microsecond precision. SetFileTime rounds; the kernel doesn't.
🪞Alternate Data Streams (ADS)

An ADS is simply a second (or third, or fourth) $DATA attribute on the same MFT record, distinguished by a stream name. Reference syntax: filename:streamname:$DATA.

  • Legitimate: :Zone.Identifier — Mark-of-the-Web. Written by browsers / mail clients on downloaded files; SmartScreen reads it.
  • Abused: hiding executables, scripts, exfil archives behind innocuous host files. notepad.exe c:\tmp\cover.txt:payload.ps1 works; dir shows nothing.
  • Discovery: live host → dir /R, Get-Item -Stream *. Image → MFTECmd dumps every named stream with size and resident/non-resident flag.
  • An ADS has its own resident/non-resident behaviour but shares the host file's $SI MACB — the host's timestamp covers the hidden payload.
🧊Volume Shadow Copies (VSS)

VSS are point-in-time snapshots stored at \System Volume Information\. Each snapshot is a copy-on-write delta — together they expose the past state of every file on the volume, often weeks back.

  • Live host: vssadmin list shadows then mklink /D C:\vss1 \\?\GLOBALROOT\Device\HarddiskVolumeShadowCopy1\ to walk it as a folder.
  • Image: mount with AIM, or feed KAPE the --vss switch (§03) — KAPE re-runs every target against each snapshot.
  • Each snapshot has its own $MFT; diff today's MFTECmd output against a snapshot's to spot files that existed and were then wiped.

VSS is the bridge between this section and file carving (§05): when the live FS has nothing left, the snapshot often still does.

⚙️MFTECmd — the canonical workflow

MFTECmd (Eric Zimmerman) parses $MFT, $LogFile, $Boot, $SDS and $UsnJrnl:$J into clean CSV / JSON. It is the taught tool and the de-facto standard. Run it against extracted files — never against a mounted live drive.

  • Output columns of interest: EntryNumber, SequenceNumber, ParentEntryNumber, InUse, IsDirectory, HasAds, all four $SI + all four $FN timestamps, FileSize, ZoneIdContents.
  • Pipe the CSV into Timeline Explorer for a sortable, filterable view; export to plaso/log2timeline for super-timeline merging (§13).
Full $MFT extraction → CSV (the canonical run)
MFTECmd.exe -f F:\$MFT --csv C:\out --csvf mft.csv
Decode a single MFT entry (record-seq)
MFTECmd.exe -f F:\$MFT --de 12345-1 # entry 12345, sequence 1
Parse the $LogFile transactional journal
MFTECmd.exe -f F:\$LogFile --csv C:\out
Parse the USN change journal (rename / create / delete history)
MFTECmd.exe -f F:\$Extend\$UsnJrnl:$J --csv C:\out
Walk a VSS snapshot's $MFT
MFTECmd.exe -f \\?\GLOBALROOT\Device\HarddiskVolumeShadowCopy1\$MFT --csv C:\out --csvf mft_vss1.csv
✅Why it matters: the $MFT is the index of everything. Combined with $LogFile and $UsnJrnl, you get a recent-operations timeline that nothing on the live FS exposes — including deletes the user thought were permanent.
Cross-references for corroboration

NTFS artefacts rarely stand alone — they corroborate (or contradict) every other source on the disk:

  • Timestomping detection — $SI MACB vs $FN MACB on the same record, plus the corresponding $UsnJrnl entry. If $SI says "born 2019" but USN shows FILE_CREATE last week, it's stomped.
  • VSS-mounted older $MFT vs current $MFT — diff entry sets to surface files that existed in a snapshot and were later wiped from the live volume.
  • Prefetch / AmCache (§10) — the executable's last-run time should align with its $SI.Modified; a Prefetch entry for a binary missing from the MFT means it was deleted post-execution.
  • LNK TrackerDataBlock (§12) — the GUID embedded there must match the target file's $OBJECT_ID attribute, even if the target was renamed.
  • Registry hive LastWriteTime (§06) — for a hive file like SYSTEM, the key's LastWriteTime should be ≤ the file's $SI.Modified; mismatch suggests tampering with the on-disk hive.
  • Mark-of-the-Web — :Zone.Identifier ADS on a dropped binary places it on a downloaded path; absence on a "downloaded" file is itself suspicious.
