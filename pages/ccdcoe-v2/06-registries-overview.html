🌲The registry as a hierarchical database

The registry is a tree of keys (folder-like nodes) containing values (name + type + data). Keys can have subkeys. The whole structure is a single hierarchical database fragmented across several on-disk hives.

  • HKLM — machine-wide config (hardware, OS, installed software, services).
  • HKU — one subtree per loaded user profile, keyed by SID.
  • HKCU / HKCR / HKCC — virtual views. HKCU is a pointer at HKU\<current-SID>; HKCR is a merge of HKLM\Software\Classes + HKU\<SID>_Classes; HKCC mirrors a SYSTEM subkey. Nothing lives on disk under these names.
  • Value types you will see: REG_SZ, REG_EXPAND_SZ, REG_MULTI_SZ, REG_DWORD, REG_QWORD, REG_BINARY, REG_NONE.

LastWriteTime — every key carries a single 64-bit FILETIME, updated whenever any value inside it changes. Individual values have no timestamp of their own. This is the only built-in temporal anchor the registry gives you — treat it as gold and use the surrounding key context to scope what actually fired.

🧱regf — what a hive file actually is

Each hive is a self-contained regf container with a fixed layout. Knowing this matters when the parser chokes or you need to carve.

  • Header (4 KB) — regf magic, sequence numbers, root cell offset, last-written timestamp, checksum.
  • hbins — 4 KB allocation blocks, each prefixed with hbin.
  • Cells inside hbins, signed length (negative = allocated, positive = free):
    • nk = key node (carries LastWriteTime, parent/child pointers, security ref).
    • vk = value (name, type, data offset).
    • sk = security descriptor (shared between keys).
    • lh / lf / ri / li = subkey list indexes.
    • db = big-data, for values > 16 KB.

Free / unallocated cells are the source of deleted-key recovery — they sit in the hbin's free list until the allocator overwrites them.

📝Transaction logs & dirty hives

Every hive ships with sibling transaction logs: SYSTEM, SYSTEM.LOG1, SYSTEM.LOG2 (and historically .LOG). Windows writes pending changes to the logs first, then folds them into the hive at flush time.

A dirty hive is one whose in-memory transactions were not yet flushed to the primary file — typical when you image a running box, when the OS crashed, or when you pulled the plug. Reading the primary file alone gives you a stale snapshot.

  • Registry Explorer / RECmd auto-detect and replay .LOG1 / .LOG2 on load.
  • The hive header's primary & secondary sequence numbers being unequal is the dirty-bit signal.
  • Always export the hive and both logs from KAPE / FTK Imager — they are useless on their own but mandatory when combined.
⚠️Stale-hive trap: dropping a dirty NTUSER.DAT into a parser without its .LOG1 / .LOG2 can cost you the last several hours of user activity — exactly the window an examiner usually cares about. Always grab the logs.
🗑Deleted keys, unallocated cells, slack

When a key or value is deleted, its cell length is flipped positive and the cell is linked into the hbin's free list. The cell contents typically remain intact until the allocator hands the space to a new write — directly analogous to NTFS file deletion (§04).

  • Registry Explorer surfaces orphaned cells in the Deleted Keys / Deleted Values view (rebuilds parent chains from nk back-pointers where possible).
  • RECmd exposes the same via --Recover True.
  • Recovered names & data are usually verbatim; recovered LastWriteTime is unreliable — it is the parent key's stale value, not the moment of deletion.
  • Slack at end of hbins occasionally yields fragments of older value data — useful for keyword carving, not for whole-key reconstruction.
🔤MRUList & MRUListEx — ordering

Many Explorer / Office / dialog artefacts store an ordered "Most Recently Used" list as a bag of letter-named values plus one special ordering value:

  • Values a, b, c, … each carry one entry (a path, a typed string, a binary PIDL).
  • MRUList (legacy, REG_SZ) — a string like cab meaning most recent first: c then a then b.
  • MRUListEx (Vista+, REG_BINARY) — sequence of little-endian DWORD indexes, terminated by FFFFFFFF, most recent first.

Without the MRUList value the rest is just an unordered bag. The ordering value is what turns a list of files into a timeline of usage. Registry Explorer plugins decode MRUListEx automatically; raw reg query does not.

🛠The toolchain — what each does

Registry Explorer (RegistryExplorer.exe, Eric Zimmerman) — GUI. Drag in a hive; it auto-locates .LOG1 / .LOG2 siblings and replays them. Plugins normalise common artefacts (UserAssist ROT13-decoded, ShellBags parsed, TypedURLs sorted) into a flat table view. Use Find for value-name keyword sweeps, Bookmarks for case notes. Best for interactive exploration.

RECmd (RECmd.exe, Eric Zimmerman) — the CLI sibling. Batch-driven via .reb files in RECmd\BatchExamples\. Kroll_Batch.reb is the canonical do-everything sweep; SYSTEM_Batch.reb / SAM_Batch.reb / NTUSER_Batch.reb are hive-targeted. Drops cleanly into KAPE (§03) as the RECmd_* module set.

RegRipper (rip.exe / rr.exe, Harlan Carvey) — older Perl plugin runner. One plugin per artefact: samparse, userassist, services, appcompatcache, compname, shellbags, etc. Slower, but invaluable as a second opinion and for niche plugins the EZ tools don't ship.

RECmd — Kroll batch sweep across whole config dir
RECmd.exe -d F:\Windows\System32\config --bn RECmd\BatchExamples\Kroll_Batch.reb --csv C:\out
RECmd — single hive, targeted batch
RECmd.exe --hive F:\Windows\System32\config\SYSTEM --bn RECmd\BatchExamples\SYSTEM_Batch.reb --csv C:\out
RECmd — include deleted keys / values in output
RECmd.exe --hive F:\Users\bob\NTUSER.DAT --bn RECmd\BatchExamples\NTUSER_Batch.reb --csv C:\out --Recover True
RegRipper — UserAssist from NTUSER
rip.exe -r F:\Users\bob\NTUSER.DAT -p userassist
RegRipper — run every plugin auto-mapped to hive type
rip.exe -r F:\Windows\System32\config\SOFTWARE -a > software.txt

✓Plugins, briefly: a plugin is a normalised viewer — it reads the raw key/value bytes and renders them as a table with derived columns (ROT13-decoded names, FILETIME-decoded run counts, parsed shellbag PIDLs). Without the plugin you stare at a binary blob; with it, you read a row.
Cross-references for corroboration

The registry never stands alone — its findings must be anchored against the filesystem and against the rest of the hive ecosystem:

  • NTFS $STANDARD_INFORMATION (§04) — the hive file's MFT timestamps should bracket the hive header's last-flush timestamp. A wildly newer $SI mtime than the internal flush time means somebody touched the hive offline.
  • User folder ↔ SID (§07 SAM) — match C:\Users\<user> to the SID owning the NTUSER.DAT via SAM's F/V values or SOFTWARE\…\ProfileList.
  • Deleted-key recovery here ↔ $MFT deleted-file recovery (§04 / §05) — same allocator-reuse logic, same "name & data survive, timestamps don't" caveat.
  • Application Execution (§10) — BAM lives in SYSTEM, UserAssist & RunMRU in NTUSER.DAT; cross-check a single execution event across both for confidence.
  • Cloud storage (§11) & ShellBags (§13) — both are pure registry artefacts (OneDrive/Dropbox keys in SOFTWARE + NTUSER; shellbags in UsrClass.dat) — they ride on the same hive-loading workflow taught here.
  • Memory image (§02) — Volatility's registry plugins can pull hives out of RAM and may surface keys that were written but never flushed to disk.
