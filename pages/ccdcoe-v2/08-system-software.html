🪪Why pair SYSTEM with SOFTWARE

Together these two hives produce the machine's passport. SYSTEM describes the hardware and kernel-level configuration the OS was booted with; SOFTWARE describes the OS build itself and everything installed on top. Read in tandem, in five minutes you can answer: what is this box, who owns it, where does it live, what does it run, what does it talk to.

  • SYSTEM — drivers, ControlSets, services, NIC IP config, USB device history, timezone, shutdown timestamp, mounted volumes.
  • SOFTWARE — Windows version + build + UBR, install date, registered owner, installed applications, machine-wide autoruns, every Wi-Fi network ever joined, app permissions, Defender exclusions.
  • On a live host both live under %SystemRoot%\System32\config\ and are locked. Capture them with FTK Imager → Obtain Protected Files (§01).
🔀ControlSets — and why CurrentControlSet isn't on disk

The SYSTEM hive holds multiple parallel kernel configurations called ControlSets — usually ControlSet001 and ControlSet002. A small key, SYSTEM\Select, holds four pointer values:

  • Current — which ControlSet the running OS booted with.
  • Default — which one would be used on next normal boot.
  • Failed — last ControlSet that failed to boot.
  • LastKnownGood — last ControlSet that booted to a successful logon.

SYSTEM\CurrentControlSet is a virtual mount Windows builds at boot from whichever ControlSet Select\Current points at. It does not exist on disk. When you open the hive offline you walk ControlSet001 (or follow Select\Current).

The other ControlSet is the previous boot's config — invaluable when malware altered a service or driver during the most recent boot: compare 001 vs 002.

⛔Offline gotcha: in Registry Explorer with a dead-box SYSTEM hive there is no CurrentControlSet node. Read Select\Current first, then walk that ControlSet — don't assume 001.

🧩Registry Explorer plugins — raw blobs → tables

Plugins are community-maintained .dlls in Registry Explorer\Plugins\. Each implements IRegistryPluginBase for a specific subkey and re-formats its raw values into a readable grid.

  • Click an instrumented subkey (e.g. NetworkList\Profiles, USBSTOR, Services) — Registry Explorer shows the parsed table in the lower pane instead of hex blobs.
  • Each plugin grid is CSV-exportable straight into your case notes.
  • Without plugins: NetworkList DateLastConnected is a SYSTEMTIME byte blob. With the plugin: a real timestamp column.
  • Keep the Plugins\ folder up-to-date — Eric Zimmerman ships new ones with each release.

RECmd is the same idea, automated and batched: a .reb batch file lists "plugin X against key Y" lines and emits one CSV per plugin across the whole hive.

🎯OS profiling — 5-minute workflow

Goal: from a freshly mounted image, produce a one-page profile of the machine before opening any other artefact.

  • Hostname → SYSTEM ControlSet001\Control\ComputerName\ComputerName
  • Timezone → SYSTEM ...\Control\TimeZoneInformation (do this first)
  • OS + build + UBR + install date → SOFTWARE ...\Windows NT\CurrentVersion
  • Last clean shutdown → SYSTEM ...\Control\Windows\ShutdownTime
  • IP config / DHCP lease → SYSTEM Services\Tcpip\Parameters\Interfaces\<GUID>
  • Networks joined → SOFTWARE NetworkList\Profiles + Signatures\Unmanaged
  • USB history → SYSTEM Enum\USBSTOR + MountedDevices
  • Installed apps → SOFTWARE ...\Uninstall (+ Wow6432Node)
  • Autoruns → SOFTWARE ...\Run, RunOnce, StartupApproved\Run + SYSTEM Services
  • Defender posture → SOFTWARE Windows Defender\Exclusions

Either click through with RE + plugins, or batch the lot with RECmd — both produce the same answers. The CSV output is your profile sheet.

💻CLI — batch the whole hive
RECmd — SYSTEM hive batch
RECmd.exe --hive F:\Windows\System32\config\SYSTEM --bn RECmd\BatchExamples\SYSTEM_Batch.reb --csv C:\out
RECmd — SOFTWARE hive batch
RECmd.exe --hive F:\Windows\System32\config\SOFTWARE --bn RECmd\BatchExamples\SOFTWARE_Batch.reb --csv C:\out
RegRipper — targeted SYSTEM plugins
rip.exe -r F:\Windows\System32\config\SYSTEM -p compname,timezone,shutdown,services,networklist,usbstor,mountdev
RegRipper — targeted SOFTWARE plugins
rip.exe -r F:\Windows\System32\config\SOFTWARE -p winver,uninstall,run,networklist,defender,wlan_events
Find the active ControlSet first (offline)
# Registry Explorer → SYSTEM\Select → read Current value (1 → ControlSet001, 2 → ControlSet002)
Sanity check — compare 001 vs 002
RECmd.exe --hive SYSTEM --kn "ControlSet001\Services" --csv C:\out\cs1 && same for ControlSet002 → diff for malware that touched Services last boot
✅Workflow: RECmd batch first to dump everything, then Registry Explorer + plugins for the handful of keys you need to inspect interactively. Batch for breadth, GUI for depth.
Cross-references for corroboration

SYSTEM and SOFTWARE answer "what is this box". Almost every other artefact in the case is interpreted through them:

  • TimeZoneInformation (SYSTEM) — required to convert every other timestamp in the case to wall-clock. Resolve this before anything else.
  • ShutdownTime (SYSTEM) ↔ §01 acquisition log: last clean shutdown should align with when the examiner pulled the plug.
  • USBSTOR (SYSTEM) ↔ MountedDevices (SYSTEM) ↔ Portable Devices (SOFTWARE) — same device, three views: kernel enumeration, volume-letter binding, user-facing friendly name.
  • Services (SYSTEM) ↔ Run / RunOnce (SOFTWARE) ↔ Scheduled Tasks (§03 / §10) — the three classic persistence triads. Anomaly in any of them is the hook.
  • Tcpip\Interfaces (SYSTEM) ↔ NetworkList Profiles + Unmanaged Signatures (SOFTWARE) — DHCP lease timestamps plus SSID plus AP MAC pin the machine to a place and a moment.
  • Uninstall (SOFTWARE) ↔ Application Execution evidence (§10 — Prefetch, AmCache, ShimCache). "Installed" vs "actually executed" is a real difference.
  • Defender Exclusions (SOFTWARE) ↔ any suspect paths discovered in §10. Exclusion added shortly before malware first ran is a powerful narrative beat.
  • OS build + UBR (SOFTWARE) ↔ CVE / advisory feeds — patch level decides which exploits were viable on the date of compromise.
