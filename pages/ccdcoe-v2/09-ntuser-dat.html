🔍WordWheelQuery — Start menu / Explorer search
Software\Microsoft\Windows\CurrentVersion\Explorer\WordWheelQuery

Every term the user typed into the Start menu search box or the File Explorer search bar lands here.

  • MRUListEx — REG_BINARY, sequence of little-endian DWORDs giving the order (most-recent first).
  • Numbered values 0, 1, 2, … each a UTF-16-LE search string.

Forensic value: shows what the user looked for. Filenames the user knew but couldn't find — strong tell for hidden files, deleted artefacts, or attacker tooling the user was hunting (or returning to).

🧭TypedPaths — Explorer address bar
Software\Microsoft\Windows\CurrentVersion\Explorer\TypedPaths

Paths the user typed into the Explorer address bar — not paths they clicked into.

  • Values url1, url2, … most recent = url1.
  • UTF-16-LE strings, local paths or UNC.

Forensic value: deliberate navigation, not browsing. UNC entries (\\server\share) here are gold — explicit network access the user chose to perform, not a drive-letter click.

📄RecentDocs — files the user opened
Software\Microsoft\Windows\CurrentVersion\Explorer\RecentDocs

Per-extension subkeys (.docx, .jpg, .pdf, …), each with its own MRUListEx and numbered binary values containing the filename plus a PIDL into the shell. The top-level RecentDocs aggregates across extensions.

  • Records file opening, not execution.
  • Entry survives the file being deleted afterward — proof of access.
  • Subkey last-write time ≈ the time of the most recent file of that extension.

Pair with §12 LNK files in %APPDATA%\Microsoft\Windows\Recent — the same open event materialises in both places.

⌨RunMRU — Win+R history
Software\Microsoft\Windows\CurrentVersion\Explorer\RunMRU

Every string the user typed into the Win+R Run dialog.

  • Values a, b, c, … each a UTF-16 string.
  • MRUList — ASCII string like gdcba giving the order, most-recent first (left-most letter wins).

Forensic value: one of the cleanest user-intent artefacts on the whole system. cmd, powershell, \\attacker\share, %temp%\loader.exe — if it's here, a human typed it and pressed Enter. Cross-reference with §10 (UserAssist / Prefetch / BAM) to prove the typed command actually executed.

📑Office File MRU — Word / Excel / PowerPoint / Outlook
Software\Microsoft\Office\<version>\Word\User MRU\LiveId_<hash>\File MRU

Same structure under Excel, PowerPoint, Outlook. <version> is 16.0 for current Office.

  • Values Item 1, Item 2, … each a FILETIME prefix + full path of the document.
  • Paths can be local, UNC, OneDrive HTTPS URL, or SharePoint URL — Office tracks wherever it opened from.
  • Place MRU subkey — folder paths recently picked as save targets.

Forensic value: distinct from RecentDocs because it covers network / cloud sources and carries a per-entry timestamp Explorer doesn't keep.

⚠TrustRecords — "Enable Content" clicks
Software\Microsoft\Office\<version>\<app>\Security\Trusted Documents\TrustRecords

One value per file whose macros or external content the user explicitly trusted. The artefact for "user clicked Enable Content on a malicious macro."

  • Value name = the file URL: file:///, http://, smb://.
  • Value data ends with a flag DWORD:
    • 7FFFFFFF — user clicked Enable Content (macros enabled).
    • 01000000 — Enable Editing only (Protected View dismissed, macros not enabled).
  • Entry survives deletion of the underlying file.
⚠️Flag-byte caveat: the 7FFFFFFF / 01000000 interpretation is the modern Office convention. Older Office builds (and some out-of-band patches) shift the meaning. Always record the raw hex verbatim in the report and cite the Office version from the parent key — don't paraphrase the click as "macros enabled" without the bytes to back it.
💾ComDlg32 — the common Open / Save dialog
Software\Microsoft\Windows\CurrentVersion\Explorer\ComDlg32

Every time any application uses the Windows common Open or Save dialog, three sub-artefacts get written. Even a cancelled save leaves intent here.

  • OpenSavePidlMRU — per-extension subkeys (.docx, .exe, …) plus a catch-all *. Each value is a PIDL (shell ID list) describing a file recently picked in any Open or Save dialog. Decoded, you recover the full path and a reference to the shell location at click time — including removable / network volumes that may no longer be attached.
  • LastVisitedPidlMRU — one entry per executable that opened a common dialog. Maps an .exe name to the folder it last opened a dialog in. Tells you "the last folder Word saw" or "the last folder mshta.exe browsed".
  • CIDSizeMRU — the dialog window size each application requested. Low forensic value alone, but proves an app was launched at least once and opened a dialog.

Forensic value: even if the file was never actually saved, the intent to save or open it lives here until 25–26 newer entries overwrite it. Picking a file in a phishing attachment's "Save As" dialog leaves a tombstone here long after the file is gone.

⚙Parsing — tooling & CLI

Registry Explorer plugins cover every artefact above out of the box — WordWheelQuery, TypedPaths, RecentDocs, RunMRU, OfficeMRU, TrustRecords, ComDlg32. PIDLs are decoded to human-readable paths in the Plugins panel.

RECmd (Eric Zimmerman) — batch-mode parser that fires every relevant plugin in one pass and dumps a CSV per artefact. Already introduced in §06.

RegRipper — perl-based, fast for triage. Relevant plugins: userassist, recentdocs, runmru, typedpaths, wordwheelquery, comdlg32, trustrecords, officedocs.

RECmd — all NTUSER artefacts in one pass
RECmd.exe --hive F:\Users\bob\NTUSER.DAT --bn RECmd\BatchExamples\NTUSER_Batch.reb --csv C:\out
RegRipper — targeted plugin run
rip.exe -r F:\Users\bob\NTUSER.DAT -p recentdocs,runmru,typedpaths,wordwheelquery,officedocs,trustrecords,comdlg32

# File → Load offline hive → NTUSER.DAT → expand Plugins panel → click each artefact

Cross-references for corroboration

NTUSER findings only become evidence once tied to a specific human and a specific action that actually executed. Pair every artefact below with at least one other source:

  • Who is this user? — §07 SAM → SID → SOFTWARE\Microsoft\Windows NT\CurrentVersion\ProfileList → profile path → the NTUSER.DAT you parsed. No SID match, no attribution.
  • TypedPaths UNC entries — cross-reference with MountedDevices (§08) and with Software\Microsoft\Windows\CurrentVersion\Explorer\Map Network Drive MRU to confirm the share was actually mounted, when, and under which drive letter.
  • RecentDocs entries — PIDLs resolve through ShellBags (§13) for full reconstructed paths, and pair with LNK files in %APPDATA%\Microsoft\Windows\Recent (§12) which carry MAC times and target volume serials.
  • RunMRU + TrustRecords + Office File MRU — pair with §10 Application Execution (Prefetch, UserAssist, BAM/DAM). Build a single timeline: user typed → app launched → file opened → macro enabled.
  • ComDlg32 OpenSavePidlMRU — cross-reference with files in $Recycle.Bin (§05). The file the user picked in the dialog may already be deleted; the PIDL is its tombstone, and the recycle-bin $I record gives the deletion timestamp.
  • TrustRecords URL — should match a file in RecentDocs, an LNK in Recent (§12), and (for emailed lures) an attachment artefact / browser download history. The closer those three timestamps cluster, the cleaner the phishing-to-execution story.
