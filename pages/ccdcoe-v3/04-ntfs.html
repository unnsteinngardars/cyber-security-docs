📐Why NTFS artefacts are so rich

NTFS is self-describing: its own metadata is stored as files (records 0–15 of the MFT), each named with a leading $. That self-referential design means an examiner can read the filesystem the same way the kernel does — and recover information the kernel itself has discarded. Below, each $file is treated as an artefact with its own forensic value.

The hierarchy: sector (512 B or 4 KB on AF disks) → cluster (default 4 KB, the allocation unit — leftover bytes are slack) → volume (one NTFS instance, anchored at $Boot) → file (anything with an MFT record, including directories and the filesystem's own metadata).

🗃$MFT — the index of everything
Two MACB sets Deleted entries Anchor

Where: record 0 of itself; physically a non-resident file mapped by $DATA runs.

Structure: array of 1024-byte records, each a header + sequence of typed attributes. EntryNumber + SequenceNumber is the immutable file reference (FRN).

Answers:

  • Every file that exists or has existed on the volume — unused records persist verbatim until reused.
  • Two independent MACB sets per file ($SI + $FN) for timestomping detection.
  • Tiny files (≤ ~700 B) live entirely inside the record — recovered in full even after deletion.
  • Parent-FRN field rebuilds the full path tree.

Cannot tell you: contents of non-resident files whose clusters were reused; runs without $Bitmap corroboration may point at overwritten data.

📜$LogFile + $UsnJrnl — the journals
Recent operations Names of wiped files

Where: $LogFile at MFT record 2; $UsnJrnl:$J inside $Extend.

Answers:

  • $LogFile — transactional log of NTFS metadata operations (allocations, attribute writes). Replay reconstructs recent create / delete / rename activity even when the MFT entry is gone.
  • $UsnJrnl:$J — name-level audit. Each entry: filename + parent FRN + USN reason code (FILE_CREATE, RENAME_OLD_NAME, FILE_DELETE, …). Often the only place a wiped filename survives.

Cannot tell you: events older than the journal's rotation horizon — these are ring buffers sized to a few hundred MB on a typical workstation (minutes to days of history).

🕒MACB rule: $SI is what the UI shows, what SetFileTime rewrites. $FN is what the kernel writes on create/rename/move. Divergence = timestomping. $SI.Born > $FN.Born or $SI sub-second = .0000000 while $FN shows microseconds → flag.
📦Resident vs non-resident $DATA
Full-file recovery

NTFS stores file contents in $DATA. Where it lives depends on size:

  • Resident — payload inline in the 1024 B record. No clusters allocated. The MFT entry is the file.
  • Non-resident — header stays; payload lives in clusters described by data runs (LCN, length). The record is a map.

Forensic value: when a tiny file is deleted, its full content persists inside $MFT until the slot is reused — small scripts, ransom notes, credential blobs recovered without carving. Non-resident files: clusters can be overwritten while the record still describes them → cross-check $Bitmap before trusting recovered runs.

🪞Alternate Data Streams (ADS)
Hidden payload Mark-of-the-Web

A second (or third…) $DATA attribute on the same record, distinguished by a stream name. Syntax: file:stream:$DATA.

  • Legitimate: :Zone.Identifier — Mark-of-the-Web, written by browsers / mail clients on downloaded files. Tells you the file came from outside.
  • Abused: hidden executables, scripts, exfil archives. dir shows nothing — only dir /R, Get-Item -Stream *, or MFTECmd reveals them.
  • Detail: an ADS has its own resident/non-resident flag but shares the host file's $SI MACB — the host's timestamps cover the hidden payload.
🧊Volume Shadow Copies (VSS) — historical filesystem snapshots
Historical Wiped files Diff baseline

Where: \System Volume Information\ on the live FS; exposed as \\?\GLOBALROOT\Device\HarddiskVolumeShadowCopyN\.

Structure: each shadow is a copy-on-write delta. Together they reconstruct the state of every file at each snapshot point — often weeks back.

Forensic value:

  • Every snapshot has its own $MFT — diff today's MFTECmd output against a snapshot's to surface files that existed and were later wiped.
  • Registry hives inside a snapshot expose the past state of services, persistence keys, user accounts.
  • KAPE's --vss re-runs every target against each snapshot — turns the triage tarball into a historical record.

Cannot tell you: changes between two snapshots that were undone before the next snapshot. Snapshots are not continuous — they are discrete points.

✅The bridge to §05 file carving: when the live FS no longer contains a file, the shadow often still does — try VSS before resorting to signature-based carving.

Cross-references for corroboration

NTFS artefacts are the substrate other artefacts reference — almost every later finding ties back to a field here:

  • $SI / $FN / $UsnJrnl triangle — three independent MACB witnesses; divergence is the timestomping signal.
  • $Volume VSN ↔ LNK TrackerDataBlock (§12) ↔ JumpLists (§13) ↔ ShellBags ↔ Event 1006 (§14) — the VSN is the join key that ties a file event to a specific physical volume / device.
  • $OBJECT_ID ↔ LNK TrackerDataBlock — the GUID embedded in a LNK file matches the target's $OBJECT_ID attribute, even after rename.
  • $Bitmap ↔ recovered data runs — confirms whether the bytes a deleted record points at are still intact or already reused.
  • VSS $MFT ↔ live $MFT — diff surfaces files that existed historically and were later removed.
  • $Secure ACLs ↔ Security.evtx 4663 — the descriptor says who could touch the file; the event log says who did.
  • Hive file $SI.Modified (§06 hive on disk) vs key LastWriteTime — registry-key write time should be ≤ hive file modify time; mismatch suggests on-disk hive tampering.
