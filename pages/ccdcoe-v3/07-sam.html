📐Why "who is a user" is a multi-hive question

SAM is named after the Security Account Manager — and the name is honest about its scope: SAM only describes local identities. Anything authenticated against a domain controller, Microsoft Entra, or a consumer Microsoft account leaves no account record in SAM at all. The full picture is stitched from at least four hives, and an examiner who reads SAM alone will miss every domain admin who ever touched the box.

The forensic question splits into three orthogonal axes: who could authenticate (SAM, IdentityStore, SECURITY\Cache), who actually did (SAM F-blob last-logon, Security.evtx, ProfileList timestamps), and what they were entitled to (Builtin\Aliases group membership, primary group RID, profile location). Map all three before drawing conclusions.

🗂Where account data actually lives
Identity Cross-hive

Where: identity is sharded across SAM, SOFTWARE, SECURITY and every user's NTUSER. SAM is local-only; the rest fills in the cloud and domain story.

  • C:\Windows\System32\config\SAM — local accounts & groups: usernames, hashes, flags, last logon.
  • SOFTWARE\Microsoft\IdentityStore\LogonCache + \IdentityStore\Cache — cached identities for MSA / Azure AD / school accounts. Display name, e-mail, last login.
  • SOFTWARE\Microsoft\Windows NT\CurrentVersion\ProfileList — every SID with a profile on disk → its ProfileImagePath (e.g. C:\Users\bob). The bridge from event-log SID to a folder you can grep.
  • SECURITY\Policy\Secrets — LSA secrets / cached domain credentials. Adjacent, encrypted, not SAM proper.
  • NTUSER.DAT\Software\Microsoft\OneDrive\Accounts\Personal (also Business1, Business2…) — per-user OneDrive metadata: e-mail, CID, sync root, last sync. Ties a Microsoft account to a Windows profile. Deep dive in §11.

Answers: "every identity that has authenticated to this host" — but only by triangulating all four locations. Cannot tell you: session content, just session existence; for that you need Security.evtx (§14) or LSASS (§02).

⚠️SAM is local-only. A username absent from SAM does not mean they never logged in. Domain and MSA users authenticate elsewhere; their footprint is in ProfileList, IdentityStore and Security.evtx.
🔑Inside SAM\Domains\Account\Users\<HEX-RID>
Local users Hashes

Where: each local user is a subkey named with the user's RID in hex, zero-padded to 8 digits. Built-ins are fixed; custom users start at 000003E8 (1000) and increment.

Structure:

  • 000001F4 = 500 — Administrator
  • 000001F5 = 501 — Guest
  • 000001F7 = 503 — DefaultAccount
  • 000003E8 = 1000 — first locally created user; subsequent +1.

Under each RID key there are two binary values that hold everything interesting:

  • F — fixed-length blob: last logon, last password change, last bad password attempt, account flags, failed-login count, logon count, primary group RID.
  • V — variable-length blob: username, full name, comment, home dir, LM hash, NT hash, machine SID prefix.

Answers: the offline-credential and last-logon questions for every local account that ever existed (deletions remove subkeys, but residual unallocated cells often survive). Cannot tell you: nothing about domain or MSA users — they will not have a RID subkey here at all.

👥Group membership — who is actually local admin
Entitlement Lateral path

Where: local groups live under SAM\Domains\Builtin\Aliases\<group-RID-hex>. The Members subkey lists member SIDs (binary). Cross-reference each SID against the per-user V blob (machine-SID prefix + RID) to resolve to a username; for domain SIDs you'll need the DC.

Answers:

  • 00000220 — Administrators. Anyone here is local admin. Domain SIDs here = domain users granted local admin.
  • 0000022B — Remote Desktop Users. Can RDP in. Pair with §08 TerminalServer keys.
  • 0000022E — Backup Operators. Can read any file via backup privilege — under-watched lateral path.

Cannot tell you: when a member was added — the Members key's LastWriteTime is the only timestamp, and it covers any modification to the membership set. Membership is fact, not intent: a SID added during the incident window — even if it is a known domain user — is a privilege-escalation artefact.

🧭Rule of thumb: if a SID appears in ProfileList but the matching RID isn't in SAM, you're looking at a domain or MSA logon. Pivot to IdentityStore + Security.evtx, not SAM.
⚠️Cached domain credentials in SECURITY\Cache are machine-encrypted (SYSKEY / BootKey). Mount the SYSTEM hive alongside SAM/SECURITY or decryption fails silently — you'll get a clean parse with garbage in every secret.
⚙️Tool invocations — parsing SAM & IdentityStore
Offline parse

Where: use the §06 tooling (Registry Explorer / RECmd / RegRipper) against the offline hives lifted from C:\Windows\System32\config\. Always mount the SYSTEM hive in the same session — SAM hashes and SECURITY secrets are bootkey-encrypted.

  • RECmd — SAM batch (parses F + V + aliases): RECmd.exe --hive F:\Windows\System32\config\SAM --bn RECmd\BatchExamples\SAM_Batch.reb --csv C:\out
  • RegRipper — samparse plugin: rip.exe -r F:\Windows\System32\config\SAM -p samparse
  • RECmd — IdentityStore LogonCache (MSA / Azure AD): RECmd.exe --hive F:\Windows\System32\config\SOFTWARE -kn "Microsoft\IdentityStore\LogonCache" --csv C:\out
  • RegRipper — ProfileList (SID ↔ ProfileImagePath): rip.exe -r F:\Windows\System32\config\SOFTWARE -p profilelist
  • Registry Explorer — GUI workflow: File → Load offline hive → SAM → expand Domains\Account\Users\<RID>; right-click value → "View data in hex editor" to verify F/V layout by eye.

Cannot tell you: nothing the hive doesn't already record. Tools just decode — they cannot reconstruct a deleted RID subkey beyond what registry unallocated-cell recovery (RECmd's deleted-record mode) surfaces.

🪪IdentityStore & OneDrive — naming the human behind the SID
Identity resolution Cloud ↔ local

Where: a SID in ProfileList tells you "someone with this SID has a profile here" — not who. Two registry locations turn the SID into a person, plus per-user OneDrive metadata clinches the cloud-account binding.

Structure:

  • SOFTWARE\Microsoft\IdentityStore\LogonCache\<version>\Name2Sid — display name + e-mail per cached identity. Resolves Microsoft accounts and Azure AD users that have ever signed in.
  • SOFTWARE\Microsoft\IdentityStore\Cache\<SID>\IdentityCache — same identity data keyed by SID. Useful when you start from the SID end.
  • NTUSER.DAT\Software\Microsoft\OneDrive\Accounts\Personal — under that user's NTUSER: e-mail, CID, UserFolder, LastSignInTime. Tightly couples a Microsoft account identity to a specific Windows profile.
  • \Business1, \Business2, … — same shape for OneDrive for Business / school tenants. Multiple work tenants stack as separate subkeys.

Answers: triangulate ProfileList SID → IdentityStore e-mail → NTUSER OneDrive e-mail. Three sources, one human. Mismatch = investigate (profile takeover, manual SID injection, hive transplant).

Cross-references for corroboration

SAM is the identity backbone. Almost every other artefact references a user by SID or username — but you can only resolve those once SAM + ProfileList + IdentityStore are mapped. The join keys to remember:

  • ProfileList SID ↔ C:\Users\<name> folder ↔ NTUSER.DAT — the three-way handshake that lets every per-user artefact (§09) be attributed to a human.
  • SAM F last-logon ↔ Security.evtx 4624 logon events — agreement = corroboration, disagreement = log tampering or non-interactive auth (e.g. service logon type 5, network type 3).
  • SAM group membership ↔ Scheduled Tasks & Services running as a specific user (§03 / §08) — privileged group + persistence task = canonical "what attacker got" picture.
  • SAM NT hash ↔ LSASS memory (§02) — same hash present in both confirms offline-crack feasibility and pass-the-hash potential.
  • IdentityStore Microsoft account ↔ OneDrive sync DB (§11) — ties cloud sync activity to the local user that signed in.
  • Account RID ↔ $Recycle.Bin\<SID>\ ownership (§05) — deleted files are bucketed by the deleting user's full SID; resolve via SAM machine-SID + RID.
  • SECURITY\Cache ↔ SYSTEM bootkey — cached domain creds only decrypt when the SYSTEM hive is mounted alongside; silent garbage otherwise.
  • NTUSER per-user activity (§09) is meaningless until you've answered "whose NTUSER?" — that answer always comes from ProfileList + SAM.
