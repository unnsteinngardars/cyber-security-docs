📐Why NTUSER.dat is the intent layer of the system

Where §04 NTFS records what happened to the bytes and §10 records what code ran, NTUSER.dat records what a human chose — every term typed into a search box, every path entered in the address bar, every file picked in an Open / Save dialog, every macro the user explicitly trusted. It is the closest the registry comes to a transcript of intent.

Every artefact below is scoped to one user — one profile, one SID, one hive file. Nothing in this chapter attributes to a human until the hive itself is bound to a SID via §07 (SAM → ProfileList → profile path → this NTUSER.DAT). Without that binding, every "the user searched for X" sentence is structurally unsound.

The structural rhythm of HKCU is: per-extension subkeys + MRU ordering value + numbered or named children. Master that pattern once and the same shape appears across WordWheelQuery, RecentDocs, RunMRU, ComDlg32 OpenSavePidlMRU, and Office File MRU — only the payload format differs.

📌MRUListEx vs MRUList: MRUListEx is binary little-endian DWORDs (RecentDocs, WordWheelQuery, ComDlg32). MRUList is an ASCII string of single characters (RunMRU, e.g. gdcba). They are not interchangeable — Registry Explorer parses each correctly, but eyeballing the binary value gets the order wrong in half the cases.
🔍WordWheelQuery — Start menu / Explorer search
Per-user Intent MRUListEx order

Where:

Software\Microsoft\Windows\CurrentVersion\Explorer\WordWheelQuery

Structure: MRUListEx (REG_BINARY, little-endian DWORD sequence, most-recent first) + numbered values 0, 1, 2, … each a UTF-16-LE search string.

Answers:

  • Every term the user typed into the Start menu search box or the File Explorer search bar.
  • The order in which queries were issued (via the MRUListEx DWORD list).
  • Strong signal for filenames the user knew but couldn't locate — hidden files, deleted artefacts, attacker tooling the user was hunting or returning to.

Cannot tell you: the search results the user saw, nor whether they clicked any result. Pair with §12 LNK files / RecentDocs to confirm a found-and-opened follow-through.

🧭TypedPaths — Explorer address bar
Per-user Deliberate nav UNC tell

Where:

Software\Microsoft\Windows\CurrentVersion\Explorer\TypedPaths

Structure: values url1, url2, … (UTF-16-LE). Most recent is always url1 — overwriting is positional, not MRU-listed.

Answers:

  • Paths the user typed into the address bar — not paths they clicked into via the tree pane.
  • Local paths and UNC entries (\\server\share) alike.
  • UNC entries here are gold — explicit, deliberate network access typed by a human, not a drive-letter shortcut click. The path tells you the share name and (often) the server.

Cannot tell you: whether the resource was reachable or what was viewed inside it. Cross-reference with Map Network Drive MRU and §08 MountedDevices to confirm the share actually mounted.

📄RecentDocs — files the user opened
Per-user Subkey LastWrite Open ≠ exec

Where:

Software\Microsoft\Windows\CurrentVersion\Explorer\RecentDocs

Structure: per-extension subkeys (.docx, .jpg, .pdf, .exe, …), each with its own MRUListEx and numbered binary values containing the filename plus a PIDL into the shell. The top-level RecentDocs aggregates across extensions.

Answers:

  • Files the user opened via Explorer — note: opening, not executing.
  • Subkey LastWriteTime ≈ time of the most recent file of that extension — a per-extension last-activity proxy.
  • Entries survive deletion of the underlying file; the PIDL retains the original shell location.

Cannot tell you: what the user did with the file once opened. Pair with §12 LNK files in %APPDATA%\Microsoft\Windows\Recent — the same open event materialises in both places with mutually corroborating timestamps.

⌨RunMRU — Win+R dialog history
Per-user Typed → Enter MRUList (ASCII)

Where:

Software\Microsoft\Windows\CurrentVersion\Explorer\RunMRU

Structure: values a, b, c, … each a UTF-16 string. MRUList — ASCII string like gdcba giving the order, most-recent first (left-most letter wins).

Answers:

  • Every string the user typed into the Win+R Run dialog and pressed Enter on.
  • One of the cleanest user-intent artefacts on the entire system — cmd, powershell, \\attacker\share, %temp%\loader.exe: if it's here, a human typed it.

Cannot tell you: whether the command actually executed or what it did. Pair with §10 (UserAssist / Prefetch / BAM / Amcache) to prove the typed command resolved to a process that ran.

🎯UserAssist — GUI-launched program history
Per-user Confirmed exec ROT13 obfuscation

Where:

Software\Microsoft\Windows\CurrentVersion\Explorer\UserAssist\{GUID}\Count

Structure: value names are full executable paths obfuscated with ROT13 (Microsoft's transparent shroud — cmd.exe becomes pzq.rkr). Each value's binary data carries a run counter, total focus time in ms, and a FILETIME of the last launch.

Answers:

  • GUI-initiated launches — programs started from the Start menu, desktop, taskbar, or a double-click. Not services or scheduled tasks (those go in BAM / DAM, see §10).
  • Run count and last-execution timestamp per executable — the closest the registry comes to a per-user process history.
  • Focus time hints at whether the user actually interacted with the window or just opened and closed it.

Cannot tell you: command-line arguments, parent process, or non-GUI launches. CLI-only invocations (e.g. powershell launched by a parent process) skip UserAssist entirely.

🏷MUICache — friendly names of executables run
Per-user Ran at least once

Where:

Software\Classes\Local Settings\Software\Microsoft\Windows\Shell\MuiCache

Structure: value names are full executable paths (often with the .FriendlyAppName / .ApplicationCompany suffix); value data is the friendly name pulled from the PE version resource the first time the binary was launched in this user context.

Answers:

  • Every executable the user has launched at least once — populated lazily by Explorer the first time a binary's icon or label is rendered.
  • Original paths persist even when the binary is moved or deleted — useful for surfacing the launch path of staging directories like %TEMP% or %APPDATA%\Roaming.
  • Friendly name / publisher harvested from version metadata — flags impersonation (e.g. an executable claiming Microsoft Corporation publisher running from C:\Users\Public).

Cannot tell you: when, how many times, or with what arguments — MUICache is a one-shot existence flag, not a counter.

📑Office File MRU — Word / Excel / PowerPoint / Outlook
Per-user Per-entry FILETIME Cloud paths

Where:

Software\Microsoft\Office\<version>\Word\User MRU\LiveId_<hash>\File MRU

Same structure under Excel, PowerPoint, Outlook. <version> is 16.0 for current Office.

Structure: values Item 1, Item 2, … each a FILETIME prefix + full path of the document.

Answers:

  • Documents opened by Office, with a per-entry timestamp Explorer doesn't keep.
  • Paths may be local, UNC, OneDrive HTTPS URL, or SharePoint URL — Office tracks wherever it opened from.
  • Adjacent Place MRU subkey lists folder paths recently picked as save targets.

Cannot tell you: whether the document contained a macro or was actually edited. For trust-prompt outcomes, see TrustRecords directly.

⚠TrustRecords — "Enable Content" clicks
Per-user Trust decision Macros enabled

Where:

Software\Microsoft\Office\<version>\<app>\Security\Trusted Documents\TrustRecords

Structure: one value per file whose macros or external content the user explicitly trusted. Value name is the file URL (file:///, http://, smb://). Value data ends with a flag DWORD.

Answers: the canonical artefact for "user clicked Enable Content on a malicious macro."

  • 7FFFFFFF — user clicked Enable Content (macros enabled).
  • 01000000 — Enable Editing only (Protected View dismissed, macros not enabled).
  • Entry survives deletion of the underlying file — the tombstone outlives the lure.

Cannot tell you: what the macro did — pair with §10 process execution + child-process evidence.

⚠️Flag-byte caveat: the 7FFFFFFF / 01000000 interpretation is the modern Office convention. Older Office builds and some out-of-band patches shift the meaning. Always record the raw hex verbatim in the report and cite the Office version from the parent key — never paraphrase a click as "macros enabled" without the bytes to back it.
💾ComDlg32 — the common Open / Save dialog
Per-user PIDL decoding Intent ≠ outcome

Where:

Software\Microsoft\Windows\CurrentVersion\Explorer\ComDlg32

Structure: three sub-artefacts get written every time any application invokes the Windows common Open or Save dialog. Even a cancelled save leaves the intent here.

Answers:

  • OpenSavePidlMRU — per-extension subkeys (.docx, .exe, …) plus a catch-all *. Each value is a PIDL (shell ID list) describing a file recently picked in any Open or Save dialog. Decoded, you recover the full path and a reference to the shell location at click time — including removable / network volumes that may no longer be attached.
  • LastVisitedPidlMRU — one entry per executable that opened a common dialog. Maps an .exe name to the folder it last opened a dialog in. Tells you "the last folder Word saw" or "the last folder mshta.exe browsed".
  • CIDSizeMRU — the dialog window size each application requested. Low forensic value alone, but proves an app was launched at least once and opened a common dialog.

Cannot tell you: whether the file was actually saved / opened — the dialog write happens at pick time, before Cancel. PIDLs decay slowly: 25–26 newer entries overwrite an old one, so a picked file's tombstone lives long after deletion.

ℹ️PIDL decoding: Registry Explorer's plugin panel resolves PIDLs to human paths automatically. Manual decoding is rarely necessary — but understand that a PIDL is a chain of shell folder IDs, not a string path. That's why it can describe a file on a USB stick that's now unplugged: the chain reconstructs through shell-folder GUIDs, not drive letters.
🌐Map Network Drive MRU + MountPoints2
Per-user Network attribution

Where:

Software\Microsoft\Windows\CurrentVersion\Explorer\Map Network Drive MRU Software\Microsoft\Windows\CurrentVersion\Explorer\MountPoints2

Answers:

  • Map Network Drive MRU — UNC paths the user explicitly mounted as a drive letter via Explorer's Map Network Drive… wizard. Distinct from TypedPaths (which is address-bar typing): this is "I want this share as Z:".
  • MountPoints2 — subkeys for every volume the user's account has seen. Drive-letter subkeys (A:–Z:) reflect mounts; GUID subkeys describe removable media that was attached at least once; UNC subkeys (##server#share) prove a share was accessed under this user's context.
  • Subkey LastWriteTime often pinpoints first-attach / first-access time per device.

Cannot tell you: the device's volume serial — that comes from SYSTEM\MountedDevices (§08). Pair the two to bind the user-context view (MountPoints2) to the machine-context view (MountedDevices).

🔎Other search-history surfaces
Per-user Search intent

Where (depending on Windows build):

Software\Microsoft\Windows\CurrentVersion\Search\RecentApps Software\Microsoft\Windows\CurrentVersion\SearchSettings

Answers:

  • Search\RecentApps (Win10) — recently used apps surfaced to the Start menu / Cortana, each with a launch counter and last-launch FILETIME. Equivalent to a Start-menu-side UserAssist.
  • Cortana / web-search history sometimes leaks here on builds where local search and web search share the WordWheel pipeline — treat WordWheelQuery as a superset and let the plugin decide.

Cannot tell you: what was found. Treat these as supplementary corroboration alongside WordWheelQuery, not standalone evidence.

⚙Parsing — tooling notes

Registry Explorer + plugins cover every artefact above out of the box — WordWheelQuery, TypedPaths, RecentDocs, RunMRU, UserAssist, MUICache, OfficeMRU, TrustRecords, ComDlg32. PIDLs are resolved to human-readable paths in the Plugins panel; UserAssist values are de-ROT13'd inline.

RECmd (Eric Zimmerman, batch-mode parser) fires the same plugins from the CLI in one pass using NTUSER_Batch.reb and dumps a CSV per artefact — the right call for fleet-wide triage.

RegRipper — perl-based, fast for targeted runs. Relevant plugins: userassist, recentdocs, runmru, typedpaths, wordwheelquery, comdlg32, trustrecords, officedocs, muicache, mp2.

Always collect the transaction logs: NTUSER.DAT.LOG1 and .LOG2 alongside the hive. Registry Explorer replays them and surfaces dirty data the live hive doesn't yet show — recent activity that hadn't been flushed at acquisition time.

Cross-references for corroboration

NTUSER findings only become evidence once tied to a specific human and a specific action that actually executed. Every artefact above is intent — pair it with at least one independent witness:

  • §07 SAM → SID → ProfileList → NTUSER.DAT — the attribution chain. No SID match, no human. Every paragraph above implicitly assumes this binding has been done.
  • RecentDocs ↔ §12 LNK files in %APPDATA%\Microsoft\Windows\Recent — the same open event surfaces in both places. LNK carries MAC times, target volume serial, and a TrackerDataBlock GUID; pair with §04 NTFS $Volume VSN to bind the open to a specific physical device.
  • RecentDocs / Office File MRU ↔ §13 Jump Lists — Jump Lists store the same recent-document data per application and persist across logons; the JL stream often contains entries that have already aged out of RecentDocs.
  • TypedPaths UNC ↔ MountPoints2 ↔ §08 MountedDevices ↔ Map Network Drive MRU — confirms the typed share was actually mounted, when, and under which drive letter. MountedDevices ties drive letter → volume serial; cross-reference §04 NTFS $Volume VSN for the physical device.
  • RunMRU + UserAssist + TrustRecords + Office File MRU — pair with §10 Application Execution (Prefetch, Amcache, BAM/DAM). Build a single timeline: user typed → app launched → file opened → macro enabled → child process spawned.
  • ComDlg32 OpenSavePidlMRU ↔ §05 $Recycle.Bin $I records — the picked file may already be deleted; the PIDL is its tombstone, and the recycle-bin record gives the deletion timestamp and original path.
  • TrustRecords URL ↔ RecentDocs ↔ LNK in Recent ↔ browser download history — for emailed lures, the closer those timestamps cluster, the cleaner the phishing-to-execution story.
  • RecentDocs / Recent\*.lnk MACB ↔ §04 NTFS $SI vs $FN — when LNK target timestamps disagree with NTFS witnesses, suspect timestomping on the target file.
  • UserAssist / MUICache executable paths ↔ §10 Amcache / Prefetch hash — the same binary surfaces multiply; mismatched paths or missing Prefetch indicates a renamed / relocated executable.
