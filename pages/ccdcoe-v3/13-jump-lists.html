📌What Jump Lists are — and why forensics loves them
Proof of opening from app Per-user, per-application Embedded LNKs Per-entry timestamp

The feature: introduced in Windows 7, Jump Lists are the lists you see when you right-click a taskbar icon — "Recent", "Pinned", and per-app "Tasks". To build them, Windows maintains two file types per application: automatic destinations (the OS-managed MRU) and custom destinations (developer-controlled lists for app-specific tasks).

Why they matter forensically: Jump Lists are the per-application opening ledger. Where RecentDocs gives a global, extension-bucketed MRU, Jump Lists tell you "Word opened these documents in this order" or "Edge browsed to these URLs." They also retain entries the user has actively pinned, preserving intent even after the file was deleted or moved.

Storage: per-user, hidden, under %AppData%\Microsoft\Windows\Recent\ in two subfolders. Files are named by 16-hex-character AppID + extension. Each file is an OLE Compound File (the same container format as legacy .doc / .msi), so it has streams and a directory just like a mini filesystem.

ℹJump Lists exist only for applications that opted in via an AppUserModelID. Console programs (cmd / powershell), most malware, and many in-house tools never produce them. Absence here is not absence of execution.
📁AutomaticDestinations — the OS-managed MRU
DestList ordered MRU Embedded LNK per item
%AppData%\Microsoft\Windows\Recent\AutomaticDestinations\<AppID>.automaticDestinations-ms

This is the file Windows itself writes when the user opens a document with an opted-in application. It is an OLE Compound File whose streams are:

  • DestList stream — a header plus an array of entries, each containing: entry number, ID counter (the stream where the LNK lives), MRU index, last-access timestamp (FILETIME, UTC), pinned-state byte, NetBIOS name of the host that opened it (yes — like LNK's TrackerDataBlock), MAC address, target path, and an arrival timestamp. The DestList header tracks total entries created and the highest entry ID — by joining these you can detect Jump List deletions by gap in numbering.
  • Per-entry streams named by hex ID — each is a complete .lnk file, parsed exactly like §12 standalone LNKs. Volume serial, target MAC times, drive type, TrackerDataBlock — all present.

Capacity: default ~20 recent items per app (user can change via taskbar properties). When the cap is reached the lowest-MRU entry is evicted — JLECmd flags this clearly.

🛠CustomDestinations — the developer-controlled list
App-driven tasks Pinned items live here too
%AppData%\Microsoft\Windows\Recent\CustomDestinations\<AppID>.customDestinations-ms

Not an OLE Compound File — instead, a custom binary container: a header followed by a sequence of LNK structures back-to-back, terminated by an end marker. The app populates this file at runtime; e.g. Word's recent-documents-by-category list, Chrome's "Recent Tabs", PuTTY's session list (yes — PuTTY uses Custom destinations).

Forensic value: for some apps Custom destinations preserve session history the developer chose to expose. PuTTY sessions, mIRC channels, RDP target hosts (mstsc.exe), 7-Zip archive paths — each can be reconstructed from CustomDestinations even after the recent-files list was cleared.

🔢AppID — the 16-hex-character join key
Application identity Survives renames if AppUserModelID set

Every Jump List file is named by an AppID — a 16-hex-character string. The AppID is either:

  • A CRC64 of the application's full executable path (default for apps that don't declare otherwise) — changes if the EXE is moved or renamed.
  • A declared AppUserModelID string (set by Modern / UWP apps and many modern desktop apps) — stable across reinstalls and updates.

Practical lookup: tools like JLECmd ship with a known-AppID database covering common apps (Office, Edge, Chrome, Notepad, VS Code, mstsc, putty, etc.). Online community lists exist (EricZimmerman/JumpList/JumpListExplorer reference). For unknown AppIDs, you can compute the CRC64 of suspect paths and look for matches in SOFTWARE\Microsoft\Windows\CurrentVersion\App Paths, Amcache, or Prefetch.

The AppID is also recorded inside many other artefacts (BAM, SRUM, Activities cache, Task scheduler XML), giving you a one-key join across apparently unrelated sections.

⚠When an attacker side-loads a tool by copying it to C:\ProgramData\<random>\<name>.exe, the resulting AppID changes versus the legitimate install location. A previously-unknown AppID in AutomaticDestinations is itself an IOC — chase the CRC64.
📍Pinned vs Recent — intent vs activity
User intent

Inside the DestList stream every entry carries a pinned byte. Pinned entries persist beyond the normal eviction cap and are user-chosen: an attacker who pinned a network share after exfil leaves a long-lived breadcrumb. Pinned items are also what feed the right-click "Pinned" group on a taskbar icon.

Recent (unpinned) items are evicted as new ones arrive. The MRU index is independent of the absolute access timestamp — sort by both columns and reconcile to see the user's recency model.

📦Per-app gold-mine examples
Worth looking at first
  • mstsc.exe — RDP target hosts and most recently used host names. Critical for lateral movement timelines.
  • powershell_ise.exe — recent .ps1 paths even when ConsoleHost history is wiped.
  • cmd.exe Jump List exists only on Win10+ when run as default — limited but sometimes captures recent directories.
  • WinRAR / 7-Zip — archive paths handled, including the password-protected ones.
  • Notepad / Word / Excel — recent doc list, often surviving after RecentDocs is cleared.
  • Edge / Chrome — recent profile / tab info via Custom destinations.
🧪Parsing & reading DestList — practical notes
JLECmd workflow

Workflow:

  • Acquire both directories together — AutomaticDestinations and CustomDestinations — they correlate.
  • Run JLECmd.exe -d <dir> --csv <out> --html <out>. Two CSVs result: one per-Jump-List-file summary, one per-entry detail. Time-zone is UTC; JLECmd respects -tz for normalisation.
  • For each interesting AppID, also extract the embedded per-entry LNKs and run LECmd over them — gives you full TrackerDataBlock detail (MAC, MachineID) that's flattened in JLECmd's summary view.
  • Diff DestList header's last entry ID vs the number of entries actually present — gaps mean an attacker selectively scrubbed entries by stream removal (a common anti-forensics step).
  • The whole file's NTFS $STANDARD_INFORMATION and $FILE_NAME timestamps from §04 record creation and last write — first DestList entry ≈ first opened with that app, last write ≈ most recent open.
!OLE Compound Files allocate space in sectors, and deleted streams may remain in slack. When investigating selective tampering, parse the raw OLE structure with libfwsi or olefile and look for "unreferenced" streams holding former Jump List entries.
🪟Win10+ Activities & Timeline (parallel artefact)
Activities ledger Optional, but very rich
%LocalAppData%\ConnectedDevicesPlatform\<guid>\ActivitiesCache.db

Adjacent to Jump Lists in spirit: the Windows Activities feature stores per-app, per-document timeline events in a SQLite database. Tables Activity, ActivityOperation, and ActivityPackageId record application AppUserModelIDs (the same AppID concept), payload JSON describing the document opened, the device class, and start / end times.

Enabled by default on Win10 1803+ for local activity, off by default for cross-device sync. Even on hardened endpoints local rows are usually present. Parse with WxTCmd or a SQLite browser.

🧨Anti-forensics & tampering signs
Detection
  • "Clear recent items" wipes RecentDocs but leaves Jump Lists mostly intact — a user who knows this trick is informed.
  • Deleting AutomaticDestinations\ wholesale leaves a hole in §04 $UsnJrnl (mass FILE_DELETE of *.automaticDestinations-ms in one minute is a strong tell).
  • An automaticDestinations file whose modification time is much newer than the latest DestList entry suggests selective stream removal.
  • Empty Jump List directory on a long-used profile is a deliberate wipe — note especially when §10 Amcache / Prefetch show many opted-in app launches.
Cross-references for corroboration

Jump Lists hand you per-app context that other artefacts only hint at — the best work joins them aggressively:

  • §04 NTFS — $MFT / $UsnJrnl — each Jump List file's MFT timestamps span its lifetime (created on first opt-in launch; modified on every entry). $UsnJrnl mass deletions of *.automaticDestinations-ms document anti-forensics.
  • §07 SAM — Recent\AutomaticDestinations is per-user, so the owning profile's SID joins back to a real user via SAM.
  • §08 SYSTEM/SOFTWARE — App Paths & UninstallKeys — when you have an unknown AppID, the CRC64 of an EXE path can be matched against installed-app paths and uninstall entries to identify the application.
  • §09 NTUSER — RecentDocs / UserAssist — UserAssist confirms an opted-in app was launched the count of times shown; Jump List DestList confirms which documents that app touched on each launch.
  • §10 Application Execution — Prefetch gives existence-of-execution for the app's EXE; Jump List gives intent-of-execution (which document). Together they answer "what did the user do, and which binary did it."
  • §11 Cloud Storage — Office Jump Lists frequently contain SharePoint and OneDrive URLs as http(s):// entries — these are sometimes the only host-side evidence a user opened a cloud-only doc.
  • §12 Shell Items & LNK — every per-entry stream inside an automaticDestinations file is a full LNK; every concept on the LNK page (TrackerDataBlock, VSN, target MAC) applies inside. Always extract and parse separately.
  • §14 USB — Jump List LNKs to removable drives carry the VSN of the USB, joining to USBSTOR / MountedDevices / Event 1006.
  • §15 Timeline & Correlation — JLECmd CSVs (one row per DestList entry, with timestamp) ingest cleanly into Timeline Explorer; combined with LECmd output for the embedded LNKs you get document-level granularity on the master timeline.
