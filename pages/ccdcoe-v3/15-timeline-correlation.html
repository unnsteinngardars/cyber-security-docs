🧭Why timelines exist — and how they go wrong
Methodology Common failure modes

A forensic timeline is the chronological projection of every artefact in this cheat sheet onto a single axis: time. Done well, it transforms fourteen separate evidence sources into a single sequence of events the analyst, lawyer, and finder of fact can read together. Done badly, it becomes a haystack so large that real findings drown.

The recurring failure modes — and the antidotes:

  • Time-zone drift. Half the artefacts on this list store FILETIME (UTC); a few (setupapi.dev.log, IIS logs, browser cache) store local time; some (Windows Event Log XML) carry both. Always parse to UTC at ingest; convert to display TZ only at the final render.
  • Clock skew on the host. A timeline assumes the system clock was accurate. Cross-check the suspect host against domain-controller-side audit logs and against any timestamp originating on the network (browser headers, sync logs). Capture the TimeZoneInformation registry key as part of every collection.
  • Granularity mismatch. NTFS FILETIME is 100-ns resolution; SQLite "modified time" is integer seconds. Event ordering at sub-second resolution requires care; round consistently or be explicit about ties.
  • Source confusion. Two artefacts can record "the same event" with different semantics — Prefetch's last-run is the OS's view of process create; Sysmon 1 is the kernel hook's view. The two will not agree to the millisecond; treat them as separate corroborating sources.
ℹPractical rule: never quote a single source for any claim of consequence. The reasonable-investigator standard for forensic conclusions is corroboration from at least two artefact classes per finding.
📐Mini-timeline vs super-timeline
Pick the right scope

Mini-timeline: a narrow scope built around a specific question — usually a single artefact class or a single hour. Examples: "show me the MFT events in the hour around 14:32 UTC" or "show me every LNK created today." Tools: Timeline Explorer over a single CSV; grep against a sorted MFT export.

Super-timeline (a.k.a. l2t / Plaso timeline): the aggregation of every parseable artefact on the image into one event stream. Run with log2timeline.py / psort.py and explored in Timesketch, or built incrementally in Timeline Explorer.

Start with mini-timelines around known events; promote to super-timeline only when a hypothesis demands it. Inverse order wastes hours.

🔑Universal join keys
Cross-artefact glue

These identifiers reliably bridge artefact classes — treat them as first-class columns in any timeline:

  • SID — from §07 SAM to BAM, SRUM, Security 4624/4625/4688, MountPoints2.
  • Volume Serial Number — §04 $Volume to LNK, MountedDevices, Event 1006, Prefetch volume blocks.
  • ContainerID / USB Serial — §14 USB to Sysmon device events, setupapi.dev.log.
  • AppID (CRC64 or AppUserModelID) — §13 Jump Lists to Activities cache, BAM, Amcache.
  • SHA-1 / SHA-256 / MD5 of binary — §10 Amcache to Sysmon 1, EDR telemetry, M365 file-upload audit, threat-intel feeds.
  • Resource ID / DocID / File ID — §11 cloud artefacts to tenant-side audit logs.
  • NTFS FILE reference (MFT entry # + sequence) — §04 across $MFT, $UsnJrnl, $LogFile; also embedded in LNK shell items.
  • EventRecordID + Channel — within EVTX, your unique handle for any given event row.
⏰$STANDARD_INFORMATION vs $FILE_NAME
Anti-forensics detection Dual timestamp model

Every NTFS file carries two sets of four MAC timestamps: $SI (the user-facing copy, freely writable by SetFileTime / timestomp tools) and $FN (the copy maintained by the OS and updated on rename / link only). When the two diverge, you almost always have anti-forensics.

Standard practice: present timelines using $SI by default but add a $SI ≠ $FN flag column that lights up on every divergent row. MFTECmd emits both natively.

⚠Modern timestomp tools (e.g. SetMACE, timestomp.exe v2) update both $SI and $FN via direct MFT writes. $LogFile transactions (§04) survive much longer than the manipulated timestamps and are the best detection. Always carry the row's $LogFile LSN in any high-trust timeline.
🛠A working super-timeline workflow (Plaso + Timesketch)
From image to query
log2timeline.py --storage-file case.plaso disk.E01 psort.py -o l2tcsv -w case.csv case.plaso "date > '2026-06-01 00:00:00'"
  1. Acquire & verify. Image with FTK Imager / Magnet ACQUIRE; hash with SHA-256 at collection and again at first read.
  2. Triage with KAPE to pull just the artefacts you need (saves hours over full Plaso of a TB+ disk). KAPE compounds (!SANS_Triage, !EZParser) drive most of this section's parsers in one shot.
  3. Parse with EZ Tools (MFTECmd, PECmd, AmcacheParser, RECmd, LECmd, JLECmd, SBECmd, EvtxECmd, SrumECmd). Each emits CSV with consistent UTC timestamp columns.
  4. Normalise. Use a single CSV schema (timestamp, source, source_id, user_sid, host, message, …). Many teams use Timesketch's Plaso JSON intermediate.
  5. Ingest into Timesketch (or Timeline Explorer for smaller cases). Tag with case ID and analyst.
  6. Query in layers. Start with broad time slices around known events; tighten the window; add filters by SID, by VSN, by AppID. Save useful queries as views; document each hypothesis tested.
  7. Annotate. Every row that supports a finding gets a comment with the source artefact path and the reasoning. The goal is reproducibility.
  8. Export. Render the final narrative as a written report; cite annotated row IDs as evidence.
🔍Forward & backward pivots
Investigation pattern

Forward pivot: start at a known event, look for downstream consequences. Example: Prefetch first-run of an unknown binary → forward to $UsnJrnl writes inside its working directory → forward to Sysmon 11 (file creates) → forward to Sysmon 3 (network connections). Each step expands the blast radius.

Backward pivot: start at an observed effect, work back to a cause. Example: ODL upload event → backward to Prefetch run that opened the source file → backward to USB plug-in event that delivered it → backward to setupapi.dev.log entry that first installed the stick. Each step narrows the cause.

Most real cases require both directions; switch when you've answered "and then what?" or "and how did that happen?" and start running out of detail.

🧨Anti-forensics — what to flag
Indicators of tampering
  • $SI ≠ $FN divergence (timestomp).
  • Event log gaps — burst of 1102 (Security log cleared) or large unexplained gaps spanning hours.
  • Registry key whose last-write is much later than its content suggests.
  • Prefetch present for binaries whose $MFT entry is missing or shows a much later create time.
  • USBSTOR / setupapi.dev.log mismatch — an SN in one and not the other.
  • OneDrive SafeDelete rows with no matching SyncEngineDatabase row — selective DB editing.
  • Sysmon 4 (service state change), Sysmon 16 (config change), or sudden silence in Sysmon stream — evasion of telemetry.
  • Jump List DestList header ID greater than entries actually present — stream removal.

📊Visualisation patterns that work
Make findings readable
  • Swim-lane diagrams with one lane per artefact source — easy for non-technical audiences.
  • Heatmaps of event density per hour — surface anomalous bursts (e.g. midnight activity for a 9–5 user).
  • Sequence diagrams for multi-host lateral movement: from logon → SMB → service create → execution.
  • Activity graphs (Timesketch's analyzers, or custom networkx) — process-tree-style with file / network nodes, showing the blast radius.
  • Static written narratives with cited rows — the only deliverable a court ultimately wants.
📦Reporting & evidence handling
Chain of custody
  • Preserve original images and parsed CSVs in write-once storage with hash manifest.
  • Record analyst, date, tool versions, and command lines for every parsing step (KAPE's CLI is friendly to this; many teams script it).
  • State assumptions explicitly: time zone applied for display, what was filtered out, what was outside scope.
  • Cite each finding back to a row ID in the source CSV, not a paraphrase.
  • Keep a "things we did not find" appendix — absences of evidence ruled the hypothesis out, and that has value.
Cross-references — the whole map

This section is the convergence point — every prior section is its input. The most-used joins:

  • §01 Evidence Collection — the integrity of every timestamp downstream depends on how the image was acquired. Document acquisition order, write-blockers, hash verifications.
  • §02 Memory Forensics — volatile process trees give the "alive at acquisition" anchor that disk forensics cannot. Sysmon 1 in EVTX joins memory's pslist for the same minute.
  • §03 KAPE & AIM — the collection pipeline that feeds this section. KAPE's !EZParser / !SANS_Triage output is what Timeline Explorer and Timesketch ingest.
  • §04 NTFS — $MFT, $UsnJrnl, $LogFile are typically the largest event source in any super-timeline. $SI vs $FN divergence is the master tamper detector.
  • §05 File Carving — for files whose MFT entry is gone, carving recovers content but loses timestamps; timeline rows for carved files should be tagged as "no native time".
  • §06–§09 Registry Hives — SAM / SYSTEM / SOFTWARE / NTUSER are the substrate of identity, policy, autostarts, and user activity. The last-write times of registry keys are first-class timeline rows.
  • §10 Application Execution — Prefetch / Amcache / BAM / UserAssist / SRUM provide six independent execution timestamps. Conflicts among them are evidence.
  • §11 Cloud Storage — ODL is the densest single timestamp source on most modern hosts; merge with care.
  • §12 Shell Items & LNK — target MAC times preserve "the file existed at this point" even when the file is gone.
  • §13 Jump Lists — per-app document timelines; DestList ordering is its own timeline.
  • §14 USB — anchor events. A plug-in is a global moment; bracket every nearby event for relevance.

The pact at the end: a timeline is correct when an independent investigator running the same tools against the same image arrives at the same sequence — and disagrees with you only on interpretation, not on data. Until you can demonstrate that, it is not a timeline. It is a draft.
