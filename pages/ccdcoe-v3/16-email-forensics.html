📨Email headers — the immutable trail
Per-hop timestamps Auth attribution Hardest to fake
RFC 5322 / RFC 5321 headers — present in every message regardless of client

Read every header from the bottom up. Each MTA prepends a Received: line as it accepts the message — the oldest hop is closest to the body, the most recent is at the top. Read in that direction and you reconstruct the path.

Forensically interesting headers:

  • Received: — each line records the receiving server, sending server, protocol (SMTP / ESMTP / ESMTPSA), TLS state, queue ID, and UTC timestamp. The first Received line (bottom of the stack) reveals the actual originating IP, often very different from the From address.
  • From: — sender as the user wrote it. Trivially forged. Check Reply-To and Return-Path for divergence.
  • Reply-To: — where replies go. Mismatch with From is a classic phishing IOC.
  • Message-ID: — a globally unique ID issued by the originating MTA. Format <random@originating-host> exposes the sender's mail platform.
  • Date: — sender's claimed send time. Compare to the bottom-most Received timestamp; large skew suggests clock manipulation or replay.
  • Authentication-Results: — receiver's verdict on SPF, DKIM, DMARC, ARC. The single most important header for attribution. Multi-hop systems may have several of these, each from a different verifier.
  • X-* headers — vendor-specific. X-Originating-IP, X-Mailer, X-MS-Exchange-CrossTenant-AuthSource, X-Sender-IP, anti-spam scoring (X-Microsoft-Antispam, X-Forefront-Antispam-Report) all carry intelligence.
  • References: / In-Reply-To: — Message-IDs of prior messages in the thread. Reconstructs conversation graphs even when individual mailboxes are missing.
ℹAuthentication-Results is not optional intelligence — it's evidence. A message where SPF=pass DKIM=pass DMARC=pass from a legitimate domain is hard to forge without compromising sender infrastructure; one where all three fail is hard to defend as legitimate.
🛡SPF — sender path validation
DNS-published policy

What it does: the sending domain publishes a DNS TXT record (v=spf1 …) listing the IPs allowed to send mail for that domain. Receivers compare the SMTP envelope MAIL FROM address against the IP that connected.

Forensic value:

  • pass — sender IP is in policy.
  • fail / softfail — out of policy; usually phishing or misconfiguration.
  • neutral / none — domain has no policy or did not commit.
  • SPF does not authenticate the visible From: header — only the envelope sender. A spoofed From: with a legitimate envelope sender can pass SPF. Always join with DKIM.
✍DKIM — cryptographic signature
Signed-by-domain

The sending mail system computes an RSA / Ed25519 signature over selected headers + body and publishes the verifier key as a DNS TXT record under <selector>._domainkey.<domain>. Receivers fetch the key and verify.

What survives a DKIM pass:

  • Sender domain controlled the signing infrastructure at send time.
  • Body and signed headers are byte-identical to what the sender produced.
  • Selector + algorithm preserve a fingerprint of the sending platform.

An old DKIM-passing message is self-authenticating evidence years later — provided the historical key was rotated and recorded.

⚠Short DKIM keys (1024-bit RSA) are forgeable by well-resourced adversaries — and old leaked keys cannot be re-rotated for past mail. Document key length and selector when basing findings on DKIM.
🎯DMARC — alignment + policy
Policy-driven verdict Visible From alignment

DMARC layers on top of SPF and DKIM: published in DNS as _dmarc.<domain>, it specifies the From:-domain owner's policy (none / quarantine / reject) and reporting addresses. The key test DMARC adds is alignment — the From-header domain must match SPF's MAIL FROM domain (relaxed/strict) or the DKIM d= tag.

This is the gap that purely SPF or purely DKIM cannot close: an attacker can SPF-pass with their own domain in the envelope; alignment forces it to be the same domain the user sees.

What to do with the DMARC verdict:

  • pass — the visible From: was signed-or-pathed by the domain owner. Treat as authentic absent further indicators.
  • fail with a published reject policy that the receiver respected — message would not normally reach an inbox. Its presence in a mailbox means policy bypass (allow-listing, M365 quarantine release, mailbox-rule retrieval, etc.) and is itself an event of interest.
  • fail with none policy — sender published intent only.
✓DMARC RUA aggregate reports (sent to the address in rua=) give defenders historical visibility into who has been sending mail claiming their domain. They are an underused intelligence source for tracing impersonation campaigns.
📦PST — Outlook's offline mailbox
User-owned archive Mailbox export
%UserProfile%\Documents\Outlook Files\*.pst (older defaults) %LocalAppData%\Microsoft\Outlook\*.pst

What it is: a Personal Storage Table — a self-contained MAPI database used by Outlook for archives, mailbox-exports, and (historically) primary IMAP/POP storage. ANSI PSTs (old, 2-GB limit, deprecated) and Unicode PSTs (current, up to 50 GB) both exist.

Why it matters forensically:

  • Self-contained — readable on any machine without server access.
  • Common exfil format: enterprise users export entire mailboxes to PST.
  • Carries the full message store: folders, items, attachments, sent items, deleted items (often recoverable from internal slack).
  • Hidden / restricted folders (Recoverable Items) inside the PST sometimes preserve messages a normal Outlook user thought were gone.

Open in Outlook by attaching the file, or read offline with Xst Reader, libpff / pffexport, readpst, or commercial Aid4Mail. Extract attachments and headers in one pass.

💾OST & NST — synced caches
Server-side backup Survives mailbox purge
%LocalAppData%\Microsoft\Outlook\*.ost %LocalAppData%\Microsoft\Outlook\*.nst (Teams / "New Outlook" cache)

OST is Outlook's offline replica of the server-side mailbox in Cached Exchange Mode. Bit-compatible with PST internally, but the file is the user's own copy of what was on the Exchange / M365 server. This makes OST invaluable when the server-side mailbox has been wiped, retention-purged, or admin-deleted.

NST is a parallel cache used by Teams chat and the "New Outlook" preview — same OLE-derived binary structure, different folder schema. Often holds Teams messages that have been deleted server-side.

Both can be opened with Xst Reader / libpff. Convert OST to PST before importing into Outlook (Outlook itself refuses to attach a foreign OST).

!OST encryption uses a per-mailbox key derived from the user's profile. Acquisition of a live profile preserves key material; cold-boot acquisition of a logged-out machine may lose it. Image while the user is logged in if at all possible.
🔬Extended MAPI headers — the inside view
PR_TRANSPORT_MESSAGE_HEADERS Internal-only metadata

Once a message lands in Exchange / Outlook, it accumulates extended MAPI properties beyond the RFC-822 headers. These are not visible in normal mail clients but are extracted by PST/OST parsers and by Microsoft Exchange admin tools.

Particularly useful properties:

  • PR_TRANSPORT_MESSAGE_HEADERS (0x007D) — the original Internet headers as they arrived, preserved even after Outlook reformats the rest.
  • PR_SENT_REPRESENTING_* — the actual sending principal vs displayed sender (delegation / send-on-behalf detection).
  • PR_LAST_VERB_EXECUTED — last user action (reply / forward / mark-read). Distinguishes received-but-unread from received-and-read.
  • PR_CLIENT_SUBMIT_TIME vs PR_DELIVERY_TIME — sender's clock vs server's clock; divergence is a Date-header sanity check.
  • PR_CREATION_TIME / PR_LAST_MODIFICATION_TIME — when the item arrived in this store and last changed locally.
  • PR_INTERNET_MESSAGE_ID — joins to Message-ID for cross-store threading.
  • PR_RULE_ID / hidden rule-store items — inbox rules an attacker added to forward / hide mail. Live in a hidden folder named FAI (Folder Associated Items); only specialised parsers expose them.
ℹInbox rules used for stealth. Compromised accounts commonly get a rule like "forward everything from CFO to attacker@…" placed via OWA or PowerShell. The rule lives in hidden MAPI items, not in the user's visible Rules UI. Always enumerate hidden items.
🌍Server & cloud logs
Tenant-side truth

Host-side artefacts are necessary but rarely sufficient in modern email cases. Pair with:

  • M365 Unified Audit Log — Send, SendAs, SendOnBehalf, HardDelete, SoftDelete, UpdateInboxRules, MoveToDeletedItems. Default 90/180-day retention; place hold immediately.
  • Exchange Message Trace — per-message delivery path including spam / quarantine actions. Retained ~10 days; export early.
  • Microsoft Graph mailAttachment / message endpoints — for ad-hoc retrieval with full fidelity, including hidden FAI items.
  • Exchange on-prem — message tracking logs, transport-rule logs, IIS logs for OWA / EAS.
  • Google Workspace Email Log Search — equivalent message-trace for Gmail; Vault for retention.
🌐Webmail traces on the host
Browser cache & storage

OWA, Gmail, Yahoo, Proton all leave host-side residue even without an installed client:

  • IndexedDB / LocalStorage (per-profile under %LocalAppData%\Microsoft\Edge\User Data\<profile>\IndexedDB\ etc.) — Gmail/OWA cache message previews, drafts, contacts.
  • Browser cache — HTML message bodies and inline images.
  • Service-Worker caches for PWA-style webmail (Outlook PWA, Gmail offline).
  • History / autofill — typed addresses, attached-file paths.

These can be the only host-side proof a message was viewed when the user does not run a fat client at all.

📱Mobile storage & ActiveSync
EAS partnership

Mobile devices configured with Outlook for iOS/Android or native mail use Exchange ActiveSync. The mobile device's data is out of scope for Windows-host forensics, but the partnership records are not:

  • M365 / Exchange Admin records every EAS device by deviceID — useful when a phone has been recycled.
  • OST hidden FAI items sometimes contain mobile-device-syncstate metadata.
  • Outlook for iOS/Android adds the M365 tenant under multiple device IDs over time — historical device list reveals lost / stolen / replaced devices.
📎Attachment recovery
Inside the store

Attachments live inside the PST/OST as MAPI attachment objects on a message. They are extracted not by file-system carving but by walking the store:

  • libpff pffexport dumps every attachment to disk with its original filename.
  • Xst Reader exposes attachment lists per message; right-click → save.
  • Outlook can save attachments via VBA scripting in bulk.
  • Embedded (OLE) messages — replies that include the prior message as attached message — recursively contain headers and attachments of their own. Walk them.
  • Deleted-item attachments sometimes survive in PST slack space — libpff's --mode=all recovers what is reachable.
🎯Common case patterns
Workflow
  • Phishing landing. Headers → identify originating IP → DMARC / SPF / DKIM verdict → join cloud audit for delivery time → check inbox-rule changes immediately after (auto-forward installed?) → host-side trace user clicked link (Edge history, browser cache).
  • Insider mailbox export. PST creation timestamp on disk → Outlook "Import/Export" action in §10 UserAssist / Office MRU → file size of PST vs mailbox size → cloud audit New-MailboxExportRequest if PowerShell-driven.
  • Compromised account hidden rule. Enumerate hidden FAI in OST → cloud audit UpdateInboxRules → join with Azure AD sign-ins around the change time.
  • Spoofed sender impersonating exec. Visible From vs Authentication-Results DMARC verdict → publish-and-history check on the impersonated domain's DMARC policy → check whether the receiver's tenant has impersonation protection.
  • Server-side delete cover-up. Local OST has the message; M365 Unified Audit Log shows HardDelete by the same user shortly after view; messages survive in Recoverable Items of the OST.

  • §12 LNK — every attachment the user opened (which materialises to %Temp%\Outlook Logging\ or to a chosen Save path) gets a Recent\ LNK.
