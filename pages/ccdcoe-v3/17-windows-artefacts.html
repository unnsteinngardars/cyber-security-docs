🔎Windows Search — the system-wide ESE database
Indexes everything reachable Outlook + Browser + Files in one DB Survives original file deletion
C:\ProgramData\Microsoft\Search\Data\Applications\Windows\Windows.edb …\Windows-gather.db (gather queue) …\Windows-usn.db (USN journal cursor used to drive incremental indexing) …\*.log / *.jrs / WAL files (ESE recovery + write-ahead)

What it is: Windows.edb is an ESE (Extensible Storage Engine, the JET Blue database engine) database holding the full-text Search index. It records, per indexed item: file name, file path, size, attributes, MAC timestamps, the GatherTime at which Search picked it up, and, for many file types, a chunk of partial content — extracted text, properties, EXIF, email headers and bodies, browser history snippets.

What it indexes by default:

  • The user's profile folders (Documents, Pictures, Music, Videos, Desktop).
  • Outlook mailbox content (via the Office Search Provider) — bodies, subjects, sender, attachment names.
  • Edge / Internet Explorer browser history and favourites (via Internet Explorer history provider).
  • OneDrive content where Files-On-Demand has hydrated it.
  • Any additional paths added by group policy / user customisation.

Why it matters forensically: Search's copy of a file's content persists in the EDB after the original file is deleted. When NTFS metadata is wiped and Recycle Bin is emptied, partial content often remains here for weeks. WinSearchDBAnalyzer (Yogesh Khatri) extracts the rows; output is a CSV-of-everything Search ever saw, ready for Timeline Explorer.

ℹAcquisition note: the database is locked while the Search service runs. Either acquire from a hibernated / shut-down host, use VSS, or use esentutl.exe /y /vss. On a logged-in target, StopService SearchIndexer first, copy, restart — and document the service interruption.
🩹ESE recovery — the dirty-database problem
Repair before parse
esentutl.exe /mh Windows.edb # header check esentutl.exe /r edb /l . /s . /i # replay log files (clean recovery) esentutl.exe /p Windows.edb # forced repair (lossy, last resort)

If acquisition was less than perfect, the EDB will be dirty. The header (/mh) reports it as such. Two paths to a parseable copy:

  • Soft recovery — replay the captured .log WAL files with esentutl /r. Lossless. Requires the matching log generation set.
  • Repair — esentutl /p rebuilds page-by-page; rows in corrupted pages are dropped. Document the loss and re-run twice for stability.

Modern WinSearchDBAnalyzer handles many dirty EDBs directly; older parsers need a clean DB.

🗺Notable Windows.edb columns
Field map

A few of the property columns most often needed:

  • System_ItemUrl — full URL/path of the indexed item.
  • System_DateCreated, System_DateModified, System_DateAccessed — file MAC at index time.
  • System_Size, System_ItemType, System_FileAttributes.
  • System_Search_GatherTime — when Search read this row. Acts as an independent timestamp.
  • System_Search_AutoSummary / System_Search_ContentSnippet — partial extracted text.
  • System_Message_* — Outlook-only: sender, recipient list, subject.
  • System_Image_* / System_Photo_* — EXIF (DateTimeOriginal, GPS, Make/Model).
  • System_Document_* — Office metadata (author, last-saved-by, revision).
🖼Thumbcache & Iconcache — Explorer's preview vault
Image content survives Per-user, per-resolution CacheID is the join key
%LocalAppData%\Microsoft\Windows\Explorer\thumbcache_32.db (and 96/256/1024) %LocalAppData%\Microsoft\Windows\Explorer\thumbcache_idx.db (lookup index) %LocalAppData%\Microsoft\Windows\Explorer\iconcache_*.db (icons + small thumbs)

What: Explorer caches a thumbnail of every image, video, and many document types it has displayed in the file browser at sizes 32, 96, 256, 1024, and (Win10+) 2560 px. Each thumbcache file is a Microsoft custom binary format (an Image Mounted Memory Database by file layout). Per-thumbnail entries contain: cacheID, header signature, size, padding, and the raw image data (PNG, JPEG, or BMP).

Why forensics loves it:

  • Thumbnails of photos, video frames, and document previews survive after the original file is deleted — including after Recycle Bin emptied.
  • Per-user (under %LocalAppData%) — proves this user opened a folder containing this content.
  • Independent of file extension — renaming or trying to hide a JPG as .txt still produced a thumbnail.

How to map thumbnail back to filename: Thumbcache stores cacheIDs, not filenames. The mapping back to filenames takes two routes:

  • Thumbcache_idx.db + Search index (Windows.edb) — preview thumbnail's cacheID is recorded against the indexed item URL.
  • Visual — open the thumb in Thumbcache Viewer; if the preview shows a document, the filename is often visible in the corner / footer of the displayed content. This is the technique behind "explain how to find file name information from a thumbcache finding (preview + cacheID)".
  • Iconcache rows for the same path also encode the associated executable / file extension via shell32.dll icon indexes.
⚠Thumbnails of network and removable drives are cached just like local ones. A thumbcache hit for a file with no corresponding $MFT record on the local volume is strong evidence the user opened the file from a USB, network share, or cloud namespace.
🛠Thumbcache parsing & workflow
Tooling
  • Thumbcache Viewer (Eric Kutcher) — GUI for browsing every thumbnail, exporting all images, viewing cacheID.
  • thumbcache_viewer_cmd — CLI version for batch export and CSV of cacheID ↔ image-hash.
  • Magnet AXIOM — automated extraction with attempted filename correlation against the Search DB.

Typical case workflow:

  1. Export all thumbs at all resolutions into a per-user folder, one image file per cacheID.
  2. Hash with SHA-256 to dedupe across resolution files.
  3. Eyeball the result for sensitive content; for matches, attempt filename recovery via Windows.edb join.
🔁Iconcache — extension & protected file types
Application install evidence

Iconcache files store the icon resource for each registered file extension and ProgID. Per-user — populated as the user encounters new file types. Two practical uses:

  • Confirm an application has been installed on this profile even when it has since been uninstalled — its custom icons remain.
  • Pre-2019 versions of Windows also cached small thumbnails here; can be useful for very old images.
🗑Recycle Bin — $I & $R per-user vault
Recovers metadata + content SID-scoped Shared ID joins the pair
C:\$Recycle.Bin\<user-SID>\$I<ID> (metadata) C:\$Recycle.Bin\<user-SID>\$R<ID> (recovered original content)

Mechanism: when the user "deletes" a file from Explorer, NTFS does two things — it moves the file to C:\$Recycle.Bin\<SID>\ renaming it to $R<random6chars>.<ext>, and writes a parallel $I<samerandom6chars>.<ext> metadata file. The shared random ID is the join key between the two halves.

$I file contents (Win10+: 544 bytes):

  • Header (signature 0x02 for Win10+, 0x01 for Win7/8).
  • Original file size (8 bytes, uint64).
  • Deletion FILETIME (UTC).
  • Original filename length + path (UTF-16). Variable length on Win10+, fixed 260 chars on Win7.

$R file is the renamed original. It is byte-identical to what the user had — open it directly.

Tool: RBCmd (Eric Zimmerman) parses every $I in a tree and emits a CSV with original path, original size, deletion time, owning SID, and the matching $R location.

✓$I files are tiny and incredibly evidentially rich. Even when the user has emptied the Recycle Bin, $I records survive in NTFS slack and unallocated more often than $R bodies do — partial recovery of deletion metadata is common, full file body recovery less so.
📐Per-SID scoping
Attribution

The <SID> subfolder under $Recycle.Bin equals the SID of the user who deleted the file. This is rich attribution data when multiple profiles exist on the host. Join the SID to §07 SAM for the username; cross-check with §09 NTUSER load events for the same user.

On domain-joined systems the SID is a domain SID — resolve via AD or via cached domain SIDs in SOFTWARE\Microsoft\Windows NT\CurrentVersion\ProfileList\.

📜Versions & OS quirks
Format history
  • Win Vista – Win 8.1: $I header signature 0x01; filename field fixed 260 chars (520 bytes) UTF-16.
  • Win 10 (1809+) and later: header signature 0x02; filename field becomes variable-length-prefixed.
  • NTFS-only feature — FAT volumes use a different (simpler) Recycle Bin model lacking $I/$R.
  • Shift+Delete bypasses the Recycle Bin entirely — no $I/$R created. Look to $UsnJrnl (§04) for the deletion record.
  • Recycle Bin on removable drives: each connected USB stick gets its own $RECYCLE.BIN\<SID>\ with the same $I/$R format. Joins to §14 USB.
🎯Practical workflow — recovering "what did the user delete"
End-to-end
  1. Recycle Bin first. RBCmd.exe -d C:\$Recycle.Bin --csv out for every SID subfolder. CSV gives every "still in the bin" deletion with original path, time, and owning user.
  2. Windows.edb for older deletions. Run WinSearchDBAnalyzer. Compare items with high GatherTime to current $MFT presence — rows describing files no longer on disk are deletion candidates pre-dating Recycle Bin retention.
  3. Thumbcache for image evidence. Export thumbs; visually triage; for hits, cross-check whether the original file still exists. Absence + presence of preview is the deletion signal.
  4. USN journal & MFT slack (§04). Pick up the deletion events directly. Slack of $Recycle.Bin\ directories sometimes holds $I bodies even after the bin is emptied.
  5. Carving (§05). For images whose thumbnail was found, attempt full-file carving on the partition. JPEG/PNG headers are easy targets.
  6. Volume Shadow Copies. VSS snapshots often retain a complete pre-deletion view; mount with vshadowmount / KAPE.
Cross-references for corroboration

These three artefact families are independent records of what the user touched, viewed, and deleted — that's why corroborating them against the rest of the case is so productive:

  • §04 NTFS — $MFT / $UsnJrnl — $UsnJrnl records the deletion of $R files when the user empties the bin; $MFT entries for $I files persist briefly in unallocated and slack.
  • §05 File Carving — when $R is gone but thumbcache or Windows.edb partial content names a media file, carving the partition is the next move.
  • §07 SAM — Recycle Bin SID subfolder resolves to the deleting user.
  • §09 NTUSER — RecentDocs & TypedPaths — recently-deleted files were probably recently opened first; RecentDocs often names them.
  • §10 Application Execution — Search Index ingests partial content of Office docs, mail, browser history; AmCache + Prefetch confirm which app the user used to open a now-deleted file.
  • §12 LNK — Recent\*.lnk LNKs for the deleted target preserve target MAC times and volume serial even when both the original and the $R copy are gone.
  • §14 USB — removable-drive Recycle Bins ($RECYCLE.BIN\<SID>\ on the stick) follow the same $I/$R model; join VSN to identify the specific device.
  • §16 Email Forensics — Windows.edb stores indexed Outlook content; Outlook attachments saved-then-deleted show up here long after the email itself was purged.
  • §18 SRUM — explorer.exe + SearchIndexer.exe show as high-resource users when large index rebuilds happen; useful sanity check for "was Search even running at time T?"
  • §19 Event Log Analysis — Microsoft-Windows-Search-ProfileNotify and similar logs record indexer state changes; mass deletions can correlate with Audit Object Access events 4663 when SACLs are enabled.
