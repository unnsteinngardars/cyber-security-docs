Event Log Analysis
Windows EVTX — the auditor's spine of every host-side investigation
Authoritative Time-rich XPath Logon Pivot
TL;DR Windows writes nearly everything it cares about into EVTX files under C:\Windows\System32\winevt\Logs\. Each event is a structured XML record tagged with an Event ID, a Provider, a TimeCreated timestamp, and (when relevant) a SID + LogonId. The investigator's job: scope by channel, filter by Event ID with XPath, and stitch sessions together via Logon ID.

▸ Tooling
🧰
EvtxECmd
ParserBulkCSV/JSON

Eric Zimmerman's EvtxECmd normalises raw .evtx into flat CSV/JSON suitable for Timeline Explorer, ELK, or grep. Map files convert noisy XML payloads into human-friendly columns.

  • EvtxECmd.exe -d C:\Windows\System32\winevt\Logs --csv .\out --csvf evt.csv
  • --maps for community-maintained schema decoders (PowerShell, RDP, Sysmon)
  • Per-file: -f Security.evtx
✦
Always parse from a copy — never the live winevt\Logs dir, since the EventLog service holds handles and dirty-state recovery may rewrite headers.
🔍
XPath
QueryFilter

The native filter language for wevtutil, Get-WinEvent, and the Event Viewer "Filter Current Log" dialog. Targets the EVTX XML schema.

*[System[EventID=4624 and (TimeCreated[@SystemTime>='2026-06-01T00:00:00.000Z'])]]

PowerShell: Get-WinEvent -LogName Security -FilterXPath '*[System[EventID=4624]]'

▸ Artefact #1 — winevt\Logs\*.evtx (the EVTX file collection)
📂
Path & Format
Per-channelSID-attributedBinary XML
C:\Windows\System32\winevt\Logs\*.evtx

Each channel is its own file. Internally a binary XML (EVTX) container with a header, chunks (~64 KB each), and records. The EventLog service appends in-memory and flushes chunks; this is why a hard pull of running disks may leave the "dirty" bit set, and why parsers like EvtxECmd handle truncated final chunks gracefully.

▸ Core channels (sub-artefacts)
🛡️
Security.evtx
AuthAccount

The auditor's bible — only written if the Audit Policy is enabled. Contains logon/logoff, account changes, privilege use, object access (file/registry SACL), policy edits.

If Security.evtx is small or unusually clean, suspect log clearing (Event ID 1102) or audit policy tampering (4719).

⚙️
System.evtx
ServicesBoot

Service start/stop (7036, 7045), driver loads, boot/shutdown (6005/6006), time-changes (1/520), USB plug events (Kernel-PnP), Windows Update activity.

7045 (new service installed) is one of the highest-signal IDs for persistence hunting.

📦
Application.evtx
CrashesInstalls

App-level events: MSI installer (1033/1034), crash reports (Application Error 1000, WER 1001), .NET runtime errors, antivirus product events (Defender 1116/1117).

🖥️
TerminalServices-*
RDPRemote

The RDP forensics goldmine. Found under Microsoft-Windows-TerminalServices-*:

  • RemoteConnectionManager/Operational — 1149 (user authentication succeeded; includes source IP)
  • LocalSessionManager/Operational — 21/22/25/24 (session logon, shell start, reconnect, disconnect)
  • RDPClient/Operational — outbound RDP (1024/1102) on the attacker's jump host
📶
WLAN-AutoConfig
Wi-FiSSID

Microsoft-Windows-WLAN-AutoConfig/Operational. SSID, BSSID, auth type and connection success/fail per attempt — places a laptop on a specific Wi-Fi network at a specific time.

Pivots cleanly with §08 NetworkList\Signatures (last-seen MAC of default gateway) and §15 NetworkList profile GUIDs.

📧
Office Alerts
OfficeMacro

OAlerts.evtx (also under winevt\Logs). Captures the modal dialogs Office shows the user — macro enable prompts, "do you want to save", protected-view warnings.

An OAlerts entry for "Enable Macros?" minutes before a child winword.exe → cmd.exe chain is one of the cleanest phishing-execution narratives you can build.

▸ Record anatomy (every EVTX record exposes these)
🔢
Event ID + Provider

An Event ID is only meaningful scoped to its Provider. EventID=1 means utterly different things under Sysmon (process create) vs PowerShell/Operational (engine start). Always pair System/Provider/@Name with System/EventID.

📊
Event Level

Numeric severity in System/Level:

LevelMeaning
1Critical
2Error
3Warning
4Information
5Verbose
0LogAlways (Security audit-success / failure)
⏱️
TimeCreated

System/TimeCreated/@SystemTime — UTC, sub-millisecond. Single source of truth for sequencing within a host. Cross-host correlation requires confirming each host's clock vs §08 TimeZoneInformation + NTP sync events (System log 35/36/37).

👤
TargetUser / Subject

In Security events, two principals coexist:

  • SubjectUserSid / SubjectUserName — who initiated the action
  • TargetUserSid / TargetUserName — who the action affected

For 4624 (logon) the target is the user who logged on.

🔗
LogonId
Session Key

Hex value uniquely identifying a logon session on a single host (until reboot). Use it to join 4624 (logon) → all activity within that session → 4634/4647 (logoff). See the tip block below.

▸ Concept — RDP (Remote Desktop) sessions, end-to-end

RDP doesn't live in one channel — it fans out across half a dozen. To narrate a single RDP session, stitch:

ChannelIDMeaning
Security4624 (Type 10)RDP logon at the OS authentication layer
Security4778 / 4779Session reconnect / disconnect
TerminalServices-RemoteConnectionManager/Operational1149User authentication succeeded — has Source Network Address
TerminalServices-LocalSessionManager/Operational21Session logon succeeded (Session ID + IP)
TerminalServices-LocalSessionManager/Operational22Shell start notification (desktop visible)

Triage flow: pull 1149 for the candidate source IP → grab the matching 4624 by TimeCreated → harvest the LogonId → fan out into all other Security events with the same TargetLogonId.

▸ Tip — Identify a logon session using the Logon ID
💡
The LogonId pivot

Every 4624 mints a unique TargetLogonId (e.g. 0x4E7A11B). That ID then appears in every subsequent event tied to that session: process creates (4688), object access (4663), privilege use (4672/4673), and finally 4634/4647 at logoff.

The pivot, step-by-step:

  • Find the suspect 4624 (e.g. unusual hour, foreign IP, sensitive account).
  • Note TargetLogonId from its EventData.
  • XPath: *[EventData[Data[@Name='TargetLogonId']='0x4E7A11B' or Data[@Name='SubjectLogonId']='0x4E7A11B']]
  • Filter across Security, Sysmon, and PowerShell logs for that LogonId.
  • Bound the result by the matching 4634 / 4647 — anything outside is a different session.
✓
Why it works: LogonId is the only field that distinguishes two simultaneous sessions for the same user on the same host — e.g. an admin who is RDP'd in (Type 10) while a scheduled task (Type 4) runs as the same account.
!
Caveats: LogonId is reused after reboot — always scope by TimeCreated within a single boot window (anchor with System log 6005/6006). LogonId is also host-local — for the same user logging into another host you get a different LogonId there.
▸ XPath quick-reference

Single ID, single channel:

*[System[EventID=4624]]

Range of IDs:

*[System[(EventID>=4720 and EventID<=4738)]]

Filter on EventData field (e.g. specific user):

*[EventData[Data[@Name='TargetUserName']='svc_backup']]

Combine time + ID + provider:

*[System[Provider[@Name='Microsoft-Windows-Sysmon'] and EventID=1 and TimeCreated[timediff(@SystemTime) <= 86400000]]]

Negate (anything but 4624):

*[System[EventID!=4624]]
