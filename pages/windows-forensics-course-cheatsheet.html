01 Evidence Collection & Disk Acquisition Image first, analyse later — preserve the original
📐Core principles

Acquisition is the act of producing a forensically sound copy of digital media so that the original can be set aside untouched. Everything that follows in the case is built on this copy; if it is contaminated, every later finding is suspect.

  • Bit-for-bit copy — every sector, including unallocated space and slack, not just the live files.
  • Hash before and after — MD5 + SHA-256 of the source and the image. Identical hashes prove the image was not altered.
  • Work on copies — analyse a working copy of the image; keep the master image read-only.
  • Chain of custody — log every transfer, action, and tool version against the evidence ID.
💾Image formats — E01 vs raw (dd)

raw / dd is a flat sector-by-sector copy. Identical to the source on the wire, no compression, no metadata — biggest, simplest, universally readable.

E01 (EnCase Evidence File / Expert Witness Format) is the de-facto forensic container. It stores the bitstream plus embedded metadata, splits into segments (.E01, .E02, …), compresses, and writes per-chunk CRCs.

  • Header carries case number, examiner, evidence ID, acquisition notes.
  • Embedded MD5 / SHA-1 let the format self-verify on every read.
  • CRC per 32 KB chunk localises corruption instead of trashing the whole image.
  • Compression typical 1.3–3× depending on data entropy.
  • Read by virtually every forensic suite (FTK, EnCase, Autopsy, X-Ways, libewf).

Other formats you'll meet: AFF4 (WinPmem, modern, supports memory + disk in one container) and VHD/VHDX/VMDK for VM-image style acquisition.

🛠FTK Imager — the workhorse

FTK Imager (AccessData / Exterro) is free, runs on Windows and from a USB stick, and is the standard tool for taught courses. It can:

  • Image a physical drive, logical volume, or single folder.
  • Write the output as raw dd, E01, SMART, or AFF.
  • Preview a mounted image — file tree, hex view, properties, even Registry hives — before doing a full acquisition.
  • Mount an image read-only as a Windows volume.
  • Capture memory and protected system files (Registry, $MFT) from a live host.

Capture flow → File → Create Disk Image → Physical/Logical → choose format → fill case metadata → set destination (always a different drive!) → verify hashes after.

FTK Imager CLI (E01)
ftkimager \\.\PHYSICALDRIVE1 case01 --e01 --compress 6 --case-number 001 --examiner "U.G."
Live protected-files capture
# File → Obtain Protected Files → grabs SAM, SYSTEM, SOFTWARE, SECURITY from running host
Verify image
# After acquisition, FTK Imager auto-hashes; cross-check with sha256sum case01.E01
🚧Write blockers — Tableau & co.

A write blocker sits between your workstation and the suspect drive, passing reads through but silently dropping every write at the protocol layer. Without one, just attaching the disk lets Windows update last-access times, mount the journal, and break the source hash.

  • Hardware (Tableau, WiebeTech, CRU) — physical bridge for SATA / SAS / USB / NVMe. Forensically validated, court-accepted.
  • Software — Linux blockdev --setro, Windows registry USB-read-only flag, or imaging from a forensic distro (CAINE, SIFT, Tsurugi).
  • Always test — hash the source, mount, attempt a write, hash again. Record the model and firmware on the chain-of-custody form.

Tableau is the most common in classrooms — distinctive orange bridges with status LEDs that light when a write is blocked. Pair the bridge with FTK Imager on the workstation side for the full evidence-sound workflow.

📋Acquisition workflow — quick reference
Before touching the disk
  • Photograph the scene + serials.
  • Power-state decision — pull plug vs. graceful (volatile-data trade-off).
  • Document examiner, witness, timezone, UTC time.
  • Open the chain-of-custody log.
During acquisition
  • Engage write blocker; verify with a test write.
  • Hash source (MD5 + SHA-256).
  • Image to a different physical drive than the source.
  • Choose E01 with compression 5–6 for storage efficiency.
  • Embed case metadata in the E01 header.
After acquisition
  • Verify image hash matches source.
  • Make a working copy — analyse only the copy.
  • Seal original in tamper-evident bag.
  • Record tool versions and command lines.
02 Memory Forensics Volatile evidence — capture before reboot
🧠Why memory matters

Once the machine is powered off, RAM is gone. Memory holds the only copies of running processes, open sockets, decrypted data, command-line arguments, in-memory-only malware, loaded DLLs, clipboard contents, BitLocker / LUKS keys, and Kerberos tickets.

Order of volatility says: RAM first, pagefile/hibernation second, disk last. If you reboot before capture you destroy ~80 % of what an IR memory analyst would have found.

📦What you can recover from RAM
  • Process tree — running, hidden (unlinked), and recently exited.
  • Command-line arguments — full powershell -enc … blobs.
  • Network sockets — listening ports and live C2 connections.
  • Loaded modules / DLLs, handles, mutexes.
  • Injected / hollowed code in legitimate processes.
  • Registry hives mapped in memory (often more up-to-date than on-disk).
  • Credentials — NTLM hashes, Kerberos tickets, plaintext passwords in LSASS.
  • Clipboard, console history, encryption keys.
⚡Magnet RAM Capture

Free GUI tool from Magnet Forensics. Tiny footprint — designed to perturb the system as little as possible. Outputs a raw .raw/.dmp ready for Volatility.

Flow
# Run as Admin → choose segment size → output path → Start. Single button.
⚡DumpIt

Single self-contained executable from Comae / MoonSols. Double-click and say yes — produces HOSTNAME-DATETIME.raw beside the binary. Newer Comae builds output the Microsoft crash-dump format directly so the file is usable in WinDbg as well.

CLI use
DumpIt.exe /OUTPUT E:\mem\host.raw /Q # quiet, no prompts
⚡WinPmem

Open-source acquirer from the Rekall / Velociraptor team. Loads a signed kernel driver to walk the physical-address map directly. Writes the modern AFF4 container (compressed, indexed, can bundle the pagefile and drivers alongside RAM).

Quick capture
winpmem_mini_x64.exe mem.aff4
Convert to raw for Volatility
winpmem_mini_x64.exe mem.aff4 --export PhysicalMemory --output mem.raw
📝Acquisition gotchas
  • Write the dump to external media — never to the system disk you may want to image later.
  • Hash the dump immediately after capture.
  • Run as Administrator/SYSTEM, EDR allowing.
  • Expect the dump to be the size of physical RAM (8 GB host → ~8 GB file).
  • Snapshots from VMs (.vmem, .vmsn) are an equivalent, zero-touch alternative.
💤hiberfil.sys — frozen RAM on disk

C:\hiberfil.sys is created when the system hibernates (S4). Windows compresses RAM and writes it to this file so the next boot can restore the session. From a forensic standpoint it is a past snapshot of RAM — sometimes weeks or months old.

  • Format: Xpress-Huffman compressed pages with a small header (HIBR / RSTR magic). Windows 8+ uses a newer slot-based layout.
  • hibr2bin (Comae) or Volatility's imagecopy converts to a flat raw memory image that Volatility can ingest.
  • Even on systems that never visibly hibernate, hybrid-sleep and Fast Startup write a partial hiberfil — useful pre-incident forensic value.
  • Disabled by powercfg /h off; absence ≠ system was never powered down.
Convert to raw
hibr2bin.exe /PLATFORM X64 /MAJOR 10 /MINOR 0 hiberfil.sys mem.raw
Volatility direct read
vol -f hiberfil.sys windows.info # auto-detects compression
📄pagefile.sys / swapfile.sys

C:\pagefile.sys is virtual-memory backing store — pages the kernel evicted from RAM. swapfile.sys is the Windows 8+ companion that holds suspended Modern (UWP) apps. Together they are a messy, unstructured second tier of RAM.

  • No file system, no allocation map — pages may be from any process and may be partial.
  • Parse with bulk_extractor, strings, YARA, or Volatility running over a combined raw image.
  • Recoverable: URL fragments, IM messages, plaintext passwords, malware strings, command-line fragments, BitLocker recovery markers.
  • Without a memory tool: strings -n 8 pagefile.sys | grep -E "https?://|@|cmd.exe" still surfaces useful leads.

Always grab pagefile.sys, swapfile.sys, and hiberfil.sys together with the RAM dump — they're protected files, so use FTK Imager's Obtain Protected Files or KAPE's !SANS_Triage target.

🔍Memory artefacts without a memory tool

If you have nothing but the raw .raw / .mem / pagefile, you can still do meaningful work with general-purpose tools — useful when Volatility lacks a profile for the OS build.

strings + grep
strings -el mem.raw | grep -iE "powershell|http://|@\w+\.com|password"
bulk_extractor
bulk_extractor -o out/ mem.raw # emails, URLs, CCNs, IPs, AES keys
YARA scan
yara -r rules/ mem.raw # malware family identification
03 KAPE & Arsenal Image Mounter Fast triage and read-write-safe image mounting
🗜Arsenal Image Mounter (AIM)

Arsenal Image Mounter takes a disk image (E01, raw, VHD/VHDX, VMDK) and presents it to Windows as a real disk — not just a logical volume. Because Windows sees a full disk, you can:

  • Mount and explore the file system natively in Explorer.
  • Run anti-virus and EDR scanners against the image.
  • Boot the image as a VM with BIOS / UEFI launch mode — actually start the suspect OS in a VM without modifying the original.
  • Cache writes to a temporary differencing file so the underlying image stays read-only — analyst can poke around without breaking integrity.
  • Mount and decrypt BitLocker volumes given the recovery key / TPM bypass.

Why not FTK Imager mount? FTK mounts volumes; AIM mounts disks. Drivers, services, EDR signatures, and offline-Registry tools all expect a real disk and break on a volume-only mount.

⚙KAPE — the idea

KAPE = Kroll Artefact Parser & Extractor. Built by Eric Zimmerman. Two-stage triage tool:

  • Targets — collect a curated set of forensic artefacts (files / Registry hives) from a live or mounted system. Fast, surgical, doesn't grab the whole disk.
  • Modules — run third-party parsers (the EZ-Tools, RegRipper, Hayabusa, etc.) over what the Targets gathered, producing CSV/JSON ready for Timeline Explorer.

Typical use: 5–10 minutes of triage on a 1 TB host produces a few GB of parsed CSV instead of a multi-hour full image. Perfect for "is this host worth a full image?" and for environments with hundreds of endpoints.

🎯Targets — what gets pulled

Targets are .tkape YAML-style files in KAPE\Targets\. Each lists glob patterns to copy verbatim, preserving paths.

  • !SANS_Triage — the canonical "grab everything triage" compound target.
  • EvidenceOfExecution — Prefetch, AmCache, BAM/DAM, UserAssist.
  • FileSystem — $MFT, $J, $LogFile, $Boot.
  • RegistryHives + RegistryHivesUser — SAM, SYSTEM, SOFTWARE, SECURITY, NTUSER.DAT, UsrClass.dat.
  • EventLogs — every .evtx under winevt\Logs.
  • Browser, LNK, Jump Lists, Recycle Bin, ScheduledTasks, USB, Volume Shadow Copies.

Targets understand Volume Shadow Copies — set --vss and KAPE will also pull each artefact from every shadow.

⚙Modules — parsing the haul

Modules are .mkape files in KAPE\Modules\. Each is a thin wrapper that runs a parser binary and routes its output.

  • !EZParser — the compound module that fires every EZ-Tools parser. The 80 % button.
  • MFTECmd, PECmd, LECmd, JLECmd, SBECmd, RECmd, EvtxECmd, AmcacheParser, AppCompatCacheParser…
  • Hayabusa / Chainsaw modules for Sigma-rule hunting over the collected EVTX.
  • Outputs land in %mdest%\HOSTNAME\ as CSV ready for Timeline Explorer.
🖱gkape — the GUI

gkape.exe is a graphical front-end that builds the CLI command for you.

  • Target source/dest on the left, Module source/dest on the right. Source for modules is usually the same path as the target destination — chain them.
  • Tick Targets and Modules from filterable lists.
  • Toggles: VSS, Add to Zip, Flush destination, Process VSCs.
  • Bottom panel shows the equivalent kape.exe command line — copy it for scripts and SOAR playbooks.

gkape is the friendliest way to learn KAPE; once the command line is familiar, run kape.exe headless from a USB / network share.

📁Typical artefacts to extract
  • $MFT, $UsnJrnl:$J, $LogFile
  • All Registry hives (system + per-user)
  • EVTX — Security, System, PowerShell-Operational, Sysmon
  • Prefetch (*.pf), Amcache.hve
  • LNK files + AutomaticDestinations / CustomDestinations
  • Browser History (Chromium, Firefox, Edge)
  • SRUM database (SRUDB.dat)
  • Scheduled tasks (System32\Tasks\)
⌨KAPE one-shot — collect & parse
Triage collection (Target stage)
kape.exe --tsource C: --target !SANS_Triage --tdest E:\Triage\%m --tflush --vss --zip %m
Parse the collection (Module stage)
kape.exe --msource E:\Triage\HOST --module !EZParser --mdest E:\Parsed\%m --mflush
Combined (single invocation)
kape.exe --tsource C: --tdest E:\Triage --target !SANS_Triage --msource E:\Triage --mdest E:\Parsed --module !EZParser --vhdx %m
Useful flags
# %m=machine name, %d=date. --debug for verbose log. --gui to launch gkape with prefilled values.
04 Windows Filesystem & NTFS $MFT · MACB · journals · ADS · VSS
🏗Core concepts — sectors, clusters, volumes, files

Sector = the smallest physical IO unit of the disk (typically 512 B, modern drives 4 KB / "Advanced Format"). Cluster = the smallest unit NTFS allocates to a file — a power-of-two group of sectors, usually 4 KB on Windows. Volume = a logical container formatted with a filesystem (NTFS, ReFS, FAT32, exFAT). File = a named collection of cluster runs plus metadata recorded in the MFT.

NTFS is a journaling, transactional, attribute-oriented filesystem. Everything — files, directories, even the metadata describing the filesystem itself — is a file with attributes. The "everything is a file" rule is the key insight that makes the rest of NTFS forensics make sense.

📚$MFT — the Master File Table

The $MFT is the master index of every file and directory on the volume. One record per file, 1024 bytes by default. The first 16 records are reserved for NTFS metadata files ($MFT, $LogFile, $Volume, $Bitmap, …). Records are referenced by MFT entry number.

Each record is a header followed by attributes. The forensically critical ones:

  • $STANDARD_INFORMATION ($SI, type 0x10) — the four timestamps most tools display (created / modified / MFT-changed / accessed), DOS-style flags. Easily modified by user-mode APIs → spoofable.
  • $FILE_NAME ($FN, type 0x30) — filename, parent directory reference, and another set of MACB timestamps. Only the kernel updates it during create/rename → much harder to spoof.
  • $DATA (type 0x80) — the actual file content (resident or non-resident).
  • $INDEX_ROOT / $INDEX_ALLOCATION — directory contents.
  • $ATTRIBUTE_LIST — present when a file has so many attributes they spill into other MFT records.
⏱MACB timestamps

Four timestamps per attribute, written as M·A·C·B:

  • M — Modified: file content last written.
  • A — Accessed: file last read. Often disabled on modern Windows for performance (NtfsDisableLastAccessUpdate) — don't trust if absent.
  • C — MFT-Changed: any metadata change to the MFT record (rename, permission change, attribute add).
  • B — Born / Created: file creation time.

Both $SI and $FN carry their own MACB set. Tools commonly display the $SI quartet by default; ask for both to spot tampering.

SourceUpdated byTamperable?
$SI MACBUser-mode + kernelYes (SetFileTime)
$FN MACBKernel only (create/rename)Hard — anti-forensics tell
💼Resident vs non-resident data

An MFT record is 1024 bytes. If the file's $DATA attribute fits in what's left after the header and other attributes (~700 bytes), NTFS stores the entire file inside the MFT record itself — this is a resident file.

If the file is larger, $DATA becomes non-resident: the MFT record only stores a list of cluster runs (start-cluster + length) pointing into the disk where the data really lives.

  • Resident files have no on-disk clusters, so file-carving tools miss them — the MFT is the only copy.
  • Deleting a resident file leaves the content in the MFT slack until the record is reused — easy recovery via $MFT alone.
  • Non-resident files leave their clusters in unallocated space when deleted — carving territory.
🕵Timestomping & how to spot it

Timestomping = an attacker rewrites a file's MACB timestamps to blend with surrounding files or pre-date the incident. Tools: SetMACE, timestomp.exe (Metasploit), PowerShell (Get-Item).CreationTime = ….

Detection clues — compare $SI vs $FN:

  • $SI timestamp earlier than $FN — impossible naturally: $FN is set at create/rename, $SI starts equal and only moves forward.
  • $SI timestamps with whole-second precision while $FN has 100-ns granularity — most stomping APIs zero the sub-second.
  • Identical M/A/C/B values on $SI — all four wiped to one moment.
  • $SI timestamp inconsistent with $UsnJrnl / $LogFile entries for the same file.
  • Cross-check Prefetch, Amcache, Event Logs — if those show execution at a different time, $SI is the liar.
📜$LogFile & $UsnJrnl

$LogFile — NTFS transaction log used for crash recovery. Records low-level metadata changes as redo/undo entries. Small (~64 MB), so it wraps quickly — useful for recent activity, not long-term history. Parse with LogFileParser or NTFSLogTracker.

$UsnJrnl:$J — Update Sequence Number Journal under \$Extend\$UsnJrnl. A change log of every file create / rename / write / delete / security-change on the volume, with a reason code. Far larger and longer-lived than $LogFile.

  • Surfaces files that no longer exist — the journal records the create + delete even after the MFT record is reused.
  • Reason codes (FILE_CREATE, RENAME_OLD_NAME, DATA_OVERWRITE, CLOSE) reconstruct the lifecycle.
  • Excellent for detecting ransomware (mass RENAME) and wipers (mass DELETE).
  • Parse with MFTECmd -f $J --csv ….
🎭Alternate Data Streams (ADS)

NTFS lets a single file have multiple $DATA attributes, each with a name. The unnamed default stream is the "normal" content; named streams (file.txt:hidden.exe) are Alternate Data Streams. Invisible to dir, type, and most file browsers — used by attackers to hide payloads, and by Windows itself for benign purposes.

  • Zone.Identifier ADS — Mark-of-the-Web. Set by browsers / email clients on downloaded files: [ZoneTransfer] ZoneId=3. Goldmine for proving "this file came from the internet on this date".
  • Attackers stash EXE/PS1 in ADS, then launch via wmic process call create or powershell -.
  • Enumerate: dir /r, Get-Item file -Stream *, streams.exe (Sysinternals).
  • Forensic tools list them automatically — MFTECmd, FTK, Autopsy.
🌑Volume Shadow Copies (VSS)

VSS is the Windows snapshot framework. The Volume Shadow Service periodically (and at restore-point / backup events) captures a point-in-time copy of changed blocks. Forensically: multiple historical versions of the entire volume sitting on disk.

  • List snapshots: vssadmin list shadows.
  • Access via \\?\GLOBALROOT\Device\HarddiskVolumeShadowCopyN\.
  • Mount with vshadowmount (libvshadow, Linux) or expose to KAPE with --vss.
  • Recover deleted files, prior Registry hives (great for spotting clean-up), pre-ransomware originals.
  • Attacker counter-move: vssadmin delete shadows /all — flag in System log + Sysmon.
🔧Parsing with MFTECmd + Timeline Explorer

MFTECmd (Eric Zimmerman) parses $MFT, $J, $LogFile, $Boot, $SDS to CSV. Timeline Explorer is his interactive CSV viewer with Excel-style filters, grouping, highlight rules, and bookmarks. The pair is the day-to-day NTFS workflow.

Parse the MFT
MFTECmd.exe -f $MFT --csv out --csvf mft.csv
Parse the UsnJrnl
MFTECmd.exe -f $J --csv out --csvf usnjrnl.csv
Parse $LogFile
MFTECmd.exe -f $LogFile --csv out
Open in Timeline Explorer
TimelineExplorer.exe out\mft.csv # filter columns, colour rules, pivot on entry number

05 File Carving & Deleted Data When the MFT says gone, the bytes often aren't
🗑What "deleted" really means on NTFS

Deleting a file on NTFS is a metadata bookkeeping change, not data destruction:

  • The MFT record's in-use flag (header byte at offset 0x16) is cleared — the record is now eligible for reuse.
  • The clusters listed in the file's $DATA cluster runs are marked free in $Bitmap — eligible to be overwritten.
  • The filename + parent reference + MACB times stay in the record until something reuses it.
  • The cluster contents stay on disk until something else writes there.

So "recently deleted" usually means "still entirely intact". The window shrinks every time the volume is written to.

🕳Where deleted & hidden data live
  • Unallocated clusters — old file content, full or partial.
  • Slack space — the unused tail of the last cluster of a smaller file; data from a previous, larger occupant.
  • MFT entry slack — leftover bytes inside a reused MFT record.
  • $Recycle.Bin — per-SID folder; $I… = metadata (original path, size, deletion time), $R… = content. User-emptied bin → still in unallocated until overwritten.
  • Volume Shadow Copies — historical snapshots may still have the file.
  • Alternate Data Streams — hidden, not deleted.
  • Pagefile / hiberfil — fragments paged out from RAM.
  • System Volume Information, $Extend — system files that don't show in Explorer.
📚Recover via $MFT

If the MFT record hasn't been reused, the original filename, MACB and cluster runs are still there. Tools follow the cluster runs back into unallocated space and rebuild the file. Best fidelity — you get the original name and metadata, not just bytes.

  • MFTECmd / Autopsy / FTK do this automatically — "deleted" files show in the listing.
  • Resident files recover 100 % from the MFT record itself.
🌑Recover via VSS

If a Volume Shadow Copy exists from before the deletion, just pull the file out of the snapshot. Best-case recovery — original file, original timestamps, full provenance.

vssadmin list shadows # then robocopy \\?\GLOBALROOT\...VolumeShadowCopy3\Users\u\Documents out\ /E
🧰Autopsy / Autorec

Autopsy (Sleuth Kit GUI) parses NTFS, surfaces deleted files via MFT + carving, and indexes content for keyword search. The IngestModule "PhotoRec Carver" runs PhotoRec over unallocated. Autorec automates running Autopsy / EZ-Tools against a triage collection.

🔬Hex editor — last resort

When tools fail or you need to verify a structure by hand, open the image in HxD / 010 Editor / wxHexEditor, jump to the cluster offset (cluster × cluster_size) and read raw bytes. Useful for partial recovery, malformed files, and proving exactly what's on disk.

✂File carving — the concept

File carving recovers files based on their internal structure, ignoring the filesystem entirely. The carver scans raw bytes for known magic numbers / file signatures (file headers) and known footers, then extracts everything in between.

  • JPEG — header FF D8 FF E0 / FF D8 FF E1, footer FF D9.
  • PNG — header 89 50 4E 47 0D 0A 1A 0A, footer 49 45 4E 44 AE 42 60 82.
  • PDF — header 25 50 44 46 (%PDF), footer %%EOF.
  • ZIP / Office (docx/xlsx) — header 50 4B 03 04.
  • PE / EXE — header 4D 5A (MZ).
  • SQLite — header 53 51 4C 69 74 65 20 66 6F 72 6D 61 74 20 33 00.

Tools: PhotoRec (broadest signature DB), Scalpel (config-driven), Foremost, bulk_extractor (features rather than full files — emails, URLs, IPs, AES keys).

🚧Limitations of carving
  • No filename, no path, no timestamps — carving recovers content only. The MFT is what carries the metadata.
  • Fragmentation breaks carving — if the file's clusters aren't contiguous, simple header-to-footer carving stitches together garbage.
  • No footer = no end — formats without a reliable footer (raw text, MP3) are guessed at size and often over- or under-recovered.
  • Overwrites are fatal — even one cluster overwritten and the file is corrupted past easy repair.
  • Encrypted / compressed content has no recognisable signature in the bytes.
  • Resident NTFS files have no clusters in unallocated and so are invisible to a carver — recover via $MFT.
  • Lots of false positives — signatures collide with embedded thumbnails, screenshots in documents, etc. Verify by opening.

EZ-Tools GUI — load offline hive files, browse the tree, see deleted keys / values recovered from unallocated cells, apply bookmarks that label forensically interesting keys with a hover-over explanation. Best learning tool.

CLI counterpart to Registry Explorer. Runs batch files (.reb) — collections of regex queries against many keys at once. KAPE's RECmd_Kroll batch is the de-facto sweep.

📐Choosing between them
  • Learning a key → Registry Explorer + bookmarks.
  • Batch / scripted sweep → RECmd + Kroll batch.
  • Quick triage report → RegRipper.
  • All three handle offline hives — use the files KAPE extracted, not the live Registry.
👤SAM — local accounts in detail

The SAM hive (Security Account Manager) holds the database of local Windows accounts under SAM\Domains\Account\Users\. Domain accounts and Microsoft / Azure-AD accounts are not here — they live in AD or are profile-only on this host.

  • RID (Relative Identifier) — last sub-authority of the user's SID. Built-ins are fixed: 500 Administrator, 501 Guest, 503 DefaultAccount, 504 WDAGUtilityAccount. First created local user → 1000, then 1001, 1002…
  • Per-user values: F (account flags, last-logon, failed-login count, password-fail time), V (username, full name, comment, hash containers).
  • Useful properties: account-disabled bit, password-never-expires, must-change-on-next-logon.

Cloud / Microsoft accounts aren't in SAM — they show up only as profile folders (C:\Users\<name>) and in SOFTWARE\Microsoft\Windows NT\CurrentVersion\ProfileList\<SID>. The SID's ProfileImagePath ties the SID to the on-disk username; for MS-account users the last login values plus the IdentityCRL keys link the local SID to the cloud email — that linkage is how you prove a real human used the account.

💻SYSTEM + SOFTWARE — profiling the machine

Together they answer what machine is this, what OS, when was it installed, who configured it?

  • OS version & install date: SOFTWARE\Microsoft\Windows NT\CurrentVersion — ProductName, CurrentBuild, DisplayVersion, InstallDate (epoch).
  • Computer name: SYSTEM\ControlSet001\Control\ComputerName\ComputerName.
  • Time zone: SYSTEM\…\Control\TimeZoneInformation — critical for converting all timestamps consistently.
  • Network interfaces & DHCP hints: SYSTEM\…\Services\Tcpip\Parameters\Interfaces\{GUID} — IP, gateway, DHCP server, lease times, last-known networks.
  • Network profiles (SSIDs): SOFTWARE\Microsoft\Windows NT\CurrentVersion\NetworkList\Profiles & Signatures\Unmanaged — name of every Wi-Fi network the host joined + first-/last-connect.
  • Installed applications: SOFTWARE\Microsoft\Windows\CurrentVersion\Uninstall (and WOW6432Node for 32-bit).
  • App permissions (camera, mic, location): SOFTWARE\Microsoft\Windows\CurrentVersion\CapabilityAccessManager\ConsentStore\*.
  • Autoruns: SOFTWARE\Microsoft\Windows\CurrentVersion\Run, RunOnce; per-user equivalents in NTUSER.DAT.
  • Shutdown time: SYSTEM\ControlSet001\Control\Windows → ShutdownTime (FILETIME).

CurrentControlSet is a runtime alias for the active ControlSet (usually ControlSet001); on offline hives you read the explicit number, picking the one named in Select\Current.

🗂NTUSER.DAT — per-user activity gold

One NTUSER.DAT per profile (C:\Users\<user>\NTUSER.DAT). The richest single source of "what did the user do on this machine".

KeyPath under NTUSER.DATForensic value
WordWheelQuery\Software\Microsoft\Windows\CurrentVersion\Explorer\WordWheelQueryEvery search term typed into Explorer's search box — direct insight into intent
TypedPaths\Software\Microsoft\Windows\CurrentVersion\Explorer\TypedPathsPaths the user typed into the Explorer address bar (incl. UNC shares)
RecentDocs\Software\Microsoft\Windows\CurrentVersion\Explorer\RecentDocsRecently opened documents, bucketed by extension; ordered MRU
RunMRU\Software\Microsoft\Windows\CurrentVersion\Explorer\RunMRUEverything typed into Win+R — fast lookup of suspect commands and tools
Office File MRU\Software\Microsoft\Office\<ver>\<app>\User MRU\…\File MRUPer-application recent files (Word, Excel, …) with last-opened timestamps

Why these matter together: WordWheelQuery + TypedPaths + RunMRU = intent and tooling. RecentDocs + Office MRU + OpenSavePidlMRU + LastVisitedPidlMRU = what files were touched, by which app, when. TrustRecords = which malicious documents the user explicitly approved.

🗃UsrClass.dat — Shellbags & per-user app state

UsrClass.dat lives at C:\Users\<user>\AppData\Local\Microsoft\Windows\UsrClass.dat and is mounted as HKCU\Software\Classes. Its forensic crown jewels are Shellbags:

  • Path: Local Settings\Software\Microsoft\Windows\Shell\Bags + BagMRU.
  • Records that a user navigated into a folder in Explorer — even if the folder was on a USB drive, a network share, or has since been deleted.
  • Surfaces folder names, layouts, and the rough last-time-this-folder-was-viewed.
  • Parsed by SBECmd (EZ Tools).

Other useful keys under UsrClass: MUICache (display names of every binary the user launched), LocalServer32 / InprocServer32 COM registrations (persistence), and per-user file-association overrides.

Parse Shellbags
SBECmd.exe -d C:\KAPE\Users --csv out
07 Application Execution Artefacts Did it run? When? How often? By whom?
🎯The execution-evidence stack

No single artefact reliably proves "this binary ran". Forensic confidence comes from multiple artefacts agreeing. Each one captures execution differently — Prefetch tracks user-mode launches, ShimCache tracks the compat-check, Amcache tracks file-system scans, UserAssist tracks Explorer-launched GUIs, BAM/DAM tracks the OS scheduler's view. Disagreement is itself a clue (timestomping, anti-forensics).

📅BAM — Background Activity Monitor

Windows kernel service that tracks per-user background activity for power management. Hive path:
SYSTEM\CurrentControlSet\Services\bam\State\UserSettings\<SID>

  • One value per executable the user ran, named after the full NT-format path (\Device\HarddiskVolume1\Windows\System32\notepad.exe).
  • Data carries the last-execution FILETIME.
  • Persists across reboots, ties execution to a user SID.
  • Survives long after the binary is deleted — easy proof of "this user ran this tool at this time".
🖥DAM — Desktop Activity Moderator

The companion to BAM for foreground UWP / Modern app activity. Hive path:
SYSTEM\CurrentControlSet\Services\dam\State\UserSettings\<SID>

  • Same shape as BAM: value per executable, last-execution FILETIME.
  • Captures desktop-side power-managed apps.
  • Combine BAM + DAM for a full per-user execution timeline.
  • Parse via RegRipper (bam.pl) or RECmd Kroll batch — output is CSV ready for Timeline Explorer.
⌨RunMRU

Stores everything typed into Win+R. NTUSER.DAT key:

\Software\Microsoft\Windows\CurrentVersion\Explorer\RunMRU
  • One value per entry (a, b, c…) with command text + a trailing \1.
  • MRUList value gives the most-recent order.
  • Quick way to spot operator-style usage: cmd, powershell, \\dc01\c$, mmc.
🖱UserAssist

Tracks GUI applications launched via Explorer / Start menu. NTUSER.DAT key:

\Software\Microsoft\Windows\CurrentVersion\Explorer\UserAssist\{GUID}\Count
  • Value names are ROT13-encoded paths — decode to read.
  • Records run count, focus count, focus time, last-execution time.
  • Misses CLI launches and process-launched-by-other-process — pair with BAM / Prefetch.
🔁FeatureUsage

Windows 10+ telemetry-ish key under NTUSER.DAT:

\Software\Microsoft\Windows\CurrentVersion\Explorer\FeatureUsage
  • Sub-keys: AppLaunch, AppSwitched, ShowJumpView, TrayButtonClicked.
  • Per-app launch counts and taskbar interaction — useful for proving repeated, deliberate use.
  • Granularity per application identifier (AppID), not full path.
🚀Prefetch

Files at C:\Windows\Prefetch\<EXE>-<hash>.pf. Windows pre-loads commonly-used binaries to speed launches — the side-effect is rich execution evidence.

  • Last eight execution timestamps (Win10+).
  • Run count.
  • List of every file and DLL referenced during the first 10 s of execution — often surfaces tooling and config files the binary touched.
  • Volume serial + path the binary was launched from — proves location, even for binaries since deleted.
  • Disabled by default on SSD-only systems in some Win11 builds — absence ≠ wasn't run.
🧱ShimCache (AppCompatCache)

SYSTEM hive key — populated by the Application Compatibility engine. Hive path:

SYSTEM\CurrentControlSet\Control\Session Manager\AppCompatCache\AppCompatCache
  • One entry per binary the compat engine examined — not strictly proof of execution on modern Windows (more "was seen / checked").
  • Carries: full path, $SI last-modified timestamp, sometimes an "executed" flag (Win7) — flag dropped in Win8+.
  • Written to disk only at shutdown — live extraction is partial; image-after-reboot is empty for the most recent activity.
  • Bounded (~1024 entries on Win10) — old entries drop off.
  • Excellent for spotting binaries that no longer exist on disk.
🗄AmCache

Hive at C:\Windows\AppCompat\Programs\Amcache.hve. Populated by the Program Compatibility Assistant whenever a binary is scanned.

  • Per-binary entries with: full path, SHA-1 hash of the file, file size, PE link-time, first-seen timestamp on this host, publisher / product / version.
  • Recoverable even after the binary is deleted — the hash alone is investigative gold.
  • Covers a far wider set of artefacts than Prefetch: every .exe the OS noticed, not just ones that ran often.
  • Replaces RecentFileCache.bcf on modern Windows.

👀WinPrefetchView — quick GUI

NirSoft GUI for Prefetch. Doesn't replace PECmd for casework but is invaluable for fast browsing on the analyst workstation.

  • Loads the local Prefetch\ folder by default; Advanced Options → Prefetch Folder to point at a copy from KAPE.
  • Upper pane: each PF — process, run count, last run, creation, file path.
  • Lower pane: every file/DLL referenced during execution — quick "what did this binary touch".
  • Export selected rows to CSV / TSV / HTML for reports.
