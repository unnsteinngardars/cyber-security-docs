⚡Why memory matters — order of volatility

RAM evaporates the moment the box loses power. Disk survives. So the rule from RFC 3227 is simple: capture the most volatile thing first. RAM → hiberfil / pagefile → disk → backups.

A lot of what an examiner actually wants only ever exists in memory on a running host:

  • Process tree with parent/child links — who spawned powershell.exe from winword.exe?
  • Command lines — including ones obfuscated, base64-blob'd, or wiped after launch.
  • Network sockets — live TCP/UDP endpoints, owning PID, remote IP.
  • Loaded DLLs & injected code — reflective loaders, process hollowing, unbacked executable pages.
  • Registry hives mapped in memory — values not yet flushed to disk.
  • LSASS credentials — NTLM hashes, Kerberos tickets, plaintext passwords (mimikatz territory).
  • Clipboard, console history, encryption keys — BitLocker FVEK, TrueCrypt/VeraCrypt, SSH agent keys.

If you pull the plug before dumping, you lose all of that. Forever.

⛔Live host found running? Memory image before anything else. Even before the disk image — the second you hit Tableau the source hash includes a now-stale RAM you can't recover.
📦RAW vs AFF4 — the output formats

RAW (a.k.a. .raw, .bin, .dmp, .mem) is a flat byte-for-byte copy of physical memory. No headers, no compression, no metadata — just RAM as it was, padded over reserved regions. Size on disk ≈ installed RAM.

  • Universally readable — every parser eats raw without fuss.
  • Volatility 3 / MemProcFS expect raw by default.
  • No self-verification — you must hash and store hashes separately.

AFF4 (Advanced Forensic Format v4, used natively by WinPmem) is a modern container. Think ZIP-like archive holding multiple streams with embedded metadata and per-chunk hashes.

  • Single file can carry memory + pagefile + hiberfil + drivers + system info.
  • Snappy / deflate compression — typical 2–4× shrink on RAM.
  • Self-describing: acquisition time, host name, tool version, page table base.
  • Self-verifying: SHA-256 per chunk catches bit rot on storage.
  • Convert to raw for Volatility with winpmem.exe --export PhysicalMemory -o mem.raw input.aff4.

Rule of thumb: capture in AFF4 if the tool supports it (full evidence bundle), then export a raw copy for analysis.

🧲Magnet RAM Capture — GUI, minimal footprint

Magnet RAM Capture (Magnet Forensics, free) is the easy-button tool: tiny GUI, one button, dumps to .raw / .dmp. Designed to leave the smallest possible memory footprint so it overwrites as little of the target RAM as possible.

  • Single-screen UI — pick output path, click Start.
  • Output: raw only. No AFF4, no pagefile bundling.
  • Run as Administrator from external media (USB / network share).
  • Pick when: responder needs a click-and-go capture and doesn't trust their own typing under pressure.
Launch (must be elevated)
# From responder USB, right-click → Run as administrator E:\MagnetRAMCapture.exe
Hash immediately after
certutil -hashfile HOST01-mem.raw SHA256
💉DumpIt — single-EXE quiet capture

DumpIt (originally MoonSols, now Comae / Magnet) is a single self-contained executable. No installer, no GUI by default — just drop and run. Output is a raw Microsoft crash dump-style image saved next to the binary, named after the host.

  • Zero install, ~1 MB binary — fits on any IR thumb-drive.
  • Modern builds also emit .dmp (Windows crash-dump format), which WinDbg and Volatility both read.
  • Flags for unattended capture: /Q (quiet), /O (output path), /T (output type).
  • Pick when: scripted IR, locked-down kiosk, or you need to drop a single file via PsExec / WinRM.
Interactive (confirm + dump next to EXE)
DumpIt.exe
Quiet, scripted, to external share
DumpIt.exe /Q /T RAW /O \\IR-SHARE\dumps\HOST01.raw
🔧WinPmem — signed driver, AFF4, the kitchen sink

WinPmem (Rekall / Velocidex) is the open-source heavyweight. Ships an embedded signed kernel driver (works on Secure Boot / driver-signing-enforced hosts) and writes natively to AFF4, which lets you bundle physical memory plus the pagefile, hiberfil, loaded driver list and a copy of the binary itself into one self-verifying container.

  • CLI only — built for IR scripting. Same binary for Win7 → Win11 / Server 2022.
  • Can write raw or AFF4. AFF4 mode is the differentiator.
  • --volume_format aff4 + -p pagefile.sys bundles paging into the same evidence archive.
  • Loads its driver, dumps, unloads cleanly — leaves a minimal disk & registry footprint.
  • Pick when: you want the full evidence bundle in one file, or the host has Secure Boot and the other two tools' drivers won't load.
Plain raw dump
winpmem_mini.exe HOST01.raw
Full AFF4 bundle: RAM + pagefile + drivers
winpmem.exe --format map --volume_format aff4 -p C:\pagefile.sys -o HOST01.aff4
Export raw memory stream from AFF4 for Volatility
winpmem.exe --export PhysicalMemory -o HOST01.raw HOST01.aff4
ℹ️AFF4 streams are addressable like files inside the container — --list shows every embedded artefact (memory, pagefile, drivers, sysinfo) before you export.
🛏️hiberfil.sys — RAM frozen to disk

C:\hiberfil.sys is Windows' hibernation file. When the user hibernates, the kernel compresses physical memory with Xpress-Huffman and writes it here, then the box powers off. On resume, the kernel decompresses it back into RAM. Forensically: a free historical RAM snapshot, sitting on disk.

  • Magic bytes: HIBR (or RSTR after resume — partially overwritten).
  • Size ≈ 40–75% of installed RAM (configurable via powercfg /h size N).
  • Convert to a flat raw image with hibr2bin (Comae) or Volatility 3's hibernation layer.
  • Hybrid Sleep and Fast Startup (default on Win10/11 laptops) write a partial hiberfil even when the user thinks they never hibernate — always check.
  • Lives on the disk image — extract from the E01 you collected in §01 before parsing.
Convert hiberfil to raw (Comae hibr2bin)
hibr2bin.exe /PLATFORM X64 /MAJOR 10 /MINOR 0 /INPUT hiberfil.sys /OUTPUT hiber.raw
Volatility 3 — parse hiberfil directly
vol -f hiberfil.sys windows.pslist
📄pagefile.sys — paged-out fragments

C:\pagefile.sys is Windows' on-disk swap. When physical RAM is tight, the kernel pages out cold memory regions here. Not a memory image on its own — there's no structure, no headers, just a soup of 4 KB pages — but it commonly holds fragments of process memory that have since been overwritten in live RAM.

  • Locked by the kernel while running — grab it via FTK Imager Obtain Protected Files or from the §01 disk image.
  • Carve it: strings, URLs, command-line fragments, plaintext credentials, base64 blobs.
  • Pair with the live RAM dump in Volatility 3 (--swap-locations) to resolve paged-out process memory.
  • swapfile.sys is the modern Windows app (UWP) swap — smaller, but check it too.
  • Wiped on shutdown only if ClearPageFileAtShutdown=1 is set (rare).
Quick triage — strings of length 6+
strings -n 6 pagefile.sys > pf-strings.txt
Volatility 3 with pagefile attached
vol -f HOST01.raw --swap-locations file:///cases/pagefile.sys windows.cmdline

⚙️Acquisition gotchas & the analysis hand-off

Same workflow regardless of tool — only the flags change:

  • Run as Administrator. No exceptions — every capture tool loads a kernel driver.
  • Write to external media (USB / network share). Writing the dump to C: overwrites the very RAM regions you're trying to capture (page cache thrash, dirty writes).
  • Expected size = installed physical RAM + small overhead. A 16 GB host produces a ~16 GB raw dump. AFF4 will compress it to 4–8 GB.
  • Hash immediately. SHA-256 the dump on the responder USB before moving it. Repeat at the lab — values must match.
  • VMs are easier. A guest's .vmem file (plus the .vmsn snapshot for register state) is functionally equivalent to a live memory dump — and the host doesn't need a driver. Suspend the VM and copy.
  • BitLocker / full-disk encryption: if the box is running, the FVEK is in RAM. Capture now or lose the keys.
  • Anti-forensics aware. Some malware detects the WinPmem / DumpIt drivers loading. Mitigation: be quick, capture early, don't run interactive recon on the box first.

The hand-off: acquisition is half the job. Parsing is the other half — covered next:

  • Volatility 3 — symbol-driven plugin framework. The textbook parser. windows.pslist, windows.netscan, windows.malfind, windows.cmdline.
  • MemProcFS — mounts the dump as a virtual filesystem. F:\proc\by-name\lsass.exe\modules\ — browse memory like files. Brilliant for triage.
  • strings + grep — never underestimated. URLs, IPs, PowerShell blocks, credentials all surface this way.
⚠️Document the tool version. Volatility symbol tables are kernel-build-specific. Note the target's exact Windows build (winver) on the chain-of-custody form so the analyst can pick the right profile / ISF.
Cross-references for corroboration

Memory findings are most powerful when they line up with on-disk artefacts. The whole point of a multi-artefact methodology is that a clever attacker has to lie consistently in too many places at once.

  • Process list ↔ Application Execution (§10) — every PID in windows.pslist should have a matching entry in Prefetch, AmCache, or ShimCache. A process in RAM with no on-disk execution artefact = injected or process-hollowed.
  • Network sockets ↔ Browser history & DNS cache (§09) — remote IPs from windows.netscan should resolve to domains seen in browser history, %SystemRoot%\System32\drivers\etc\hosts, or the DNS resolver cache. Lone IPs with no DNS context are suspicious.
  • LSASS credentials ↔ SAM & SECURITY hives (§07) — NTLM hashes lifted from RAM must match the hashes recoverable from the offline SAM. Mismatch = injected creds or a different user context cached.
  • hiberfil & pagefile timestamps ↔ SYSTEM hive shutdown timestamp (§08) — file MFT timestamps on hiberfil.sys / pagefile.sys should line up with the last clean shutdown recorded in the SYSTEM hive (ShutdownTime). Drift implies tampering or a crash between sleep and shutdown.
  • Loaded modules ↔ Disk image (§01 E01) — every DLL path resolved in memory should exist on disk with a matching SHA-256. Memory-only or hash-mismatched modules = reflective loading.
  • Remember: hiberfil.sys and pagefile.sys live on the system volume — they are already inside the §01 E01. Extract them from the disk image before analysis; you do not need a second acquisition.
