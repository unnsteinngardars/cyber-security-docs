🪦What NTFS actually does on delete

"Delete" on NTFS is a bookkeeping change, not a data wipe. The bytes stay; the filesystem just stops pointing at them and marks the real estate as available.

  • $MFT record — the in-use flag (bit 0 of the flags field in the MFT record header) is cleared. The record sits in place, complete with filename, MACB, parent reference and (if resident) the file's own bytes, until the slot is reused for a new file.
  • Parent directory index — the filename entry is removed from the active B+ tree in $INDEX_ROOT / $INDEX_ALLOCATION. It may still linger in index entry slack inside the index node.
  • $Bitmap — the clusters the file occupied flip from 1 (allocated) to 0 (free). The cluster contents are not zeroed; they wait, intact, until the allocator hands them to a new file.
  • Resident files — anything small enough (≈ < 700 B) lives entirely inside the MFT record's $DATA attribute. There are no clusters to free; the data dies with the record slot.
  • $LogFile records the metadata transaction; $UsnJrnl:$J records the move into $Recycle.Bin and the eventual real delete (USN reasons 0x200 rename + 0x40000200 close-after-delete).
📦Where deleted data hides

If the OS only changed pointers, the data is still somewhere. Know every place a "deleted" file leaves residue:

  • MFT slot itself — name, MACB, parent ref, and (for resident files) the entire payload.
  • Unallocated clusters — the full file content, untouched until overwritten. Carver territory.
  • File slack — the unused bytes between end-of-file and end-of-cluster. Holds tails of older files.
  • Volume Shadow Copies — past versions from earlier snapshots. From the filesystem's view they are still allocated, so they survive deletion of the live file.
  • $Recycle.Bin\<SID>\ — $I (original path, size, delete time) and $R (the actual payload) pair. Not "deleted" yet — just moved.
  • pagefile.sys / hiberfil.sys — fragments of file contents that lived in RAM at delete time.
  • Caches — browser cache, thumbcache_*.db, Office MRU previews, app-specific caches keep thumbnails and snippets long after the original is gone.
🧾MFTECmd — start here, not with the carver

MFTECmd (Eric Zimmerman) parses $MFT directly. For every deleted record it gives you the full original filename, parent path, MACB timestamps, and — if the file was resident — the file content inline. Step one of every recovery; carving is a fallback when MFT can't help.

  • Filter InUse == False to get the deleted set.
  • IsDirectory, HasAds, FileSize and the parent path column are usually enough to triage.
  • Resident content is exposed in the --de dump or the J/Log outputs.
Parse $MFT to CSV
MFTECmd.exe -f F:\$MFT --csv C:\out --csvf mft.csv
Filter deleted entries (PowerShell)
Import-Csv C:\out\mft.csv | ? { $_.InUse -eq "False" } | Export-Csv deleted.csv -NoTypeInformation
Dump a specific MFT entry (resident bytes)
MFTECmd.exe -f F:\$MFT --de 0x12345 --csv C:\out
🗂Arsenal Image Mounter + KAPE --vss

Arsenal Image Mounter (AIM) mounts an E01/raw image as a real Windows volume — disk-level, not file-level — so that Volume Shadow Copy Service sees and exposes the snapshots inside it. Then KAPE with --vss walks every VSS store and re-collects targets from each.

  • Files deleted between snapshots reappear from earlier VSS copies.
  • Even overwritten files survive if a snapshot froze the old content.
  • Combine with KapeTriage or specific targets (RegistryHives, EventLogs) for a fast pass.
Mount image with AIM (GUI: write-temporary mode)
# File → Mount disk image → write-temporary → drive letter, e.g. F:
KAPE triage across all VSS
kape.exe --tsource F: --target KapeTriage --tdest C:\out --vss
Just the registry hives, from every snapshot
kape.exe --tsource F: --target RegistryHives --tdest C:\out --vss --vhdx case01
🪚PhotoRec — signature carving for unallocated

PhotoRec (CGSecurity) ignores the filesystem entirely and scans the raw byte stream for known magic bytes. It recovers content only — no filename, no path, no MACB.

  • Best aimed at unallocated space specifically (free option in the menu).
  • Output files are renamed f00012345.jpg — original names are gone with the MFT entry.
  • Limit signatures to what you care about; "everything" produces gigabytes of false positives.
  • Run on a copy of the image, not the master.
Interactive (preferred for teaching)
photorec /d C:\out\carved image.E01
Scripted — JPG + PDF + DOCX only
photorec /d C:\out /cmd "image.dd fileopt,everything,disable,jpg,enable,pdf,enable,zip,enable,search"
Free-space only (skip allocated)
# In menu: choose partition → [ Free ] instead of [ Whole ]
⚠️Carver reality check: PhotoRec on a heavily fragmented disk produces noisy, half-broken output — a JPG header followed by a chunk of a Word document is "recovered" as a corrupt JPG. And on a BitLocker/EFS volume that hasn't been decrypted yet, every byte looks random; you will burn CPU and find nothing. Decrypt first, then carve.
🧰Autopsy & the Hex Editor of last resort

Autopsy is the one-stop GUI: it parses $MFT and lists deleted entries, can ingest VSS, runs PhotoRec as an ingest module, indexes text, and pivots into the timeline. Best didactic tool — every recovery technique in one pane.

  • Deleted Files view = MFT records with InUse=False.
  • Carved Files view = PhotoRec ingest output.
  • Keyword search indexes unallocated too — useful for plaintext fragments.

Hex Editor (HxD, 010 Editor) is the last resort. You go in by hand to:

  • Verify a carved file's header/footer are sane.
  • Read inside file slack where automated tools refuse to look.
  • Hunt known plaintext (passphrases, URLs, ransom notes) across cluster boundaries.
  • Carve a fragment by hand when the signature is unusual or the footer is missing.
Autopsy ingest — typical config
# New case → add data source (E01) → enable: File Type ID, Hash Lookup, Keyword Search, PhotoRec Carver, VSS
010 Editor — run a template to validate a carved JPEG
# Templates → JPGTemplate.bt — flags broken SOI/EOI markers immediately
🧬How file carving actually works — and the magic-byte cheat sheet

A carver ignores the filesystem and scans the raw byte stream for file signatures (magic bytes) at the start of allocation units. When it finds a known header it searches forward for the matching footer (or applies a max-size heuristic) and writes the slice out as a recovered file. Carvers know hundreds of signatures; you only need to recognise the common ones to read what a tool produced.

💡Cluster alignment matters: most carvers only check for headers at the start of each cluster (4 KB on default NTFS), so a header lurking at an arbitrary byte offset inside slack will be missed unless you ask for byte-granular scanning.
⛔What carving cannot get back

Carving is powerful but blind. Set the examiner's expectations honestly:

  • Fragmented files — header found, but the next cluster belongs to another file. Carvers stitch contiguous runs and assume continuity; a fragmented DOCX comes out corrupt.
  • Resident files — too small to leave any cluster footprint. Lives in the MFT record only; PhotoRec will miss it. That's why MFTECmd is step 1.
  • Overwritten files — once another file's data is written to those freed clusters, the original is gone. No magic recovery.
  • Filenames & paths — carved files are renamed f00012345.ext. Without the MFT record, there is no original name.
  • Timestamps — there is no metadata in raw clusters. PhotoRec stamps a "carved at" time only.
  • Encrypted content — BitLocker / EFS clusters look like random bytes; no signature matches. Carving must happen after decryption.
  • Compressed-by-NTFS files — even contiguous, the on-disk bytes don't match the raw magic until decompressed.
🪜Workflow priority — the order to actually run things

Run cheap, metadata-rich techniques first. Carve raw clusters only when those are exhausted.

  1. MFTECmd on $MFT — recover deleted records: original filename, parent path, MACB, and resident content inline.
  2. AIM + KAPE --vss — mount the image, walk every Volume Shadow Copy, re-collect targets. Anything that existed at a snapshot is back.
  3. $Recycle.Bin $I / $R pairs — soft-deleted files with full original path, size and delete timestamp. Map the <SID> to a user (see §07).
  4. PhotoRec / Autopsy carving of unallocated — recover content for anything the MFT no longer remembers.
  5. Hex editor on slack, pagefile, hiberfil — last sweep for fragments, plaintext, and string remnants.
✅Examiner rule: never report a carved blob as "the user's file" without corroborating its existence in the MFT, VSS, Recycle.Bin, Prefetch (§10), or another metadata source. Carving alone gives content, not provenance.
