📐What an acquisition artefact actually contains

An acquisition artefact is a bit-for-bit copy of a storage medium, wrapped in a container that records how, when and by whom it was produced. Two halves matter equally:

  • The payload — every sector of the source, including unallocated space, slack, swap, hibernation, alternate data streams, free-but-not-zeroed blocks. Everything later sections (§04 NTFS, §05 carving, §02 memory) parse is buried in here.
  • The provenance — examiner name, case ID, acquisition timestamps, source serial number, MD5/SHA-1/SHA-256 of the source and of the image. This is what makes the payload defensible; without it, the bits are just bits.

The forensic value of the image is therefore not just "what files does it contain" but "can I prove these files are exactly what was on the disk at acquisition time". Pick the container that preserves both.

💾E01 / Ex01 — the EnCase Evidence File
Integrity Provenance Recovery

Where it lives: wherever the acquirer wrote it. Multi-segment files: case01.E01, .E02, … (or .Ex01 for the AD1/L01 successor).

What it carries:

  • Header — case number, evidence number, examiner, acquisition / system date, OS, tool version, drive serial, optional notes.
  • Bitstream — chunked into 32 KB blocks, optionally compressed, each block CRC-protected so localised corruption stays localised.
  • Embedded MD5 + SHA-1 — computed across the whole source during acquisition; recomputed and re-verified on every read.
  • Segment footers — link the multi-file set together; missing a segment is detectable.

Forensic value: the file self-verifies. Any read that flips a bit raises a CRC error before the parser sees bad data. The header tells you who acquired what, when — without trusting any external paperwork.

What it cannot tell you: what was on the disk before acquisition started. RAM at acquisition time. Anything overwritten between the incident and the image being taken.

🧱raw / dd — the flat sector copy
Recovery Portability

Where it lives: a single .raw / .dd / .img file the size of the source, or split into .000, .001, … chunks.

What it carries: exactly the source bitstream, with no headers, no metadata, no embedded hashes. Sector 0 of the file is sector 0 of the disk.

Forensic value:

  • Trivially mountable — any loopback driver, losetup, xmount, AIM, FTK Imager — and any carving / parsing tool reads it without conversion.
  • Acts as the lowest-common-denominator format when a third tool only speaks "raw".
  • Easy to slice (e.g., extract a single partition with dd skip=… count=…) because the byte offsets are the disk's byte offsets.

What it cannot tell you: nothing about its own provenance. You must keep an external hash + CoC log alongside it — drop those and the image becomes evidence of nothing.

📦AFF4 — the multi-stream evidence bundle
Integrity Recovery Bundling

Where it lives: a single .aff4 ZIP-based container (often produced by WinPmem in IR).

What it carries: multiple independently-addressable streams in one container — typically PhysicalMemory, pagefile.sys, hiberfil.sys, the disk image, the binary that did the capture, and a Turtle/RDF manifest describing each stream (size, hash, acquisition tool, timestamp, host name).

Forensic value:

  • One file = one piece of evidence to track — no risk of orphaning a pagefile sidecar.
  • Per-chunk SHA-256 catches storage bit-rot far earlier than an end-of-file hash would.
  • Streams are listable (--list) and exportable individually — feed memory to Volatility, hiberfil to hibr2bin — without re-acquiring.

What it cannot tell you: not all forensic suites speak AFF4. For analysis you often export a raw stream and work on that — keep the AFF4 as the master, parse the export.

🗄VHD / VHDX — the Microsoft container
Mountability

Where it lives: .vhd / .vhdx file. Native to Windows / Hyper-V.

What it carries: a sparse or fixed-size disk image with a small footer (dynamic VHDs) or header (VHDX). VHDX additionally carries a metadata region and resilient log for crash-safe writes.

Forensic value: Windows mounts it as a real volume with a double-click — handy for quick review on a sandbox analysis VM. Also the format KAPE can target directly (no AIM required).

What it cannot tell you: by default it carries no examiner metadata or self-verifying hashes — treat it like raw plus a thin wrapper. Always pair with an external hash + CoC.

🔐Hash sidecar files — the independent witness
Integrity

Where it lives: a tiny text file next to the image — case01.E01.md5, case01.E01.sha256, or a combined HASHES.txt in the case folder.

What it carries: the hash of the source disk (computed during read) and the hash of the image (computed after write). For E01/AFF4 these duplicate the embedded values; for raw they are the only record.

Forensic value:

  • Source-hash ≡ image-hash ≡ working-copy-hash — three independent equal values is the gold standard the court wants.
  • Detects silent corruption: an old E01 returning the same embedded hash but a different recomputed hash on read = storage bit-rot.
  • Cheap to recompute and re-sign at any later stage of the case.

What it cannot tell you: nothing about what changed if the hash is mismatched — only that something did. Investigate, do not erase.

📋Chain-of-Custody & acquisition log
Provenance Admissibility

Where it lives: paper or PDF in the case folder; sometimes the acquisition tool's own .txt log (FTK Imager writes one automatically).

What it carries:

  • Source identifiers — make, model, serial, capacity, photographs.
  • Acquisition environment — write-blocker model + firmware, host workstation, tool version.
  • Timestamps for every transfer of custody, every analysis session, every working copy.
  • Examiner signatures at each handoff.

Forensic value: ties the bits in the image to the physical disk seized at the scene. Without it, the defence can argue the image came from somewhere else entirely.

What it cannot tell you: nothing about the contents. A flawless CoC over a contaminated image proves only that you carefully tracked contaminated evidence.

Cross-references for corroboration

The acquisition artefact itself is what you defend in court; here is how its fields tie to evidence later in the case:

  • Hash chain — source hash on CoC log ≡ embedded hash in E01/AFF4 ≡ fresh hash of the working copy. Three independent, identical values.
  • E01 acquisition timestamp ↔ Memory dump timestamp (§02) — RAM should be earlier; if it isn't, the order-of-volatility was violated.
  • E01 acquisition timestamp ↔ SYSTEM hive ShutdownTime (§08) — once parsed, the last shutdown recorded inside the image should match the time the responder pulled the plug.
  • Source serial in CoC ↔ $Volume serial in the image (§04) — both anchor the image to the physical device.
  • Tool log version ↔ MFTECmd / EvtxECmd output version — reproducibility: another examiner re-running the same versions must get byte-identical parses.
