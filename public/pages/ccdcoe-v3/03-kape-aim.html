📐Why this section is artefact-driven, not tool-driven

KAPE is not the evidence — it is a copier. Its value is the curated set of paths in KapeTriage, which encodes the community's best answer to "what should I look at first on a Windows host". This page treats those targets as the unit of analysis: each artefact family, what forensic question it answers, what it can't.

Arsenal Image Mounter sits underneath — it exposes the E01 as a real volume so KAPE's RawCopy can read locked system files. The Read Only vs Write Temporarily distinction matters because a dirty NTFS journal will refuse to mount cleanly without a place to replay; the temporary overlay gives the kernel that place without ever touching the master image.

ℹ️Two-phase rule: a Target produces files; a Module parses them. Keep the unparsed Target output forever — re-parsing is cheap, re-collecting from a now-overwritten host is impossible.
🗃$MFT — the filesystem index
Timeline Deleted files Anchor

Where: \$MFT at the root of every NTFS volume.

Answers: every file (and deleted file) that ever existed on the volume, with two sets of MACB timestamps (§04). The single most valuable filesystem artefact.

Cannot tell you: file contents for non-resident files whose clusters have been reused; only the metadata.

📜$LogFile + $UsnJrnl — recent operations
Recent ops Names of wiped files

Where: \$LogFile, \$Extend\$UsnJrnl:$J.

Answers: a rolling transaction log of metadata changes (creates, renames, deletes, attribute writes). The USN journal records filename changes by reason code — often the only place the name of a wiped file survives.

Cannot tell you: events older than the journal's rotation horizon (minutes-to-days on a busy host).

🗝Registry hives — system & per-user state
User attribution Persistence OS profile

Where: C:\Windows\System32\config\ (SAM, SYSTEM, SOFTWARE, SECURITY) + C:\Users\<u>\NTUSER.DAT + …\AppData\Local\Microsoft\Windows\UsrClass.dat.

Answers: accounts (§07), services + devices + timezone (§08-SYSTEM), installed apps + network history (§08-SOFTWARE), per-user MRUs + execution traces + ShellBags (§09 / §13).

Cannot tell you: in-memory writes that never flushed — combine with hive transaction logs (.LOG1, .LOG2) for the freshest state.

📜EventLogs (.evtx) — the auditor's diary
Timeline Logons Process audit

Where: C:\Windows\System32\winevt\Logs\*.evtx.

Answers: logon / logoff, RDP, account changes, service installs, scheduled-task runs, PowerShell script blocks, Sysmon events. The narrative spine of a Windows investigation.

Cannot tell you: what the auditor wasn't configured to capture. Default auditing on a workstation is sparse; absence of an event is rarely proof of absence of activity.

🏃Prefetch (.pf) — execution proof
Execution Last 8 runs

Where: C:\Windows\Prefetch\APPNAME-<hash>.pf.

Answers: that this executable ran, how many times, when (last eight run times in modern Win10/11), what DLLs / files it loaded during startup, and a hash of its full path.

Cannot tell you: who ran it (no SID), command-line arguments, child processes. Disabled on Server SKUs and on some SSD-tuned systems.

🗂AmCache — the executable inventory
Inventory SHA-1

Where: C:\Windows\AppCompat\Programs\Amcache.hve (a registry hive).

Answers: every PE the Application Experience service saw — with SHA-1 hash, file size, signing info, first-seen time. Catches executables that never ran (Prefetch missed them).

Cannot tell you: count of executions or per-run timing. Pair with Prefetch for "did it run", AmCache for "did it exist".

⏰Scheduled Tasks — persistence #1
Persistence Re-execution

Where: C:\Windows\System32\Tasks\ (XML) + SOFTWARE\Microsoft\Windows NT\CurrentVersion\Schedule\TaskCache.

Answers: tasks defined on the host — name, principal SID, trigger, action (binary + arguments), author. XML and registry corroborate each other — divergence is a red flag.

Cannot tell you: whether the task actually fired — pair with Microsoft-Windows-TaskScheduler/Operational.evtx.

🚀StartupInfo — boot-time process snapshot
Boot processes Persistence

Where: C:\Windows\System32\WDI\LogFiles\StartupInfo\<SID>_StartupInfo*.xml.

Answers: per-boot, per-user snapshot of every process started shortly after logon, with full command lines and parent PIDs. ~5 boots retained.

Cannot tell you: processes started after the snapshot window (~30 s) or on systems where WDI is disabled.

🔗LNK files — what the user opened
User action External media

Where: C:\Users\<u>\AppData\Roaming\Microsoft\Windows\Recent\*.lnk.

Answers: files the user double-clicked, including from now-disconnected USB / network shares — preserves target path, target size, volume serial, MAC address of the host that wrote the link (TrackerDataBlock).

Cannot tell you: programmatic opens that bypassed Explorer.

📋JumpLists — per-app MRU
Per-app history DestList timestamps

Where: …\Recent\AutomaticDestinations\*.automaticDestinations-ms + …\CustomDestinations\.

Answers: per-application recent/pinned items keyed by AppID — embedded LNK streams + DestList with timestamps. Reveals which app opened which file, in order.

Cannot tell you: activity in apps that opt out of jump-list reporting.

📁ShellBags — every folder ever browsed
Folder browsing Deleted paths External media

Where: inside NTUSER.DAT and UsrClass.dat — Software\Microsoft\Windows\Shell\BagMRU + Bags.

Answers: every folder that user ever browsed in Explorer — including on removable media, network shares, and folders later deleted. View settings, last-visited timestamps, and shell-item metadata are preserved per folder.

Cannot tell you: what was inside those folders — only that the user navigated there.

🌐Chrome history
Browsing Visit times

Where: C:\Users\<u>\AppData\Local\Google\Chrome\User Data\Default\History (SQLite).

Answers: URLs, page titles, visit counts, visit timestamps, transition types, downloads (file path + URL + bytes + state) — all per-profile.

Cannot tell you: InPrivate / Incognito sessions (no rows written). Sync-disabled accounts only — synced data lives server-side.

🦊Firefox history
Browsing Visit times

Where: …\AppData\Roaming\Mozilla\Firefox\Profiles\<prof>\places.sqlite.

Answers: same questions as Chrome — different schema (moz_places + moz_historyvisits) + bookmarks + downloads.

Cannot tell you: Private Browsing sessions. Profile folder name randomised — enumerate all profiles.

💤hiberfil.sys + pagefile.sys (collected by KAPE for §02)
Past process tree Strings

KAPE includes these in the triage set so memory analysis can begin without a live capture. See §02 for forensic value — here we just note they are part of the same collection sweep so the dump folder is self-sufficient.

🚧AIM mount modes — protecting the artefact source

KAPE's collection is only as defensible as the volume it reads from. AIM controls that volume:

  • Read Only — the kernel sees a read-only disk. Hash of the E01 is preserved. Works on a clean NTFS volume; fails on a dirty journal because the kernel can't replay it.
  • Write Temporary — Windows believes it can write; AIM redirects every write to a sparse overlay file on your analysis disk. Journal replay happens against the overlay, the source E01 stays untouched, the volume mounts cleanly.
  • Target / Module — KAPE's two-phase model. Target = collect (paths + RawCopy). Module = parse (Eric Zimmerman / Plaso / your own).
🛑Never --tdest back onto the system drive you're collecting from. KAPE's own writes pollute the very artefacts it is reading. Use an external / network volume.
Cross-references for corroboration

Single-artefact findings are weak; KAPE's value is having every nearby corroborator in one folder:

  • Prefetch ⋈ AmCache ⋈ Security 4688 — three independent witnesses to a single execution.
  • LNK volume-serial ⋈ SYSTEM USBSTOR ⋈ Event 1006 (Partition/Diagnostic) — anchors a file-open event to a specific physical USB device (§14).
  • ShellBags ⋈ JumpLists ⋈ MFT — folder browsed, file opened, file metadata — three angles on the same user action.
  • Scheduled Task XML ⋈ TaskScheduler.evtx ⋈ Prefetch — definition, firing, execution.
  • $UsnJrnl ⋈ Recycle Bin $I/$R (§18) ⋈ VSS snapshots — name-of-deleted-file, deletion-time, recovered-content.
