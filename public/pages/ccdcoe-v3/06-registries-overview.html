🌲Why the registry is so rich

The registry is a single hierarchical database — a tree of keys (folder-like nodes) holding values (name + type + data) — but it is fragmented across a handful of on-disk regf hives plus per-user files. That fragmentation is its strength: each hive is a specialised log of one slice of system state, written with its own cadence, flushed with its own transaction logs, and answering its own forensic questions. The §06–§13 chapters all ride on the substrate established here.

The hierarchy: hive file (a regf container on disk) → hbins (4 KB allocation blocks inside the file) → cells (typed records inside hbins: nk, vk, sk, …) → keys (an nk plus its subkey list and value list) → values (a vk plus its data, resident or pointed-at).

🌳Keys, values, and the virtual roots
LastWriteTime Hierarchy

Structure: keys contain values and subkeys; values carry a name, a type, and a data blob. Every key has exactly one 64-bit LastWriteTime FILETIME; individual values have no timestamp of their own.

The five logical roots:

  • HKLM — machine-wide config (hardware, OS, installed software, services). Backed by SAM / SYSTEM / SOFTWARE / SECURITY / DEFAULT.
  • HKU — one subtree per loaded user profile, keyed by SID. Each is a mounted NTUSER.DAT.
  • HKCU / HKCR / HKCC — virtual views. HKCU is a pointer at HKU\<current-SID>; HKCR is a merge of HKLM\Software\Classes + HKU\<SID>_Classes; HKCC mirrors a SYSTEM subkey. Nothing lives on disk under these names — do not look for an HKCU hive file.

Value types you will encounter: REG_SZ, REG_EXPAND_SZ, REG_MULTI_SZ, REG_DWORD, REG_QWORD, REG_BINARY, REG_NONE.

Cannot tell you: which value inside a key was last touched — LastWriteTime updates whenever any value in the key changes. Use the surrounding key context (and corroborating artefacts) to scope what actually fired.

🧱regf — what a hive file actually is
On-disk format Free-list cells

Each hive is a self-contained regf container with a fixed layout. Knowing this matters when the parser chokes or you need to carve.

Structure:

  • Header (4 KB) — regf magic, primary & secondary sequence numbers, root cell offset, last-written timestamp, checksum.
  • hbins — 4 KB allocation blocks, each prefixed with hbin magic.
  • Cells inside hbins, signed length prefix (negative = allocated, positive = free):
  • nk — key node; carries LastWriteTime, parent/child pointers, security ref.
  • vk — value record (name, type, data offset).
  • sk — security descriptor, shared between keys.
  • lh / lf / li / ri — subkey-list index nodes (hash / fast / plain / root-index).
  • db — big-data, for values larger than ~16 KB.

Answers: free / unallocated cells are the source of deleted-key recovery — they sit in the hbin's free list until the allocator hands their bytes to a new write.

📝Transaction logs & dirty hives
Dirty bit Unflushed ops

Where: every hive ships with sibling transaction logs in the same folder: e.g. SYSTEM, SYSTEM.LOG1, SYSTEM.LOG2 (and historically .LOG). Windows writes pending changes to the logs first, then folds them into the hive at flush time.

Structure: a dirty hive is one whose in-memory transactions were not yet flushed to the primary file — typical when you image a running box, when the OS crashed, or when someone pulled the plug. Reading the primary file alone gives you a stale snapshot.

Answers:

  • Registry Explorer / RECmd auto-detect and replay .LOG1 / .LOG2 on load.
  • The hive header's primary & secondary sequence numbers being unequal is the dirty-bit signal.
  • Always export the hive and both logs from KAPE / FTK Imager — they are useless on their own but mandatory when combined.

Cannot tell you: ops older than the logs' rotation window (logs are small ring buffers — minutes of history on a busy hive).

⚠Stale-hive trap: dropping a dirty NTUSER.DAT into a parser without its .LOG1 / .LOG2 can cost you the last several hours of user activity — exactly the window an examiner usually cares about. Always grab the logs.
🗑Deleted keys, unallocated cells, slack
Free-list recovery Stale timestamps

Structure: when a key or value is deleted, its cell length is flipped positive and the cell is linked into the hbin's free list. The cell contents typically remain intact until the allocator hands the space to a new write — directly analogous to NTFS file deletion (§04).

Answers:

  • Registry Explorer surfaces orphaned cells in the Deleted Keys / Deleted Values view, rebuilding parent chains from nk back-pointers where possible.
  • RECmd exposes the same via --Recover True.
  • Recovered names & data are usually verbatim.
  • Slack at the end of hbins occasionally yields fragments of older value data — useful for keyword carving, not for whole-key reconstruction.

Cannot tell you: when the key was deleted. The LastWriteTime on a recovered cell is the parent key's stale value, not the moment of deletion — treat recovered timestamps as untrustworthy and corroborate through journals / event logs.

🔤MRUList & MRUListEx — ordering
Order anchor Timeline pivot

Many Explorer / Office / dialog artefacts store an ordered "Most Recently Used" list as a bag of letter-named values plus one special ordering value:

  • Values a, b, c, … each carry one entry (a path, a typed string, a binary PIDL).
  • MRUList (legacy, REG_SZ) — a string like cab meaning most recent first: c then a then b.
  • MRUListEx (Vista+, REG_BINARY) — sequence of little-endian DWORD indexes, terminated by FFFFFFFF, most recent first.

Without the ordering value the rest is just an unordered bag. The MRUList / MRUListEx is what turns a list of files into a timeline of usage. Registry Explorer plugins decode MRUListEx automatically; raw reg query does not.

🛠The toolchain — Registry Explorer, RECmd, RegRipper
Plugin-driven Batch sweeps

Registry Explorer (RegistryExplorer.exe, Eric Zimmerman) — GUI. Drag in a hive; it auto-locates .LOG1 / .LOG2 siblings and replays them. Plugins normalise common artefacts (UserAssist ROT13-decoded, ShellBags parsed, TypedURLs sorted) into a flat table view. Use Find for value-name keyword sweeps, Bookmarks for case notes, View → Deleted Keys / Deleted Values for unallocated cells. Best for interactive exploration.

RECmd (RECmd.exe, Eric Zimmerman) — the CLI sibling. Batch-driven via .reb files in RECmd\BatchExamples\. Kroll_Batch.reb is the canonical do-everything sweep; SYSTEM_Batch.reb / SAM_Batch.reb / NTUSER_Batch.reb are hive-targeted. Drops cleanly into KAPE (§03) as the RECmd_* module set. --Recover True turns on deleted-key/value emission.

RegRipper (rip.exe / rr.exe, Harlan Carvey) — older Perl plugin runner. One plugin per artefact: samparse, userassist, services, appcompatcache, compname, shellbags, … Slower, but invaluable as a second opinion and for niche plugins the EZ tools don't ship.

What a plugin actually is: a normalised viewer — it reads the raw key/value bytes and renders them as a table with derived columns (ROT13-decoded names, FILETIME-decoded run counts, parsed ShellBag PIDLs). Without the plugin you stare at a binary blob; with it, you read a row.

Cross-references for corroboration

The registry never stands alone — its findings must be anchored against the filesystem and against the rest of the hive ecosystem. The §06 substrate fans out into nearly every later chapter:

  • NTFS $STANDARD_INFORMATION (§04) — the hive file's MFT $SI timestamps should bracket the hive header's last-flush timestamp. A wildly newer $SI.Modified than the internal flush time means somebody touched the hive offline.
  • User folder ↔ SID (§07 SAM) — match C:\Users\<user> to the SID owning the NTUSER.DAT via SAM's F/V values or via SOFTWARE\…\ProfileList.
  • Deleted-key recovery (here) ↔ $MFT deleted-file recovery (§04 / §05) — same allocator-reuse logic, same "name & data survive, timestamps don't" caveat. The regf free list is to hives what the $MFT free slot is to files.
  • Application Execution (§10) — BAM lives in SYSTEM, UserAssist & RunMRU in NTUSER.DAT; cross-check a single execution event across both for confidence.
  • Cloud storage (§11) & ShellBags (§13) — both are pure registry artefacts (OneDrive / Dropbox keys in SOFTWARE + NTUSER; ShellBags in UsrClass.dat) — they ride on the same hive-loading workflow taught here.
  • Memory image (§02) — Volatility's registry plugins can pull hives out of RAM and may surface keys that were written but never flushed to disk — the only way to see truly transient registry state.
  • Hive file $SI.Modified vs key LastWriteTime — registry-key write time should be ≤ hive file modify time; mismatch suggests on-disk hive tampering or out-of-band hive editing.
