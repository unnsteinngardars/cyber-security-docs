🪪Why pair SYSTEM with SOFTWARE

Together these two machine-scope hives produce the box's passport. SYSTEM describes the hardware and kernel-level configuration the OS was booted with — drivers, services, mounted volumes, NIC IP plumbing, USB enumeration history, the wall-clock offset. SOFTWARE describes the OS build itself and everything installed on top of it — product name, edition, build, monthly patch level, registered owner, every application, every Wi-Fi network ever joined, machine-wide autoruns, app permission grants, AV exclusions.

Read in tandem, in five minutes you can answer: what is this box, who owns it, where does it live, what does it run, what does it talk to, and what does it persist across reboots. Every other artefact in the case is interpreted through this profile — timestamps without TimeZoneInformation are lies, malware paths without UBR cannot be mapped to CVEs, a downloaded LNK without USBSTOR loses half its provenance.

Where: on a live host both hives live at %SystemRoot%\System32\config\SYSTEM and ...\SOFTWARE and are locked by the kernel. Capture them with FTK Imager → Obtain Protected Files (§01), with KAPE's RegistryHives target, or from a triage image directly.

🔀ControlSets — why CurrentControlSet isn't on disk
Boot anchor Diff baseline

Where: the SYSTEM hive holds multiple parallel kernel configurations called ControlSets — usually ControlSet001 and ControlSet002. A pointer key SYSTEM\Select holds four DWORDs:

  • Current — which ControlSet the running OS booted with.
  • Default — which one would be used on next normal boot.
  • Failed — last ControlSet that failed to boot.
  • LastKnownGood — last ControlSet that booted through to a successful logon.

Structure: SYSTEM\CurrentControlSet is a virtual mount the kernel builds at boot from whichever ControlSet Select\Current points at. It does not exist on disk. When you open the hive offline you walk ControlSet001 directly — or follow Select\Current if you want the boot the user actually had.

Answers: the other ControlSet is the previous boot's configuration. When malware altered a service or driver during the most recent boot, diffing 001 against 002 surfaces exactly what changed — a free historical snapshot baked into the hive.

Cannot tell you: anything before the previous boot. Older ControlSets are not retained; only the current and the LastKnownGood survive.

⛔Offline gotcha: in Registry Explorer with a dead-box SYSTEM hive there is no CurrentControlSet node. Read Select\Current first, then walk that ControlSet — don't assume 001.
🪟SOFTWARE — the OS passport key
InstallDate UBR ↔ CVE

Where: SOFTWARE\Microsoft\Windows NT\CurrentVersion.

Structure: a flat key of named values that together identify the operating system to byte-level patch precision.

Answers:

  • ProductName, EditionID — "Windows 10 Pro" vs "Enterprise" decides which group policies and features are available.
  • ReleaseId / DisplayVersion + CurrentBuild — feature update level ("21H2", build 19044).
  • UBR — Update Build Revision. The monthly patch number. Build alone says "Windows 10 21H2"; only build.UBR tells you which Patch Tuesday this host has received.
  • InstallDate (UNIX seconds) + InstallTime (FILETIME) — when this image was first installed; anchors host age.
  • RegisteredOwner / RegisteredOrganization — who the box was provisioned for. Useful in stolen-laptop and BYOD cases.

Cannot tell you: what was installed after the OS, only that the OS itself is at this patch level. Application patching lives elsewhere (Uninstall, AppPaths, vendor keys).

ℹ️UBR matters: CVE feeds key on build.UBR, not just build. "19044.4170" pins the host to one specific monthly cumulative — decides which exploits were viable on the date of compromise.

🪤Services — persistence mechanism #1
Auto-start Persistence ControlSet diff

Where: SYSTEM\ControlSet001\Services\<ServiceName> — one subkey per service or driver.

Structure: each subkey holds ImagePath (the binary), ServiceDll (for svchost-hosted services, the actual code), Start (boot/system/auto/manual/disabled), Type (kernel driver, file-system driver, own-process, shared-process), plus Description, ObjectName (the account it runs as).

Answers:

  • Every service and driver registered on the box — far broader than "what's running now".
  • Auto-start services (Start=2) — the persistence shortlist.
  • Kernel drivers loaded at boot (Start=0, Type=0x1) — rootkit territory.
  • What account each service runs under — LocalSystem services with user-writable ImagePath are escalation hooks.

Cannot tell you: whether the service actually ran at the last boot — that's System.evtx 7045 / 7036 (§14) and Prefetch (§10) territory.

🔌USB chain — three views, one device
First/last insert Volume serial User binding

USB attribution is a three-key triangle. No single key tells the whole story — each holds a different layer.

  • SYSTEM Enum\USB / Enum\USBSTOR — kernel enumeration. Vendor / product / serial of every device the bus ever saw. First-seen and last-connect timestamps in the subkey metadata.
  • SYSTEM MountedDevices — the volume binding. \DosDevices\E: values contain a blob whose first byte distinguishes drive-letter mappings (0x44) from volume mount points (0x49); the tail holds the volume GUID or USB device signature that ties E: back to a specific USBSTOR serial.
  • SOFTWARE Microsoft\Windows Portable Devices\Devices — the friendly view. Subkey name encodes the device path; FriendlyName is the label the user actually saw in Explorer.

Per-user binding (which logon saw the device first) lives in NTUSER (§09) under ...\MountPoints2. Add the System.evtx 1006 / 2003 driver-install events (§14) and you have a full chain of custody for any USB device.

🚀Autoruns — Run / RunOnce / StartupApproved
Logon trigger Persistence

Where: SOFTWARE\Microsoft\Windows\CurrentVersion\Run, ...\RunOnce, and ...\Explorer\StartupApproved\Run. The first two are machine-wide; per-user equivalents live in NTUSER (§09).

Structure: Run and RunOnce are flat — value name is the entry label, value data is the command line. StartupApproved\Run stores binary state blobs: byte 0 = enabled (02) or disabled (03), bytes 4–11 = FILETIME of the last toggle.

Answers:

  • Run — executes at every interactive logon. The textbook persistence key.
  • RunOnce — executes once at next boot, then the value is deleted. Loved by installer cleanup and by dropper stage-2 payloads that want one shot at SYSTEM context.
  • StartupApproved\Run — what the user enabled or disabled in Task Manager → Startup. Tells you what was actually live versus what was merely registered.

Cannot tell you: whether the entry actually fired without errors — confirm with Prefetch (§10), AmCache, or Security.evtx 4688.

🌐NetworkList — the travel diary
DateCreated / DateLastConnected BSSID geolocation

Where: SOFTWARE\Microsoft\Windows NT\CurrentVersion\NetworkList\Profiles\<GUID> for each network the OS has ever connected to; Signatures\Unmanaged\<sig> and Signatures\Managed\<sig> for the underlying network fingerprints.

Structure: DateCreated and DateLastConnected are 16-byte SYSTEMTIME blobs (year/month/dow/day/hour/min/sec/ms WORDs). Without an RE Plugin they look like hex noise; the plugin renders them as real timestamps.

Answers:

  • Every Wi-Fi SSID and wired profile the box has ever joined — first-seen, last-seen, network category (public/private/domain).
  • The default-gateway MAC stored under Signatures\Unmanaged is the BSSID of the AP — geolocate against WiGLE for a physical-location pin.

Cannot tell you: the wireless PSK (lives in WLAN profile XML under %ProgramData%\Microsoft\Wlansvc\Profiles\Interfaces, not in the hive).

🧩Registry Explorer plugins — raw blobs to tables
Parser layer

Where: community-maintained .dlls in Registry Explorer\Plugins\. Each implements IRegistryPluginBase for a specific subkey path and reformats its raw values into a readable grid.

Answers:

  • Click an instrumented subkey (NetworkList\Profiles, USBSTOR, Services, Uninstall) — Registry Explorer renders the parsed table in the lower pane instead of hex blobs.
  • Each plugin grid is CSV-exportable straight into case notes.
  • Without plugins: NetworkList DateLastConnected is a SYSTEMTIME byte blob. With the plugin: a real timestamp column.
  • Keep the Plugins\ folder current — Eric Zimmerman ships new ones with each release.

RECmd is the same idea automated and batched: a .reb batch file lists "plugin X against key Y" lines and emits one CSV per plugin across the whole hive. Batch for breadth, GUI for depth.

🎯OS profiling — 5-minute workflow
Time-anchor first One-page passport

Goal: from a freshly mounted image, produce a one-page profile of the machine before opening any other artefact.

  • Hostname → SYSTEM ControlSet001\Control\ComputerName\ComputerName
  • Timezone → SYSTEM ...\Control\TimeZoneInformation (do this first)
  • OS + build + UBR + install date → SOFTWARE ...\Windows NT\CurrentVersion
  • Last clean shutdown → SYSTEM ...\Control\Windows\ShutdownTime
  • IP config / DHCP lease → SYSTEM Services\Tcpip\Parameters\Interfaces\<GUID>
  • Networks joined → SOFTWARE NetworkList\Profiles + Signatures\Unmanaged
  • USB history → SYSTEM Enum\USBSTOR + MountedDevices
  • Installed apps → SOFTWARE ...\Uninstall (+ Wow6432Node)
  • Autoruns → SOFTWARE ...\Run, RunOnce, StartupApproved\Run + SYSTEM Services
  • Defender posture → SOFTWARE Windows Defender\Exclusions

Either click through with RE + plugins or batch the lot with RECmd — both produce the same answers. The CSV output is the profile sheet.

Cross-references for corroboration

SYSTEM and SOFTWARE answer "what is this box". Almost every other artefact in the case is interpreted through them — these are the substrate, not the evidence:

  • TimeZoneInformation (SYSTEM) — required to convert every other timestamp in the case to wall-clock. Resolve before opening Event Logs, Prefetch, LNK files, or anything else.
  • ShutdownTime (SYSTEM) ↔ §01 acquisition log ↔ System.evtx 1074 / 6006 / 6008 (§14) — last clean shutdown should align with when the examiner pulled the plug; mismatch is a power-loss or tampering signal.
  • USBSTOR (SYSTEM) ↔ MountedDevices (SYSTEM) ↔ Portable Devices (SOFTWARE) ↔ MountPoints2 (NTUSER, §09) ↔ System.evtx 1006/2003 (§14) — same USB device, five views: kernel enumeration, volume binding, friendly name, per-user first-mount, driver-install event.
  • Services (SYSTEM) ↔ Run / RunOnce / StartupApproved (SOFTWARE) ↔ Scheduled Tasks (§03 / §10) ↔ NTUSER per-user Run (§09) — the classic persistence triad expanded to four. Anomalies in any of them are the hook.
  • Tcpip\Interfaces (SYSTEM) ↔ NetworkList Profiles + Unmanaged Signatures (SOFTWARE) — DHCP lease timestamps plus SSID plus AP MAC pin the machine to a place and a moment.
  • Uninstall (SOFTWARE) ↔ AmCache / Prefetch / ShimCache (§10) — "installed" and "actually executed" are different questions; this pair lets you ask both.
  • Defender Exclusions (SOFTWARE) ↔ any suspect paths surfaced in §10 — an exclusion added shortly before a malware first ran is a powerful narrative beat.
  • OS build + UBR (SOFTWARE) ↔ CVE / advisory feeds — patch level decides which exploits were viable on the date of compromise.
  • $Volume serial (§04) ↔ MountedDevices (SYSTEM) ↔ LNK TrackerDataBlock (§12) — the VSN is the join key between a file event and the physical volume it lived on.
  • Hive file $SI.Modified (§04) vs any SYSTEM/SOFTWARE key LastWriteTime — key writes must be ≤ hive file modify time; mismatch suggests offline hive tampering.
