⚡Why "did it run?" is a stack of overlapping witnesses

Windows does not maintain a single "executed-binaries" log. Instead, every subsystem that cares about a process — the cache manager that speeds up next launch, the compatibility shim engine that decides whether to apply a fix, Explorer that ranks Start-menu suggestions, the power manager that budgets background activity — leaves its own trail. Each trail answers a slightly different question, and each survives a different set of clean-up actions taken by the attacker.

The forensic discipline is to triangulate: a single artefact rarely proves execution by itself, but two artefacts pointing at the same binary at compatible timestamps are very hard to forge. The artefacts split along two axes — system-scope vs user-scope (does it record the SID of the launcher?) and survives-deletion vs needs-the-binary (does the evidence persist after the EXE is wiped?). Map every finding onto those axes before drawing a conclusion.

🚀Prefetch — the cache that proves execution
Proves execution Last 8 runs Referenced files
C:\Windows\Prefetch\<EXENAME>-<PATH-HASH>.pf

Where: the cache manager creates a .pf file ~10 seconds after a binary first runs and updates it on every subsequent launch. Filename is EXENAME-XXXXXXXX.pf where the 8-hex-digit hash is derived from the full path passed to CreateProcess — two different paths to the same EXE produce two different .pf files.

Structure: compressed (Win10+ uses MAM/Xpress-Huffman). Once decompressed contains run count, up to 8 last-run timestamps (Win8+; Win7 stored only the most recent), a list of every file/directory loaded during the first 10 seconds of execution, and the volume serial + create date of every volume those files lived on.

Answers:

  • Whether a binary actually ran — Prefetch is created by the cache manager only after a successful process launch.
  • When it ran (last 8 launches, individually).
  • How many times in total since the .pf was created.
  • What it touched in its first 10 seconds — DLLs, configs, payload files. Often reveals the second-stage filename.
  • From which volume (VSN cross-references §04 NTFS $Volume and §12 LNK).

Cannot tell you: who ran it (no SID); arguments; runs older than the 8 slots; anything if the host is an SSD on a server SKU or if EnablePrefetcher was disabled. PECmd parses the format.

⚠Server SKUs disable Prefetch by default, and some endpoint hardening images turn it off on workstations citing SSD wear. Confirm HKLM\SYSTEM\CurrentControlSet\Control\Session Manager\Memory Management\PrefetchParameters\EnablePrefetcher before treating absence as proof of non-execution.
🧬Amcache.hve — the file inventory that outlives the binary
Hashes survive deletion SHA-1 per PE IOC join key
C:\Windows\AppCompat\Programs\Amcache.hve

Where: a standalone registry hive maintained by the compatibility appraiser scheduled task. Triggered roughly daily and on package install / uninstall — not on every execution, so its timestamps are scan times, not run times.

Structure: nested keys under Root\. The forensically useful subtrees:

  • Root\InventoryApplicationFile — every PE the appraiser ever scanned. Holds full path, SHA-1, ProductName, ProductVersion, BinFileVersion, Publisher, Size, LinkDate, and the appraiser's first-seen timestamp.
  • Root\InventoryApplicationShortcut — LNK files seen on disk; cross-references §12.
  • Root\InventoryDriverBinary — driver inventory with image path, signer, SHA-1 — the BYOVD investigation anchor.
  • Root\InventoryDeviceContainer — connected device inventory (printers, USB, etc).

Answers:

  • The SHA-1 of a binary after it has been deleted — pivot directly to threat-intel and TI feeds without recovering the bytes.
  • A first-observed window (when the appraiser first noticed the file).
  • Did a now-missing driver ever load on this host? InventoryDriverBinary.
  • Original publisher / signer string baked into the PE resources.

Cannot tell you: exact execution times — Amcache records existence, not launches. A file present at appraiser-scan time appears here even if it never ran. AmcacheParser (Eric Zimmerman) is the canonical parser.

📇ShimCache / AppCompatCache — the deceptive one
Presence ≠ execution $SI Last-Modified Path + size
SYSTEM\ControlSet001\Control\Session Manager\AppCompatCache\AppCompatCache

Where: a single REG_BINARY value inside the SYSTEM hive. Flushed to disk only at shutdown — a live host's cache lives in kernel memory until then.

Structure: a LRU-ordered array of entries. Each entry on Win10+ holds the full file path and the $STANDARD_INFORMATION.Modified timestamp of the file at the time of insertion. Cache capacity caps at roughly 1024 entries on Win10/11; older entries fall off the LRU as new files appear.

Answers:

  • A list of executables the compatibility engine looked at, with the file's modified-time at the moment it was seen.
  • LRU position gives a coarse relative ordering — earlier-position entries were touched more recently.
  • Path normalisation reveals removable-drive prefixes and UNC paths that other artefacts may have lost.

Cannot tell you: when each binary was last executed; whether it was executed at all on Win10+ (the "execution flag" present on Win7/2008R2 was removed). The timestamp is the file's mtime — not a run time. AppCompatCacheParser handles the per-OS variants.

🚫Myth: "It's in ShimCache, so it ran"
Analyst-error class

On Windows XP through Windows 8, AppCompatCache stored an explicit Execute flag set when the binary actually ran. That flag was removed in Windows 10. On modern Windows the cache is populated by the compatibility-engine's file-scan path, which fires under several non-execution code paths:

  • The user right-clicks a file and opens the Properties dialog.
  • A process enumerates a directory containing the binary (Explorer browsing, antivirus scan, indexer).
  • An installer touches the file during a copy.
  • A file is moved or renamed.

Treat ShimCache as evidence that the OS noticed a binary at a certain modified-time — never as proof of execution on its own. Cross-check Prefetch, UserAssist, BAM, or Security 4688 before drawing the run/no-run conclusion.

⛔Reporting rule: a finding worded "ShimCache shows the binary executed at <time>" is incorrect on Win10+. Correct wording: "ShimCache shows the binary was present with mtime T, and was seen by the compatibility engine at or before shutdown T'."
👤UserAssist — Explorer-launched programs, per user
Proves execution Per-user (NTUSER) Last-run + run count
NTUSER.DAT\Software\Microsoft\Windows\CurrentVersion\Explorer\UserAssist\{GUID}\Count

Where: per-user, inside NTUSER.DAT. Two main GUIDs of interest: {CEBFF5CD-...} for executables and {F4E57C4B-...} for shortcut launches. Maintained by Explorer for Start-menu / taskbar ranking.

Structure: value names are ROT13-encoded paths (a trivial obfuscation, not encryption). Each value's binary blob holds a RunCount, FocusCount, FocusTime (ms), and a LastExecuted FILETIME.

Answers:

  • That a specific user account launched a specific binary from Explorer — Start menu, taskbar, desktop double-click, file association.
  • Last-run timestamp and cumulative run count per binary per user.
  • Time spent in the foreground (FocusTime) — a long focus on a sketchy binary corroborates intent.

Cannot tell you: things launched from a console, service, scheduled task, WMI, or another process — UserAssist only records the Explorer launch path. RegRipper's userassist plugin or RECmd handles the ROT13 decode automatically.

🔋BAM / DAM — kernel-recorded execution per SID
Proves execution Per-SID Survives binary deletion
SYSTEM\ControlSet001\Services\bam\UserSettings\<SID>\<path> SYSTEM\ControlSet001\Services\dam\UserSettings\<SID>\<path>

Where: the Background Activity Moderator (BAM) and its desktop counterpart (DAM) are kernel services that throttle background processes for battery / focus assist. They keep a per-user (per-SID) list of every binary the scheduler last ran, indexed by the full image path.

Structure: each value name is the binary's image path (\Device\HarddiskVolumeN\...); the value's first 8 bytes are a FILETIME of the last execution by that SID. Available from Windows 10 1709 onwards.

Answers:

  • The most recent execution time of a binary, by a specific SID — direct mapping to a user via §07 SAM.
  • That the kernel itself observed the process — extremely hard to fabricate without admin.
  • That a now-deleted binary was once executed, because the registry value persists after the file is wiped.

Cannot tell you: run count, arguments, or pre-1709 history. Only the last execution timestamp is stored per (SID, path) pair. RegRipper's bam plugin and RECmd's BAM batch parse it.

🪟MUICache — UI-display corroborator
Per-user (NTUSER) FriendlyAppName
NTUSER.DAT\Software\Classes\Local Settings\Software\Microsoft\Windows\Shell\MUICache

Where: per-user, inside UsrClass.dat (the classes hive — distinct from the main NTUSER.DAT branch but typically grouped with it in collection).

Structure: a flat list of value names, each being a binary's image path with the suffix .FriendlyAppName or .ApplicationCompany. The value data is the string Explorer cached from the PE's version resource the first time it had to display a window-title or right-click menu for that binary.

Answers:

  • That a binary displayed a UI under this user account at some point — corroborates UserAssist, eliminates pure-CLI / pure-service runs.
  • The original FriendlyAppName the publisher embedded — useful when the file has been renamed on disk.

Cannot tell you: when the UI was displayed (no timestamps; only the key's LastWriteTime, which moves on every change to any subvalue). Less precise than Prefetch, but valuable when the binary is deleted and Amcache is unavailable.

📊SRUM — what every app did, in 60-minute buckets
Proves execution Per-hour buckets Network bytes per app
C:\Windows\System32\sru\SRUDB.dat

Where: the System Resource Usage Monitor. An ESE (Extensible Storage Engine) database; same family as NTDS.dit and Exchange stores. Written by the DPS (Diagnostic Policy Service) and the diagnostic data task.

Structure: a set of tables keyed by app ID and user SID, with one row per app per hour. Notable tables:

  • {D10CA2FE-...} — Application Resource Usage: CPU cycles, foreground/background time, working set, write/read bytes.
  • {973F5D5C-...} — Network Data Usage: bytes sent / received per (app, interface).
  • {DD6636C4-...} — Network Connectivity: which interface was up in each hour.
  • {D10CA2FE-6FCF-...} — push-notification deliveries.

Answers:

  • That a specific binary ran in a given hour (presence of a row = ran).
  • How much network traffic it generated per interface — the exfil-sized-binary finding lives here.
  • How long it spent in the foreground (intent indicator).
  • Which user SID owned the process.

Cannot tell you: exact launch / exit times — buckets are aggregated to the hour. Retention is ~30–60 days then aged out. SrumECmd / SrumDump produce per-table CSV.

📰Event log execution traces
Proves execution Sub-second precision Includes SID + parent
C:\Windows\System32\winevt\Logs\*.evtx

When auditing is configured, the event logs are the highest-fidelity execution record on the system — they include arguments, parent process, integrity level, and TokenElevationType. The catch: most of them are off by default.

  • Security 4688 — process creation. Disabled by default; requires Audit Process Creation + the optional ProcessCreationIncludeCmdLine_Enabled for command-line capture. When present, contains NewProcessId, NewProcessName, ParentProcessName, SubjectUserSid, CommandLine, TokenElevationType (1=full, 2=elevated, 3=limited).
  • Sysmon Event ID 1 — process creation, richer than 4688: image hash, signer, original filename, integrity level, parent command line, all wrapped in a configurable filter. Only present if Sysmon is deployed.
  • Application 1000 — application crash. The "Faulting application name" field reveals binaries that ran long enough to crash even when 4688 is off.
  • Security 4689 — process termination (with exit status, when audited).
  • PowerShell 4104 — script-block logging; cross-references §13.

Cannot tell you: anything before audit was turned on. The cluster of artefacts in this section exists precisely because organisations rarely enable 4688 broadly.

📷CapabilityAccessManager — mic / cam / location use
Per-resource last-used Per-user + per-system Stalkerware indicator
NTUSER.DAT\Software\Microsoft\Windows\CurrentVersion\CapabilityAccessManager\ConsentStore\<resource>\<app> SOFTWARE\Microsoft\Windows\CurrentVersion\CapabilityAccessManager\ConsentStore\<resource>\<app>

Where: the privacy subsystem keeps a per-app record of every capability the user granted (microphone, webcam, location, contacts, calendar, files, etc). HKCU side records the user's grants; HKLM side records system-wide grants.

Structure: a key per resource (microphone, webcam, location, contacts…) containing a key per app. Each app key has LastUsedTimeStart and LastUsedTimeStop FILETIMEs — the most recent access window.

Answers:

  • The last time a specific app held the microphone or webcam open — direct evidence in stalkerware / covert-recording cases.
  • The exact app identifier (AppX package family name or Win32 path) that accessed the resource.
  • That a non-AV non-meeting binary touched the camera at 03:00.

Cannot tell you: what was actually captured; only that access occurred. Only the most recent access window is stored — earlier sessions are overwritten.

📌Triangulation rule of three: the strongest execution finding combines (a) a system-scope, kernel-recorded witness (Prefetch or BAM), (b) a per-user witness with a SID (UserAssist or BAM), and (c) a file-identity witness (Amcache SHA-1 or Sysmon hash). All three pointing at the same binary at compatible timestamps is the gold standard.

Cross-references for corroboration

Execution artefacts rarely stand alone — every finding here is strengthened by joining against the substrate sections:

  • §04 NTFS — $MFT / $UsnJrnl — confirms the binary existed at the path the execution artefact names. $UsnJrnl FILE_CREATE ↔ Prefetch first-run; $UsnJrnl FILE_DELETE explains why the binary is now missing while BAM / Amcache still reference it.
  • §04 NTFS — $Volume serial — the VSN inside each Prefetch's volume-info block joins to $Volume, LNK TrackerDataBlock, and §14 USB Event 1006 — proves which physical volume hosted the binary.
  • §07 SAM — resolves the SID inside BAM UserSettings\<SID>, SRUM rows, and Security 4688 SubjectUserSid back to a human username and account-creation time.
  • §08 SYSTEM / SOFTWARE hives — services (SYSTEM\CurrentControlSet\Services), Run / RunOnce, scheduled-task XML, and Image File Execution Options all give the autostart context that distinguishes a user-launched binary from a persistence implant. Sysmon parent image points here.
  • §09 NTUSER.dat — RecentDocs / RunMRU / TypedPaths — UserAssist's Explorer-launch finding is corroborated by the user's per-extension MRU list and any path typed into the Run dialog.
  • §12 LNK & ShellBags — when UserAssist's {F4E57C4B-...} GUID lights up, the matching LNK in Recent\ reveals the original target volume, file size, and target's $OBJECT_ID. ShellBags confirm the directory was browsed.
  • §13 Jump Lists / PowerShell history — Jump Lists confirm right-click and taskbar launches; ConsoleHost history and PowerShell 4104 cover console-launched binaries that UserAssist will not record.
  • §11 Cloud Storage — for binaries staged from OneDrive / Dropbox, the cloud-sync logs cross-reference Prefetch's first-run window and explain how the file reached the host.
