📁OneDrive — SyncEngineDatabase.db
Authoritative sync state Resource ID ↔ Path eTag history
%LocalAppData%\Microsoft\OneDrive\settings\Personal|Business1\SyncEngineDatabase.db

Where: per-tenant subfolder (Personal for consumer accounts, Business1..N for each work tenant signed in). The SQLite database is the OneDrive client's source of truth: every file the user has touched in the synced root has at least one row. The companion file SyncEngineDatabase.db-wal often contains the most recent transactions and must be carved together with the main file.

Key tables:

  • od_ClientFile — one row per object with resourceID, parentResourceID, localHashDigest, serverHashDigest, size, file/folder flag, and the localWriteTime / serverLastChange pair that exposes pre-sync clock skew.
  • od_GraphMetadata_Records — sharing, links, sensitivity labels, and (post-Sept 2021 clients) per-file eTags capturing every server-side mutation.
  • od_HydrationData — for Files-On-Demand, distinguishes always available, online-only, locally available, the moment hydration occurred, and any pinning user action.

Answers: what the user has in their OneDrive even when local cache is empty; what was deleted and when; what was renamed; what hash the server holds (joins against §10 Prefetch on payload binaries staged from cloud).

ℹFiles-On-Demand placeholders are NTFS reparse points with an IO_REPARSE_TAG_CLOUD tag. They show in $MFT as zero physical bytes — §04 NTFS will appear to show an empty file. Always cross-check SyncEngineDatabase before concluding a user "had no copy."
🗑OneDrive — SafeDelete.db & UserCid.dat
Tombstones Account identity Server-side deletes
…\OneDrive\settings\<tenant>\SafeDelete.db …\OneDrive\settings\<tenant>\UserCid.dat (plaintext CID/objectID)

SafeDelete.db holds soft-delete and conflict resolution metadata — the local record of "the server said this is gone." Each row carries the resourceID, full path at time of deletion, timestamp, and the deletion source (user, server, policy, retention). When server-side admin deletes happen during an investigation (e.g. exfil cleanup), this is often the only host-side proof the file ever existed.

UserCid.dat stores the customer/tenant identifier — for Personal accounts a 16-hex-character CID, for Business the Azure objectID of the user. This is the join key against M365 Unified Audit Logs, SharePoint sign-in logs, and Azure AD audit records.

⚠The Recycle Bin in OneDrive runs server-side for 30–93 days and is NOT mirrored locally. If a threat actor used the web UI to delete uploaded data, SafeDelete will record the local tombstone but the file body lives only in M365 — request preservation immediately.
📝OneDrive ODL Logs — the verbose telemetry stream
Per-event timeline Function-level traces Survives cache wipe
%LocalAppData%\Microsoft\OneDrive\logs\Personal|Business1\*.odl[gz|sent]

What: binary, gzip-compressed structured logs written by every running OneDrive process. Each row is a function-call event with timestamp, thread ID, function name, and 1–4 free-form parameter strings (paths, error codes, HTTP responses). Path strings are obfuscated with a per-install Obfuscation Map (the ObfuscationStringMap.txt file in the same folder) — without that map you only see hashes.

Why it matters: ODL captures every upload, download, hydration, rename, conflict, throttling event, and authentication step — far more granularity than SyncEngineDatabase. Post-incident exfil scenarios live or die on ODL because it records WHICH file went up and WHEN, including the bytes-uploaded counter.

Retention: the client rotates roughly daily and keeps about 30 logs before pruning — collect early. Parsed with Yogesh Khatri's odl.py, the open-source CCL Solutions parser, or KAPE's OneDriveLogs module.

🔐OneDrive Personal Vault
BitLocker volume MFA-protected 2-FA auth event
%UserProfile%\OneDrive\Personal Vault (presents as locked drive when active)

Mechanism: Personal Vault mounts a small BitLocker-To-Go VHD as a virtual drive whenever the user authenticates with MFA. While unlocked, contents are normal NTFS files; when locked, only the VHD remains.

Forensic surface:

  • ClientPolicy_Vault*.ini in the settings folder marks vault presence.
  • ODL VaultUnlock and VaultLock events stamp every access — these tie to a specific TOTP or app-prompt MFA challenge in Azure AD.
  • SyncEngineDatabase rows for vault items carry an isolated parentResourceID beneath a sentinel "Vault Root" object.
  • Once unlocked, Vault items appear in §09 NTUSER RecentDocs / §12 LNK like any normal file — providing the only easy index of what was inside.
📦Dropbox — filecache.dbx, deleted.dbx, config.dbx
SQLite, mostly Server hashes Deleted-file ledger Per-host encryption (legacy)
%LocalAppData%\Dropbox\instance1\filecache.dbx (current) / instance_db (older clients)

filecache.dbx is a SQLite database whose tables include file_journal (server_path, local_path, server_size, server_mtime, server_hash, local_size, local_mtime, local_hash) and block_cache (hash → chunk pointers). Modern (2017+) clients store it in cleartext SQLite; very old Dropbox clients wrapped the file with an SQLite extension keyed off DPAPI material — for those, you need the user's password or DPAPI master key.

deleted.dbx is the local tombstone log — every server-deleted file leaves a row with original path, last server modification time, and deletion timestamp. Like OneDrive's SafeDelete, this often outlives the file body itself.

config.dbx holds the linked account ID, host_id (the cookie the server uses to authenticate this install — treat as a credential when handling images), client_version, and selective-sync exclusion list. The exclusion list reveals folders the user chose not to sync — a powerful tell for staging behaviour.

Hashes: Dropbox uses a unique block-hash scheme (4 MB SHA-256 blocks, concatenated, hashed again). Two installs of the same file produce identical block hashes — useful for joining across endpoints in multi-host incidents.

✖The host_id in config.dbx is enough to impersonate the Dropbox client to the server, retrieving the user's whole namespace. Treat this artefact like a session token: image it, do not casually share, and rotate the user's password before returning the device.
📂Google Drive for Desktop (DriveFS)
SQLite metadata DocID survives rename Modification log
%LocalAppData%\Google\DriveFS\<profile-id>\metadata_sqlite_db

What: the current Google Drive client (formerly Backup & Sync / File Stream) projects a virtual drive letter and caches metadata in metadata_sqlite_db. Key tables include items (the docID, parents, mime type, title, size, modified time, viewed-by-me time) and stable_id mapping rows that survive folder moves and renames.

Why docID matters: Google's docID is immutable for the life of the file — even if the user renames a document, the docID stays. This lets you correlate against Google Workspace Admin audit logs (which always log docID, not path) and prove the same document was opened across endpoints.

Companion: content_cache\ holds locally-cached file bodies named by content hash — recover deleted-from-Drive bodies here even when the row is gone from items.

📥Box Drive & Box Tools
Edit-then-upload events Enterprise file_id Local cache copies
%LocalAppData%\Box\Box\data\boxdata.sqlite (Box Drive) %LocalAppData%\Box\Box Edit\Documents\<id>\… (Box Edit working copies)

Box Drive's boxdata.sqlite tracks file_id, parent_id, name, size, modified_at, owner_id and sync_status. The Box Edit helper writes full local working copies any time a user clicks "Open" in the web UI on an Office doc — these working copies are often the only place an enterprise document touches the endpoint disk, and they live under per-doc subfolders even after the user "closes" the edit session.

Browser-driven uploads (sandbox), CLI uploads (Box CLI), and the Box for Office add-in each leave different traces — confirm which client was in use before reasoning about completeness.

🚨Exfiltration triage — what to look for first
Bulk download anomalies Off-hours sync bursts Newly-linked accounts

Cloud sync clients are dual-use — they're how data leaves and how forensic evidence arrives. A quick triage when a host is suspected of exfil:

  • New tenants: count subfolders under OneDrive\settings\. Business2 appearing where there was only Business1 means a second M365 tenant was linked — usually an attacker-controlled mailbox or a personal account.
  • config.dbx / settings timestamps: sudden modification right before the incident equals new auth or selective-sync change.
  • ODL "Upload" events vs SyncEngineDatabase size deltas: if ODL records uploads of files that no longer appear in the database, the attacker uploaded then unlinked.
  • Browser downloads of installers for sync clients the org doesn't sanction (rclone, MEGAsync, pCloud) — §15 will pivot here using §10 Prefetch / Amcache.
  • Wallpaper, screen-time, and presence APIs recorded in ODL reveal whether the user was actively at the keyboard during the upload — distinguishes interactive exfil from background sync.
⚠Sync clients can also be used to stage malware. A binary that appears in %UserProfile%\OneDrive\… and then runs out of that path will leave Prefetch / Amcache execution evidence but no Internet browser download history — the OneDrive sync was the delivery channel. Joining ODL upload events from a different host against this host's hydration events proves the lateral path.

📊Server-side data to pull in parallel
Cloud-side audit

Host-side cloud artefacts are necessary but never sufficient. Pair every endpoint with:

  • M365 Unified Audit Log — joins UserCid.dat / objectID; preserve for 180+ days via litigation hold.
  • Azure AD sign-in logs — geolocation + device ID per OneDrive auth.
  • Google Workspace Admin Audit (Drive scope) — joins docID; gives view/download/share events the client never sees.
  • Dropbox Business Audit Log — joins host_id.
  • Box Enterprise Events API — joins file_id; captures shared-link creation that's invisible client-side.
Cross-references for corroboration

Cloud sync evidence is rarely the whole story — every finding here needs joining to the local-host substrate:

  • §04 NTFS — reparse points — Files-On-Demand placeholders appear in $MFT as zero-byte reparse tags. Do not conclude a file is missing without checking SyncEngineDatabase first. $UsnJrnl records hydration as a normal FILE_CREATE / DATA_OVERWRITE.
  • §07 SAM — the Windows account that holds the sync tokens resolves to a SID in SAM; that SID joins SyncEngineDatabase metadata to a real human.
  • §08 SOFTWARE — Run / Uninstall keys — confirms which sync clients are autostarted and which were ever installed (including uninstalled remnants that still leave settings folders).
  • §09 NTUSER — RecentDocs / TypedPaths — opening a Files-On-Demand placeholder hydrates AND adds a RecentDocs entry; absence of RecentDocs for a file present in SyncEngineDatabase suggests programmatic / script access.
  • §10 Application Execution — Prefetch / Amcache for binaries staged from cloud locations close the loop on "the file came from OneDrive and ran." OneDrive.exe itself appears in BAM with the SID, anchoring sync activity to a logon session.
  • §12 LNK — when a user pinned or right-clicked a cloud file, the LNK target path carries the local cache location; the TrackerDataBlock carries the volume serial of C:, not of any cloud storage — useful for distinguishing a launched cached copy from a "View Online" click.
  • §14 USB — for full data-loss timelines, cross USB plug-in events with sudden Dropbox / OneDrive selective-sync changes that would have moved data off cloud onto removable media.
  • §15 Timeline & Correlation — ODL is the most timestamp-dense source on a Windows host and will dominate any cloud-incident timeline. Normalise to UTC and merge with EvtxECmd output before reading.
