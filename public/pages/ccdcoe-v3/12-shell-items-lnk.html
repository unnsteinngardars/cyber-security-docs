📎The LNK format — a forensic gold mine in 0.5–8 KB
Target MAC times snapshot Volume serial + object ID MAC address of last-opening host Survives target deletion
%AppData%\Microsoft\Windows\Recent\<document>.lnk %AppData%\Microsoft\Office\Recent\ %UserProfile%\Desktop\, …\Start Menu\, …\Quick Launch\

What a shell link is: Microsoft documents it formally as MS-SHLLINK. Every .lnk file is a binary structure containing a fixed-size header, an optional LinkTargetIDList (a chain of shell items representing the path as the shell parses it), a LinkInfo block, optional string data, and a sequence of typed ExtraData blocks. Each layer carries different forensic information about the target at the moment the link was last refreshed.

Header (76 bytes) provides:

  • CreationTime / AccessTime / WriteTime of the target file as it existed when the LNK was last updated. These are FILETIME values stored independently of the target — even if the target is wiped, the LNK preserves the last-known MAC.
  • FileSize of the target.
  • FileAttributes (read-only, hidden, system, archive, etc.).
  • HotKey — interactive evidence: a non-zero hotkey field proves the user customised this link.
  • ShowCommand — Maximised / Minimised / Normal as the user last left it.

LinkInfo block typically yields: drive type (fixed / removable / network / CD-ROM), VolumeSerialNumber, VolumeLabel, local base path (e.g. C:\Users\bob\…), and (when applicable) the UNC path of a network share including server name. Two LNKs that point to the same physical USB stick will share the VSN even across hosts — this is the join key behind §14 USB correlation.

ExtraData blocks of forensic value:

  • TrackerDataBlock — the Distributed Link Tracker stamp: MachineID (NetBIOS name of the host that last "refreshed" the link), and three Object IDs containing the MAC address of the network adapter at the time of refresh and a timestamp embedded in the GUIDs.
  • PropertyStoreDataBlock — extended metadata (author, application, Office document GUIDs).
  • EnvironmentVariableDataBlock — for links that point to %UserProfile%\… style targets, preserves the unexpanded form alongside the resolved one.
  • SpecialFolderDataBlock / KnownFolderDataBlock — CSIDL / KNOWNFOLDERID values when the target sits inside a special folder.
  • VistaAndAboveIDListDataBlock — a second, post-Vista shell item list that may differ from the legacy one and is often what modern Explorer actually consumes.
ℹLNKs are auto-created by Explorer / Office in Recent\ on simple file opens. The user does not have to "save a shortcut" for one to appear. Conversely, on Win10+ controlled folder access and on hardened images Explorer may suppress Recent — confirm policy before reading absence as proof.
🧬Shell item lists — the structure beneath the path
Used in ShellBags, LNK, RecentDocs MRU Records folder MAC at access

A shell item is the binary representation of one element in a path as the shell parses it: Desktop, This PC, a volume, a directory, a file. They are concatenated into shell item lists (a.k.a. ID lists / PIDLs) and embedded inside LNK files, ShellBags, RecentDocs MRU entries, and Common Dialog MRUs.

Each item type encodes its own metadata:

  • File / folder items carry a short or long name, target MFT FILE reference (NTFS only), and target FILETIMEs.
  • Volume items carry the volume serial number and drive letter at the time of access.
  • Known-folder / CLSID items reference GUIDs like {20D04FE0-3AEA-1069-A2D8-08002B30309D} (This PC), {F4E57C4B-2036-45F0-A9AB-443BCFE33D9F} (Search Folder), or vendor-specific cloud-namespace GUIDs (OneDrive, Box).
  • Network / UNC items capture server name, share name, and the protocol (SMB / WebDAV).

Because shell items are embedded in many artefacts, knowing the format pays in every later section.

🎯What LNK reveals that no other artefact does
Unique evidence Cross-host pivots

Other artefacts say "this file was opened." LNK uniquely says:

  • It came from that volume. VSN + label + drive type place the file on a specific USB / network share / cloud namespace.
  • It was opened from that machine. TrackerDataBlock MAC + MachineID. When the same LNK is found on multiple endpoints, the MAC reveals the originating host of the file.
  • The target's MAC times at access. Even if the target was wiped clean by anti-forensics, the LNK still records when it was created / modified / accessed by the last refresher.
  • That the file was real and reachable. Explorer creates the LNK only after a successful open call — distinguishes "user opened this" from "the malware put a string here."
!VSN reuse is real. Two different USB sticks formatted by the same Windows host in close succession can land on identical VSNs. Always corroborate with disk label, USB SerialNumber from §08 USBSTOR, and Event 1006.
📋RecentDocs (binary) ↔ Recent\*.lnk
Per-user MRU order
NTUSER\Software\Microsoft\Windows\CurrentVersion\Explorer\RecentDocs\<ext>

The RecentDocs key (per-user, per-extension subkey) stores binary blobs that are shell item lists referencing each opened file plus a UTF-16 caption. The companion MRUListEx in each subkey is a uint32 array recording the open order.

Pair each RecentDocs entry with the matching Recent\<name>.lnk for the full forensic story: RecentDocs gives MRU order and last-write timestamp; the LNK gives volume / tracker / target MAC. A RecentDocs entry without a corresponding LNK often signals targeted LNK deletion as anti-forensics.

📚Office Recent — its own world
Per-Office-version Independent of Explorer Recent
%AppData%\Microsoft\Office\Recent\ NTUSER\Software\Microsoft\Office\<ver>\<app>\File MRU

Word, Excel, PowerPoint and Outlook each maintain their own MRU registry keys plus an Office\Recent folder full of LNKs. Modern Office (2016+) also pins pinned documents and stores per-document UserMRU entries with absolute paths to local files and SharePoint URLs to documents that never landed on disk — recovery here is the only host-side proof a user opened a cloud-only document.

🧱ShellBags — proof a folder was browsed
Per-user First / last access per folder Survives folder deletion Network & USB browsing
USRCLASS.DAT\Local Settings\Software\Microsoft\Windows\Shell\Bags & BagMRU NTUSER\Software\Microsoft\Windows\Shell\Bags & BagMRU (legacy + some dialogs)

What ShellBags do: they cache Explorer's view preferences for each folder a user opens — column widths, sort order, icon size — so the next visit looks the same. To do this Windows must remember every folder ever browsed, and that index is what we read.

Structure: the BagMRU tree mirrors the folder hierarchy as the user navigated it. Each node stores a shell item (so volume info, target MAC times, network UNCs are all preserved) and an MRUListEx of its children. The parallel Bags subkey stores the actual view settings.

What ShellBags prove:

  • The user (specifically — this is NTUSER / USRCLASS, per-account) navigated into a folder using Explorer or the Common Dialog at least once.
  • The folder existed at that path at that moment, with the recorded MAC times.
  • For USB and network shares — the volume label, serial, server, and share name as seen by Explorer.
  • The most recent registry write-time of the bag node is the upper bound on last browsed.

What ShellBags do NOT prove: that any file was opened from inside the folder, only that the folder was viewed. Distinguish browsing from access by joining against RecentDocs / LNK.

⚠ShellBags are only written when Explorer / a Common Dialog is the navigator. cmd.exe, PowerShell, robocopy, xcopy do not create ShellBag entries. Absence of a ShellBag for a folder does NOT prove the user never went there — only that they did not go there via the shell.
🧭ShellBags use cases in real cases
Investigation patterns
  • Insider data theft: ShellBag entries with USB drive letters, network share UNCs, or unfamiliar cloud-namespace GUIDs reveal where the user went looking.
  • Counter-anti-forensics: ShellBag entries for C:\Users\<name>\AppData\… staging directories that were later wiped — the folders are gone but the ShellBag and its embedded MAC times persist.
  • Account takeover: ShellBag activity at unexpected hours or for folders the legitimate user never touched in OneDrive logs is a strong sign of remote interactive session abuse.
  • Lateral movement: ShellBag entries containing \\target-host\C$\… are uncontestable evidence of admin-share browsing.
💥Tampering & legitimate edge cases
Pitfalls
  • An LNK can be created without opening the target — drag-and-drop with right-click "Create shortcut here" works, as do crafted .lnk droppers (e.g. CVE-2010-2568 era and modern variants). Treat creation as a necessary but not sufficient indicator of opening.
  • LNKs in Recent\ are renamed by Explorer when the target is renamed — the on-disk LNK timestamp may post-date the actual open.
  • ShellBags compress over time: legacy bags get pruned on Windows version upgrades; record the OS build with every finding.
  • $MFT records of LNK files give creation / modification of the LNK itself, which usually equals first / last open of the target — a powerful sanity check.

Cross-references for corroboration

LNK and Shell Item evidence is at its most powerful when triangulated with the substrate:

  • §04 NTFS — $MFT / $UsnJrnl / $Volume — the LNK's VSN joins to $Volume serial. The FILE_CREATE of the LNK in Recent\ equals first open; $UsnJrnl records every later open as a write to the LNK.
  • §07 SAM — Recent / Office MRU paths sit under a per-user profile whose owning SID maps via SAM to a real user and account creation date.
  • §08 SYSTEM — MountedDevices & USBSTOR — the volume serial inside a LNK joins to MountedDevices and to USBSTOR serial numbers, identifying the specific physical USB stick.
  • §09 NTUSER — RecentDocs / Office MRU / TypedPaths / RunMRU — every LNK in Recent\ has a matching RecentDocs entry, and Office MRU paths cross with Office Recent\ LNKs to confirm app context.
  • §10 Application Execution — the UserAssist GUID {F4E57C4B-2036-45F0-A9AB-443BCFE33D9F} counts launches from Recent / Start; the LNK reveals which target.
  • §11 Cloud Storage — LNKs to OneDrive / Box-Edit cache directories prove a cloud document was opened locally — and the embedded VSN belongs to C:, never to the cloud namespace.
  • §13 Jump Lists — Automatic and Custom destinations contain LNK structures internally — every concept on this page also applies inside .automaticDestinations-ms.
  • §14 USB Event Log & Drive History — VSN + Volume Label from LNK joins to Event 1006 (Partition/Diagnostic), Setupapi log entries, and §08 USBSTOR to identify specific removable devices used to exfiltrate data.
  • §15 Timeline & Correlation — LECmd / SBECmd CSV outputs merge naturally into Timeline Explorer for the master timeline. LNK target MAC times often pre-date the LNK's own create time — preserve both columns.
