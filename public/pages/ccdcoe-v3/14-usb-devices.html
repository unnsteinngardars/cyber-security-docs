🔌What Windows actually does when a USB is inserted
Plug-and-play pipeline Multiple timestamps Cross-hive correlation

From an investigator's perspective, a single USB insertion fires a cascade of writes:

  • Kernel PnP enumerates the device; setupapi.dev.log records driver matching with a wall-clock timestamp (first install of this VID/PID/SN).
  • The class driver registers the device under SYSTEM\CurrentControlSet\Enum\USB\VID_xxxx&PID_xxxx\<SerialNumber> — and, for storage devices, also under USBSTOR\Disk&Ven_xxx&Prod_xxx&Rev_xxx\<SerialNumber>.
  • Properties subkeys (0064, 0066, 0067, 0083) store the First Install, Last Insertion, Last Removal, and Last Driver Install FILETIMEs respectively — added in Win7+ and refined in Win10.
  • Volume binding: SYSTEM\MountedDevices maps drive letters to volume objects (DOSDevice -> NT device path embedding the VSN); SOFTWARE\Windows Portable Devices\Devices records the user-visible label; NTUSER\…\MountPoints2 records the per-user GUID corresponding to the volume the moment that user navigated into it via Explorer.
  • Event logs: Microsoft-Windows-Partition/Diagnostic logs Event 1006 with a wealth of per-disk info; DriverFrameworks-UserMode logs each enumeration step; Kernel-PnP logs power transitions; Security 6416 logs device class events under audit policy.
ℹNo single artefact is sufficient. Modern USB triage joins setupapi.dev.log + USBSTOR + USB + MountedDevices + MountPoints2 + Partition/Diagnostic 1006 to land a specific device, a specific user, a specific volume, and a specific time interval.
💾USBSTOR — the storage-device ledger
VID/PID + Serial First / last connect
HKLM\SYSTEM\CurrentControlSet\Enum\USBSTOR\Disk&Ven_*&Prod_*&Rev_*\<SerialNumber>

Each subkey represents one storage device that was ever plugged in. Values of forensic interest:

  • FriendlyName — human-readable description ("Kingston DataTraveler 3.0").
  • ContainerID — a GUID that stays stable for the physical device even when reformatted; powerful join key across reinstallations.
  • Properties\{83da6326-97a6-4088-9453-a1923f573b29}\0064/0066/0067 — InstallDate, LastArrival, LastRemoval FILETIMEs.
  • DeviceDesc + HardwareID — strings used by setupapi.dev.log to match.

The SerialNumber in the registry path is the join key for everything below. If the SN ends in &0, the device does not have a real serial (typically cheap flash drives) and Windows synthesised one from the bus location — meaning that "serial" is bus-position-dependent and not unique across hosts.

🔗USB — the bus-level ledger
All USB devices, not just storage
HKLM\SYSTEM\CurrentControlSet\Enum\USB\VID_*&PID_*\<SerialNumber>

The bus-enumeration parent of USBSTOR (and of every other USB device — keyboards, webcams, software-defined radios, BadUSB-style impostors). Storage devices appear here too, with the same SerialNumber that USBSTOR uses, plus a LocationInformation value indicating the physical port that was used and the parent hub. Comparing arrival timestamps across nodes can sometimes show which physical port a stick was plugged into.

📌MountedDevices — drive letter ↔ volume
VSN binding Per-system, not per-user
HKLM\SYSTEM\MountedDevices

One binary value per drive letter (\DosDevices\E:) and per volume GUID (\??\Volume{guid}). The binary payload encodes the NT device path — for USB volumes this string contains the USBSTOR DeviceInstanceID and embeds the volume serial number; for fixed disks it encodes the partition signature + offset.

Use: given a VSN found in a LNK or ShellBag, search MountedDevices for the matching binary blob to recover the drive letter the system assigned. Inverse use: pick a drive letter from a Recent\*.lnk basepath and find which USBSTOR device backed it.

📅setupapi.dev.log — the install diary
Wall-clock timestamps Text log, easy to parse
C:\Windows\INF\setupapi.dev.log (current) + setupapi.dev.YYYYMMDD_HHMMSS.log (rotated)

Plain text. Every driver install or device-class enumeration writes a stanza with a UTC-offset timestamp, the device instance ID (VID/PID/SN), and the .inf file used. The first appearance of a given SerialNumber is the device's first-ever plug-in on this host — a fact USBSTOR's properties also record but is hardest to dispute when present here.

Rotated logs go back further than the registry sometimes does (especially on long-lived images that lost CurrentControlSet content during repair).

⚠setupapi.dev.log can be edited. A motivated insider can scrub lines. Sanity-check against Kernel-PnP and DriverFrameworks-UserMode event logs, which the user typically cannot edit without admin rights and tooling.

👤MountPoints2 — the per-user binding
SPECIFIC user used the volume Key last-write time
NTUSER\Software\Microsoft\Windows\CurrentVersion\Explorer\MountPoints2\{volume-guid}

Inside each profile's NTUSER there's a subkey for every volume GUID that user's Explorer instance ever navigated into. The presence of a MountPoints2 entry for volume {G} proves that this user — not just some user on the host — touched that volume.

The key's last-write time approximates the last time that user navigated to the volume in Explorer. Combined with §12 ShellBags entries under the same volume GUID, you get an immediate "did they actually browse it" signal.

🏷VolumeInfoCache & Windows Portable Devices
Volume label history
HKLM\SOFTWARE\Microsoft\Windows Search\VolumeInfoCache\<drive> HKLM\SOFTWARE\Microsoft\Windows Portable Devices\Devices\<guid>

VolumeInfoCache stores the volume label as Windows Search saw it on the most recent mount of each drive letter. Windows Portable Devices\Devices stores the FriendlyName the device presented (sometimes the manufacturer's branding, sometimes the user's chosen volume label) and the device instance ID.

Together these give you the human-readable name of a volume even after it's been wiped or reformatted.

🎯Standard USB triage workflow
From mount to user
  1. Enumerate devices. Dump USBSTOR + USB subkeys. For each, capture SerialNumber, FriendlyName, ContainerID, InstallDate / LastArrival / LastRemoval.
  2. Validate with setupapi.dev.log. First appearance line per SerialNumber.
  3. Bind to volumes. Walk MountedDevices, extract the DeviceInstanceID from each \??\Volume{} binary blob, match SN to USBSTOR. Capture drive letter mappings.
  4. Bind to users. For each profile, walk MountPoints2 — list volume GUIDs and key write times. Now you have (user, volume, time-of-explorer-access).
  5. Bind to events. Pull Event 1006 by date range and match VSN. Pull Kernel-PnP 410/411/430 by DeviceInstanceID for connect / disconnect timestamps.
  6. Bind to files. Search §12 LNK and ShellBag entries for the volume serials gathered above; search Recent\AutomaticDestinations (§13) for the same. This is where "what files did the user open from this stick" gets answered.
  7. Look for copies in the other direction. §04 $UsnJrnl on C: may record bulk reads from the removable volume (rarely) but more commonly records writes to the stick if it stayed inserted long enough — only useful when the stick itself is also under analysis.
!Spoofed serials are real. Some BadUSB / impersonation devices clone the VID/PID/SN of a known good device. Always check the MBR signature from Event 1006 — two devices reporting identical SN with different MBR signatures are not the same physical device.
🧨USB-borne malware fingerprints
Beyond storage
  • HID impersonators (Rubber Ducky, Bash Bunny, P4wnP1). Look in USB\VID_*&PID_* under the HID class for devices with FriendlyName "HID Keyboard Device" plugged in at the same minute as an unexplained PowerShell launch (§10 / Sysmon 1).
  • USB Ethernet (e.g. Bash Bunny in NAC bypass mode). A Realtek RTL8152 appearing on a host that should not have USB Ethernet is suspect.
  • Mass-storage with autorun.inf — Windows 10/11 ignores autorun on most volumes by default, but the Win7-era artefact persists; absence of execution does not erase the registry trace of insertion.
  • Encrypted USB partitions (BitLocker To Go, VeraCrypt). The outer USBSTOR entry looks normal; mounting the inner encrypted volume creates a second MountedDevices binding and a separate set of LNK volume serials.

Cross-references for corroboration

USB evidence is multi-source by nature — every confident USB finding rests on three or four of the following:

  • §04 NTFS — $Volume & $UsnJrnl — VSN inside MountedDevices, LNK, and 1006 joins to $Volume on the stick itself when you have it. $UsnJrnl on C: occasionally records writes to mapped drive letters during long mounts.
  • §07 SAM — MountPoints2 lives per-user; the SID resolved via SAM ties a specific human to a specific volume.
  • §08 SYSTEM & SOFTWARE — USBSTOR, USB, MountedDevices are all in SYSTEM; VolumeInfoCache and Windows Portable Devices live in SOFTWARE. The §08 walkthrough covers the parent structure.
  • §09 NTUSER — MountPoints2, RecentDocs, TypedPaths — proves the user navigated to the USB drive letter and what they typed into the address bar to get there.
  • §10 Application Execution — Prefetch entries naming a binary path on a USB drive letter (or Amcache rows with the USB volume's drive type) prove an EXE was executed from the stick — major distinguishing fact for "user merely browsed" vs "user ran something".
  • §11 Cloud Storage — sometimes a stick is the bridge between an air-gapped host and cloud upload elsewhere. Cross with ODL upload events on neighbouring hosts using file hashes.
  • §12 Shell Items & LNK — every LNK with drive type Removable carries the VSN that joins back here. Hot pivot: grep LECmd CSV for non-fixed drive types.
  • §13 Jump Lists — embedded per-entry LNKs in AutomaticDestinations preserve removable-drive references long after the LNK in Recent\ is gone.
  • §15 Timeline & Correlation — when assembled, USB plug-ins are anchor events. A connect at T, ShellBag write at T+5s, RecentDocs entry at T+45s, Prefetch first-run at T+60s, removal at T+10m is a textbook "user opened a binary off this stick" sequence.
