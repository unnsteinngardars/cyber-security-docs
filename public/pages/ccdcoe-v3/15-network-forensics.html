🌐Why network forensics — and how host artefacts fit in
Two evidence planes Wire ↔ host correlation Independent clock source

"Network forensics" splits into two evidence planes that are best treated as one investigation:

  • The wire. What was actually transmitted — captured live with tcpdump / Wireshark, replayed from PCAPs, or reconstructed by passive parsers like NetworkMiner.
  • The host. What the endpoint remembered about its own networking — interface configuration, networks it has ever joined, proxies it was told to use, RDP destinations it connected to. These live entirely in the registry hives covered by §08 SYSTEM / SOFTWARE and §09 NTUSER.

Either plane on its own is incomplete: PCAP tells you what travelled but not which user or process; host artefacts tell you intent and configuration but never the bytes. Together they triangulate attacker activity, lateral movement, and exfiltration with corroboration that survives challenge.

ℹWire timestamps come from the capture point's clock; host timestamps come from the endpoint's. Capture-point clock skew is the #1 mistake when overlaying PCAPs onto host timelines — record the time source for every capture device on first read.
🦈Wireshark & tshark — the GUI/CLI workhorse
Capture & analyse Dissector ecosystem
tshark -r case.pcap -Y 'tls.handshake.type == 1' -T fields -e ip.src -e tls.handshake.extensions_server_name

Wireshark is the canonical packet analyser. Its strength is the dissector library — over a thousand built-in protocol decoders that turn a byte stream into a tree of named fields. Critical features for forensics:

  • Display filters (post-capture) vs capture filters (BPF, pre-capture). Display filters are richer; learn them first.
  • Follow Stream reconstructs TCP / UDP / TLS / HTTP / QUIC application data into the user-readable conversation. The export menu yields raw bytes for malware extraction.
  • Statistics → Conversations / Endpoints for traffic volumetry; Protocol Hierarchy for protocol mix; I/O Graphs for beacon-rate analysis.
  • Export Objects recovers HTTP, SMB, FTP, TFTP, DICOM, IMF (email) objects directly to disk.

tshark is the headless equivalent — same engine, same filters, same dissectors, friendly to scripting. Use -T fields for CSV-style extraction, -z for stats.

🎯tcpdump — capture, period
Reliable capture Minimal footprint
tcpdump -i eth0 -s 0 -nn -w /tmp/case.pcap 'host 10.1.2.3 and not port 22'

tcpdump is the de-facto capture tool on Unix-like systems — small, fast, and the libpcap reference user. Use it whenever the priority is "don't lose packets":

  • -s 0 captures full frames (do not truncate).
  • -w writes raw PCAP — analyse later with Wireshark.
  • -G <seconds> -W <count> rotates capture files — essential for long-running collection.
  • BPF filter syntax is small but exact; escape carefully under shells.
  • On Windows the equivalent is WinDump or dumpcap (the capture engine inside Wireshark).
⚠Live capture mutates the system. Putting a NIC in promiscuous mode changes $LogFile, registry, and event-log state. For court-admissible captures, prefer a SPAN / TAP off-host.
🐍Scapy — packets as Python objects
Protocol RE Custom dissectors

(The JSON entry reads "Scrapy"; in network-forensics context this is universally Scapy — Philippe Biondi's packet manipulation framework.)

Scapy lets you read PCAPs as Python lists of layered objects, inspect any field, rewrite, replay, or synthesise traffic. Forensic uses:

  • Custom dissectors for protocols Wireshark doesn't know (in-house malware C2, IoT proprietary).
  • Bulk feature extraction — iterate over millions of packets and emit CSV.
  • Statistical beacon hunting: group packets by 5-tuple, compute inter-arrival times, flag low-jitter periodicity.
  • JA3 / JA3S / HASSH fingerprint computation when not pre-extracted.
  • Re-encoding traffic into known protocols to defeat anti-analysis padding.
⛏NetworkMiner — passive evidence extraction
File carving from PCAP Host + cred discovery

NetworkMiner is a passive network forensic analyser — feed it a PCAP and it categorises by host, OS-fingerprints, extracts files, credentials (HTTP basic, FTP, IMAP, POP3, SMB, NTLM challenges), keywords, DNS queries, and sessions. The free version handles most investigation needs; the Professional edition adds richer protocol coverage.

Best use: first-pass triage of large PCAPs to surface "what hosts talked to what" and "what files crossed the wire" before drilling in with Wireshark.

🧱PCAP — the format itself
Container basics Per-packet timestamps Truncation & loss

Two formats: the original libpcap (.pcap) and the modern PCAP-NG (.pcapng). PCAP-NG adds multiple interfaces per file, packet annotations, capture comments, and nanosecond timestamps. Wireshark writes PCAP-NG by default since 1.8; tcpdump still defaults to libpcap unless told otherwise.

Per-packet record contains: timestamp (seconds + microseconds or nanoseconds), captured length, original length, and the raw frame bytes. Captured < original means truncation — frames cut by a snaplen smaller than the wire MTU. Always verify snaplen before trusting payload completeness.

Hash & integrity: PCAP itself has no built-in integrity protection. SHA-256 the file at acquisition; sign the hash if chain-of-custody matters. Capture devices that emit signed PCAP-NG (e.g. Endace) are preferred for evidentiary work.

What's missing from PCAP that you'll wish you had:

  • TLS session keys — only present if the host was configured to dump them (SSLKEYLOGFILE) or if you can recover them from process memory (§02).
  • Application context — which process opened the socket. Solved by joining with Sysmon EID 3 / EDR netconn events.
  • User identity — solved by joining DHCP / RADIUS / authentication logs from the same time window.
!Encrypted traffic dominates modern PCAP. TLS 1.3, QUIC (HTTP/3), and DoH/DoT mean the bytes you capture are mostly meaningless without keys. Plan capture around metadata-first analysis (SNI, JA3/JA3S, cert SANs, timing, sizes) and grab session keys from host memory where possible.
⚙TCP/IP stack config — Tcpip\Parameters
Host identity DHCP & DNS
HKLM\SYSTEM\CurrentControlSet\Services\Tcpip\Parameters

Top-level values describe the host as a network entity:

  • Hostname / NV Hostname — the NetBIOS / DNS host name (compare to LNK TrackerDataBlock MachineID, §12).
  • Domain / NV Domain — the AD / DNS domain join.
  • DhcpDomain — domain assigned by DHCP last time the lease renewed.
  • NameServer — statically configured DNS server(s); if blank, DHCP-supplied servers are used instead and live under Interfaces\{GUID}.
  • SearchList — DNS suffix search order, useful for short-name resolution analysis.
🔌Per-interface config — Interfaces\{GUID}
IP at time of incident Lease times
HKLM\SYSTEM\CurrentControlSet\Services\Tcpip\Parameters\Interfaces\{adapter-GUID}

One subkey per NIC (matched to NIC GUIDs in NetworkList\Profiles). Captures the live config:

  • IPAddress / SubnetMask / DefaultGateway — current static config.
  • DhcpIPAddress / DhcpServer / DhcpNameServer — last DHCP-supplied values.
  • LeaseObtainedTime / LeaseTerminatesTime / T1 / T2 — DHCP lease timeline in UNIX epoch seconds. These pin the host to an IP address inside a specific time window — critical for DHCP-side log correlation.
  • EnableDHCP, NameServer override.
ℹCombined with the DHCP server logs (or M365 / Azure AD sign-in IPs), the lease times here let you place a known IP on this exact host at an exact second — the most reliable host-to-IP binding evidence available.
📡NetworkList — every network this host has ever joined
First / last connected SSID + Gateway MAC Profile category
HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\NetworkList\Profiles\{profile-guid} HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\NetworkList\Signatures\Unmanaged\{signature-guid}

Profiles stores one subkey per network the host has connected to — wired or wireless. Each carries:

  • ProfileName — the SSID for wireless, or "Network" / friendly name for wired.
  • Description — human-readable.
  • DateCreated and DateLastConnected — SYSTEMTIME-style binary blobs (16 bytes — Year/Month/DayOfWeek/Day/Hour/Min/Sec/Ms little-endian). Decode carefully; many parsers misread the byte order.
  • Category — Public (0), Private (1), or Domain (2). Reveals what firewall posture the host applied.
  • NameType — 6 = wireless, 23 = VPN, 71 = wired.
  • Managed — 1 if domain-managed.

Signatures\Unmanaged\{sig} stores the network fingerprint that identifies one network instance from another:

  • DefaultGatewayMac — MAC of the router the host saw on this network. Powerful: changing physical sites means changing this MAC, so a host that "joined Home WiFi" with two different gateway MACs has roamed to two different physical homes.
  • DnsSuffix — DNS suffix advertised on this network.
  • FirstNetwork — the SSID / network name as first seen.
  • ProfileGuid — joins back to the Profiles subkey.
✓Cross-checking with cloud sign-ins: Azure AD / M365 record the source IP. If that IP geolocates to "London" while NetworkList shows the host's last-connected WiFi gateway MAC matches a known "Madrid" router, you have evidence of token theft (the user was not where the sign-in claims).
🖥RDP MRU — Terminal Server Client\Servers
Per-user, per-host Lateral movement evidence
HKCU\Software\Microsoft\Terminal Server Client\Servers\<hostname-or-ip> HKCU\Software\Microsoft\Terminal Server Client\Default (MRU list)

Servers\<host> records every target the user typed into mstsc.exe. Per-subkey values include:

  • UsernameHint — the username most recently used to connect to that target.
  • CertHash — hash of the RDP server cert most recently accepted by the user.

Default stores MRU0..MRU9 values listing the most recent RDP targets in order.

Combine with §13 mstsc.exe Jump List and Security 4624 type 10 logon-success events on the target host for a complete lateral RDP picture.

🕸Internet Settings — proxies, WPAD, TLS
Forced-proxy attacks Configuration tampering
HKCU\Software\Microsoft\Windows\CurrentVersion\Internet Settings

Per-user WinINET / WinHTTP configuration — the very same keys that legitimate group policy edits and that malware commonly tampers with to redirect or sniff traffic:

  • ProxyEnable, ProxyServer, ProxyOverride — explicit proxy.
  • AutoConfigURL — PAC file URL. An attacker who controls this URL can selectively MitM specific destinations by returning a malicious PAC that points only those hosts to their proxy.
  • EnableAutoProxy — WPAD discovery on / off.
  • Zones subkey — Internet, Intranet, Trusted, Restricted zone mappings.
  • SecureProtocols — bitmask of TLS versions allowed.
!Sudden writes to AutoConfigURL or ProxyServer just before suspicious outbound traffic is a classic IOC. The registry key's last-write time from §06 is the timestamp you need; correlate with §10 Sysmon process-create for the tool that did it.
🎯Attacker traffic — the patterns to flag
Indicators Beacon analysis
  • Beacon periodicity. C2 frameworks (Cobalt Strike, Sliver, Mythic, Brute Ratel, Havoc) call out at fixed sleeps with configurable jitter. Compute inter-arrival times per 5-tuple — low standard deviation around a non-trivial mean is the tell.
  • DNS tunnels. Excessive subdomain length, high entropy in labels, abnormally many TXT / NULL queries to one domain.
  • Domain fronting & CDN abuse. SNI ≠ HTTP Host header, or SNI to legitimate CDN but JA3 fingerprint matches known toolkits.
  • Exfil bursts. Sudden uplink dominance — many small outbound flows or one large to a fresh destination.
  • Lateral SMB. Workstation-to-workstation TCP 445 between subnets that normally do not talk; SMB Tree Connect with administrative shares (ADMIN$, C$).
  • RDP from unusual sources. Public-IP RDP into a workstation; RDP at impossible-travel times.
  • Cleartext credentials. Basic auth / LDAP simple bind / SMB v1 from places that should know better.
🔒Encrypted channels — metadata, not payload
SNI / JA3 / certs

When the payload is opaque, the metadata still talks:

  • SNI (Server Name Indication) — destination hostname in TLS 1.2 ClientHello, cleartext. TLS 1.3 + ECH hides it; if missing, that itself is a fingerprint.
  • JA3 / JA3S — MD5 fingerprints of ClientHello / ServerHello extensions; distinguishes client libraries even when the SNI is generic. Public databases identify malware families.
  • Certificate analysis — Subject, SAN, validity dates, issuer. Self-signed or freshly-issued Let's Encrypt cert on a brand-new domain talking to a workstation is a strong lead.
  • Timing & size sidechannels — request/response sizes leak page identity for HTTPS; periodic small TLS Application Data records indicate keepalive / beacon behaviour.
  • Decryption when possible — recover SSLKEYLOGFILE from environment vars, browser preferences, or process memory (§02). Wireshark imports the file and decrypts retroactively.
🧬Data reconstruction — pulling files from the wire
Stream reassembly

Reconstruction patterns by protocol:

  • HTTP / HTTP/2 — Wireshark's Export Objects → HTTP; for chunked / gzipped content, prefer tshark -z http_req,tree + manual reassembly via the Follow Stream byte view.
  • SMB — Export Objects → SMB recovers files transferred via SMB1/2/3 reads & writes (when unencrypted).
  • FTP / TFTP / NFS — passive reassembly straightforward; Wireshark handles all.
  • Email (SMTP / IMAP / POP3) — Wireshark and NetworkMiner reconstruct full RFC 822 messages including attachments.
  • QUIC / HTTP/3 — encryption from the first packet; reconstruction requires session keys.
  • Custom protocols — Scapy + a hand-written reassembler; bridge to known formats (carving with foremost / file-magic).
⏱Timeline anchoring — PCAP as the wall clock
Independent timestamp source Host ↔ wire correlation

The wire and the host record time independently. That makes PCAP a powerful anchor: even when a compromised host's clock is suspect, the capture device sitting in front of it usually was not. Use this to:

  • Detect host clock tampering — if PCAP says a TLS handshake happened at T, and Sysmon EID 3 on the host says the corresponding connect was at T + 73 minutes, the host clock was set wrong (or was rolled back as anti-forensics).
  • Anchor host artefacts to subnetwork events — DHCP-supplied lease times in Tcpip\Parameters\Interfaces let you bound the IP before the PCAP starts; renewal traffic in the PCAP confirms continuity.
  • Replay sequence of compromise — TLS handshake → first beacon → payload download → process create. Each is a row on the timeline; they should be seconds apart when the chain is real.
ℹAlways preserve the capture device's NTP state alongside the PCAP. If the device drifted, you can math-correct only if you logged sync errors at acquisition.
🔬Analysing an unknown protocol
First-look heuristics
  • Port heuristics fail. Modern malware uses 443; some uses 53. Disregard port; let payload shape lead.
  • First bytes: magic numbers, ASCII command words, length-prefix or length-suffix? Compare entropy of first N bytes — high entropy from byte 0 usually means encryption or compression.
  • Direction patterns: who speaks first, request/response or push-only? Constant size? Constant time?
  • Field discovery: count distinct byte values at each offset across many packets; offsets with low cardinality are likely opcodes / flags.
  • Cross-source confirmation: §02 memory often contains the parsed plaintext of the same protocol — match opcodes to enums in the process.
🛠Protocol reverse engineering
Methodology

When a custom protocol must be fully decoded, the workflow is iterative:

  • Capture a long, varied sample (many sessions, different inputs).
  • Group packets by 5-tuple; align by direction.
  • Hypothesise field boundaries; verify with statistical alignment across sessions.
  • Implement a candidate dissector in Scapy or as a Wireshark Lua plugin.
  • Cross-check against §02 memory disassembly of the binary parsing the protocol.
  • Iterate — most fields fall to the third or fourth pass.
🔐Encryption in custom protocols
Key-material recovery Bridging to memory

Custom encrypted protocols usually fall into one of three buckets, each with characteristic forensic openings:

  • Symmetric with a static key (XOR, RC4, AES-ECB/CBC with a hard-coded key). Recover the key from §02 memory or §10 Amcache-linked binary reverse engineering. Replay decryption of historical PCAP works the moment you have the key.
  • Symmetric with a session-derived key (key exchange before payload). Look for an exchange phase early in each session — Diffie-Hellman, RSA-wrap of an AES key, or a hand-rolled equivalent. The key lives in process memory for the lifetime of the session.
  • Wrapped TLS / hidden-protocol-in-TLS (e.g. Cobalt Strike-over-TLS, Meterpreter HTTPS). The outer TLS is standard — use SSLKEYLOGFILE recovery or memory-scrape session keys to decrypt; then apply the protocol RE flow to the inner stream.

Document every key in the case file with its provenance — which memory image, which offset, when extracted. This is exactly the kind of evidence that gets challenged.

✓Time matters here: sessions live only as long as their keys do. PCAP captured today is undecryptable from a memory image taken tomorrow if the process has restarted in between. If you can capture memory and PCAP in one window, do.
Cross-references for corroboration

Network forensics is at its most powerful when wire evidence is joined to the host artefacts in the prior fourteen sections:

  • §01 Evidence Collection — PCAP, memory, and disk must be acquired in a documented order. Record the capture device, NTP state, and snaplen on first contact.
  • §02 Memory Forensics — TLS session keys, decrypted C2 payload, the process-to-socket mapping, and the binary parser of any custom protocol all live here. Memory is the bridge between opaque PCAP and meaningful evidence.
  • §04 NTFS — $UsnJrnl — files dropped by network downloads appear as FILE_CREATE within seconds of the matching HTTP/SMB transfer in PCAP.
  • §07 SAM — usernames in UsernameHint (Terminal Server Client\Servers) resolve to SIDs and account creation dates.
  • §08 SYSTEM / SOFTWARE — Tcpip\Parameters, NetworkList\Profiles, NetworkList\Signatures, and the Services hive together describe how the host was configured to talk to the network. The §08 walkthrough covers reading these.
  • §09 NTUSER — Terminal Server Client and Internet Settings are HKCU — they prove which user typed which RDP target and which proxy / PAC URL was active under their session.
  • §10 Application Execution — Prefetch / Amcache for the binary opening the socket; Sysmon EID 1 for process create, Sysmon EID 3 for the connect itself, Sysmon EID 22 for DNS query. These join PCAP 5-tuples back to processes and users.
  • §11 Cloud Storage — ODL upload events, OneDrive UserCid.dat, and tenant audit logs convert observed outbound TLS to specific files going to specific accounts.
  • §12 Shell Items & LNK — TrackerDataBlock — the MAC address embedded in a LNK joins to NetworkList DefaultGatewayMac and to Tcpip\Parameters NIC addresses.
  • §13 Jump Lists — mstsc.exe Jump List per-target entries confirm the RDP MRU read from Terminal Server Client\Servers and add per-launch timestamps.
  • §14 USB — USB-Ethernet impostors (BadUSB Network Adapters) appear in USB\ and add a NIC that NetworkList records; cross-check arrival timestamps when investigating NAC bypass.
