📊What SRUM is and why it's exceptional
Long-term execution record SID-attributed Hourly cadence, ~30 days Bytes per app per network
C:\Windows\System32\sru\SRUDB.dat (primary) …\sru\*.log / *.jrs (ESE write-ahead logs)

What: the System Resource Usage Monitor service (dssvc / DPS / Diagnostic Policy Service ecosystem) writes a snapshot of system activity every ~60 minutes, rotating ~30 days of history into a single ESE database. Where Prefetch tells you "this binary ran" and Amcache tells you "this binary existed", SRUM tells you how much a binary ran, which user ran it, how much network it used, and over which connection — for every hour back to a month.

Why it matters forensically:

  • Survives Prefetch clearing — SRUM is on a different write schedule.
  • Gives volumetric evidence of exfiltration: bytes-out per process per hour.
  • Carries the SID, so it ties activity to a specific human even on shared machines.
  • Tracks foreground / background time per app, distinguishing user-interactive use from background daemons.
  • Records the network interface profile a process used — joins WiFi SSID to data transfer.
ℹSRUM is a Win8+ feature, mature on Win10/11, present on Windows Server 2012+ (where many admins disable it). On servers, do not assume absence — confirm the DiagTrack / DPS services were running and check policy.
🗂Tables of interest
Schema

SRUDB.dat is an ESE database; each "provider" writes into its own table named by a GUID. The ones investigators care about:

  • Application Resource Usage ({D10CA2FE-6FCF-4F6D-848E-B2E99266FA89}) — CPU cycles, foreground / background time, faults, in-process CPU bytes per app per user per hour.
  • Network Data Usage ({973F5D5C-1D90-4944-BE8E-24B94231A174}) — BytesSent, BytesRecvd, link layer, interface GUID, per app per user per hour. The exfil-detection table.
  • Network Connectivity Usage ({DD6636C4-8929-4683-974E-22C046A43763}) — per-app per-interface connection lifetime — when a process was connected, how long.
  • Push Notifications ({D10CA2FE-6FCF-4F6D-848E-B2E99266FA8A}) — per app push notification counts.
  • Energy Usage ({FEE4E14F-02A9-4550-B5CE-5FA2DA202E37} + LT variant) — battery / power estimates per app.
  • SruDbIdMapTable — the lookup that resolves every AppId, UserId, InterfaceLuid integer into a string (process path, user SID, interface GUID). Every other table is unreadable without this one.
  • SruDbCheckpointTable — checkpoint book-keeping; not directly forensic but reveals SRUM service health.
🔑SruDbIdMapTable — the decoder ring
Universal join key

Every numeric ID in the other tables ultimately points back here. Three types of value matter:

  • App IDs — full process image path (\Device\HarddiskVolumeN\…\app.exe) or AppUserModelID for packaged apps. Resolve the volume number using §08 MountedDevices.
  • User IDs — binary-encoded SIDs.
  • Interface IDs — encoded NIC GUIDs that join to §15 NetworkList\Profiles.

SrumECmd handles this decoding automatically; parsers that just dump raw tables produce gibberish columns until joined.

⛏Acquisition & parsing
Workflow Dirty-DB recovery
esentutl.exe /r SRU /l . /s . /i # replay logs against SRUDB.dat SrumECmd.exe -d C:\Windows\System32\sru --csv out

Acquisition: the DPS service holds SRUDB.dat open with full file lock. Either:

  • Acquire from a hibernated / shut-down image where the database is clean.
  • Use VSS (esentutl.exe /y /vss or KAPE's SRUM target).
  • On a live host, stop the DPS service, copy, restart — and document the service interruption.

Recovery: if the captured DB is dirty (header shows "State: Dirty Shutdown" via esentutl /mh), replay the captured log generation set with esentutl /r. SRUM uses the prefix "SRU" for its log files — that's the value passed to /r. Without the logs, force-repair via esentutl /p is destructive but salvages most rows.

Parsing:

  • SrumECmd (Eric Zimmerman) — fastest forensic-friendly path; outputs one CSV per table, with all IDs resolved, ready for Timeline Explorer.
  • srum-dump (Mark Baggett) — uses an XLSX template; useful when you want pre-styled output.
  • ESEDatabaseView (NirSoft) — raw GUI for ad-hoc table inspection.
⚠Rows in the "live" hour are buffered in memory and not flushed to SRUDB.dat until the hourly write. The last 0–60 minutes of activity may be missing from a freshly-acquired database. For maximum coverage, combine with memory acquisition (§02).
📤Reading Network Data Usage — exfil pattern
Volumetric evidence

For each (app, user, interface, hour) tuple, the Network Data Usage table gives BytesSent and BytesRecvd. Read patterns:

  • Unknown binary with high BytesSent. Out-of-policy app uploading data — instant candidate for the malware Prefetch/Amcache hit.
  • BytesSent >> BytesRecvd for a normally request/response process (e.g. browser): suggests upload activity (forms / file uploads).
  • Activity on an unusual interface. A workstation that has only ever talked over its corp WiFi suddenly showing data on a "new" interface ID — pivot to §15 NetworkList\Profiles to see what network was joined.
  • Per-user breakdown. Two SIDs on the same host, only one with the heavy traffic — who was logged in.
⏱Application Resource Usage — process intent
CPU + foreground time

Per app per user per hour:

  • ForegroundCycleTime / BackgroundCycleTime — CPU cycles in each state.
  • ForegroundContextSwitches / BackgroundContextSwitches.
  • ForegroundBytesRead / Written / Background equivalents — disk I/O attribution.
  • FaceTime (Win10+) — wallclock time the app was actively foreground.

Distinguishes interactive use ("the user ran Word for two hours") from background execution ("backupd ran briefly every hour"). Critical when defending whether human action was involved.

🧭Practical investigative use
SRUM-led pivots
  • Find the exfiltration binary. Sort Network Data Usage by BytesSent desc; ignore browser / OneDrive / Defender; the remaining outliers are the candidates. Pivot to §10 Prefetch / Amcache for first-run.
  • Confirm user attribution. Prefetch and Amcache lack SID; SRUM has it. Joining gives "user X ran binary Y at hour H and uploaded Z bytes."
  • Reconstruct a session. Application Resource Usage FaceTime against logon-session boundaries (§19 Security 4624 / 4634) shows what the user did while logged on.
  • Detect dormant payloads. A process with high BackgroundCycleTime but tiny ForegroundCycleTime — typical for a C2 implant.
  • Defeat "I wasn't there." 30 days of hourly snapshots leave little room to deny presence. Combine with §07 SAM logon counts and §11 cloud sign-in IPs.
  • Cap the upper bound. When EDR is blind, SRUM is often the broadest historical record of what executed on a workstation.
ℹThe 30-day window is approximate. Retention varies with disk usage, ESE rollover policy, and OS version. Some images preserve closer to 60 days; some closer to 14. Always check the oldest TimeStamp column in the parsed CSV before claiming a window.
🧨Anti-forensics & integrity
Tampering detection
  • SRUM service stop / disable. Hour gaps in the Application Resource Usage table where prior hours had any process activity. Cross-check Service Control Manager events 7036 in System.evtx for DPS state changes.
  • SRUDB.dat deletion. $UsnJrnl shows the FILE_DELETE; recovery from VSS or unallocated possible.
  • Forced ESE rollover. Massive disk-fill attacks have been used to evict old SRUM rows. Free-space / chkdsk evidence around the supposed event is a tell.
  • Rare: direct row editing. Requires kernel/admin and a willingness to break the ESE checksum. Rebuilt indexes after such edits leave fingerprints; SrumECmd will skip pages it can't parse.
📐Server / Win11 nuances
Coverage variance
  • Windows Server 2016/2019/2022 — SRUM is present but often the DPS service is disabled by hardening baselines. Confirm before relying on it.
  • Windows 11 — schema additions for EnergyEstimationProvider and a few new column names; older parsers fall back gracefully.
  • Modern apps (UWP / Win32 packaged) appear with AppUserModelID rather than EXE path. Map via §08 SOFTWARE\…\App Paths or known AppID DB.
  • Volume serials in app paths — process image paths are recorded as \Device\HarddiskVolumeN\…; resolve N via §14 MountedDevices.
Cross-references for corroboration

SRUM is one of the best single sources on a modern Windows image — but every conclusion is stronger when joined back to the substrate:

  • §04 NTFS — disk I/O figures in Application Resource Usage correlate with $UsnJrnl bursts; mass deletions visible there should drive SRUM disk I/O for the responsible process.
  • §07 SAM — SRUM's per-row SID joins to SAM to name the user.
  • §08 SYSTEM — MountedDevices — resolves \Device\HarddiskVolumeN to drive letters, making SRUM AppId strings human-readable.
  • §09 NTUSER — RecentDocs / UserAssist / Office MRU often explain why a given app showed up in SRUM at a specific hour.
  • §10 Application Execution — SRUM corroborates Prefetch / Amcache / BAM. BAM records last execution per user; SRUM gives cycles over time. Together they bound "ran how often" and "ran for how long".
  • §11 Cloud Storage — OneDrive.exe / Dropbox.exe BytesSent rises around sync events; ODL log entries pin the exact files.
  • §14 USB — process I/O attributable to USB volumes via §08 mount data.
  • §15 Network Forensics — SRUM's interface-attributed totals join to NetworkList\Profiles by interface GUID, identifying which WiFi network the bytes traversed.
  • §16 Email Forensics — OUTLOOK.EXE bytes sent during a known mailbox export window quantifies the export.
  • §17 Windows Artefacts — Search indexer's CPU profile visible in Application Resource Usage; explains spikes attributable to indexing not user activity.
  • §19 Event Log Analysis — System.evtx 7036 events on the DPS service explain SRUM gaps. Security 4624/4634 frame the logon sessions whose SIDs SRUM attributes activity to.
